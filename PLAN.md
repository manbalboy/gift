# PLAN

## 1. Task breakdown with priority

### [P0] API: 휴먼 게이트 및 동시성 제어 강화 (REVIEW.md 반영)
- 휴먼 게이트 승인/거절 상태 전이 시 동시성 충돌 방어 로직 검증 및 보완
- 다중 비동기 요청(`asyncio.gather` 활용) 10건 동시 발송 시 단 1건만 처리되는지 확인하는 트랜잭션 락 테스트 구현 (`api/tests/test_human_gate.py`)
- 분산 환경 다중 워커에서 휴먼 게이트 타임아웃 노드 중복 처리를 방지하기 위한 분산 락(Distributed Lock) 로직 적용

### [P0] WEB: 마크다운 렌더링 개선 및 보안 강화 (REVIEW.md 반영)
- 기존 프론트엔드의 커스텀 정규식 마크다운 파서(`web/src/utils/sanitize.ts`)를 검증된 `marked` 라이브러리로 전면 교체
- `DOMPurify`와 결합하여 XSS 취약점 방어 및 악성 페이로드 방어 단위 테스트 추가 (`web/tests/`)
- 수십 MB 규모의 대용량 산출물(Artifact) 렌더링 시 브라우저 메모리 초과 현상 방지를 위한 라인 가상화(Virtualization) 또는 청크 렌더링 최적화 적용

### [P1] API/WEB: SSE 통신 안정성 확보 및 모의 테스트 (REVIEW.md 반영)
- 로컬 실행 환경(포트 3000 등) 및 대시보드 연결 시 Nginx Proxy 환경을 고려한 클라이언트 SSE 재연결 로직 견고화
- 네트워크 지연(15초 이상 공백) 상황을 모의하는 E2E 테스트 보완 및 Heartbeat 안정성 확보

### [P1] API: Workflow Engine v2 기반 구조 확장 (SPEC.md 반영)
- `workflow_id` 기반의 실행 기반 마련 및 RunOrchestrator 전환
- 노드별 상태 관리를 위한 `node_runs` 및 표준 출력물을 담을 `artifacts` 데이터 모델 스키마 확장

## 2. MVP scope / out-of-scope

**MVP Scope:**
- Workflow Engine v2로의 전환을 위한 기본 데이터 모델(`node_runs`, `artifacts`) 확장 및 오케스트레이터 기반 마련
- 다중 요청 동시성 제어 로직이 완비된 안전하고 일관된 휴먼 게이트(승인/반려/대기) API 및 테스트 구현
- ReDoS 위험 없이 대용량 텍스트를 안전하게 변환할 수 있는 `marked` + `DOMPurify` 기반 렌더링 프론트엔드 환경
- 불안정한 지연 네트워크 상태에서도 끊김 없는 SSE 로그 스트리밍 복원력 확보

**Out-of-scope:**
- ReactFlow 기반의 Visual Workflow Builder UI 구현 및 노드 에디팅 기능
- 외부 모델/툴 체인을 위한 Agent Marketplace 생태계 전반 (1차 스펙 정의 이후로 연기)
- 복잡한 CI/CD 파이프라인(GitHub Actions 트리거 등)과의 이벤트 버스 전면 통합

## 3. Completion criteria
- `api/tests/test_human_gate.py` 테스트 슈트에서 동시 10건 다중 비동기 요청 시 데드락 없이 오직 1건만 승인/거절 전이가 완료됨을 증명해야 합니다.
- 백엔드 스케줄러가 분산 락 제어하에 동작하여, 다중 워커 환경에서도 동일 휴먼 게이트의 타임아웃을 중복 처리하지 않아야 합니다.
- 기존 커스텀 정규식을 제거하고 `marked` + `DOMPurify`를 결합하여 렌더링이 깨지지 않으며, 관련된 XSS 및 마크다운 주입 방어 테스트가 통과해야 합니다.
- 프론트엔드 및 로컬 실행(포트 3000 등) 기준, SSE Heartbeat 지연 모의 테스트 시 15초 이상 응답이 지연되어도 연결이 회복되며 스트리밍 누락이 없어야 합니다.
- 기존 백엔드 API 테스트(122개)를 포함하여 새롭게 추가된 동시성/네트워크 모의 테스트가 모두 통과해야 합니다.

## 4. Risks and test strategy
- **대용량 산출물 렌더링 브라우저 크래시 및 ReDoS (Risk):** 
  - **Strategy:** 검증된 `marked` 라이브러리로 전환하고, 뷰어 컴포넌트에 가상 스크롤(Virtual Scroll) 구조를 도입해 DOM 부하를 최소화합니다. 극단적 중첩 코드를 담은 단위 테스트로 성능 저하와 ReDoS 공격 내성을 검증합니다.
- **다중 워커 환경 상태 전이 레이스 컨디션 (Risk):** 
  - **Strategy:** Redis 또는 Postgres 기반 분산 락(Distributed Lock)과 트랜잭션의 `with_for_update()`를 철저히 병행 활용하고, 파이썬 `asyncio.gather`를 활용해 극한의 병렬 요청 모의 테스트를 구축하여 상태 무결성을 증명합니다.
- **스트리밍 연결 불안정으로 인한 로그 누락 (Risk):** 
  - **Strategy:** Nginx 및 프론트엔드 네트워크 모의 스크립트를 통해 강제로 커넥션 대기 시간(Timeout)을 발생시키고, 자동 재연결 이후 데이터 유실 유무를 확인하는 E2E 테스트 시나리오를 적용합니다.

## 5. Design intent and style direction
- **기획 의도:** 이 플랫폼은 단순 자동화 도구를 넘어, 휴먼 개입(게이트 검수)과 실행 산출물(Artifact) 투명성을 보장하는 핵심 인프라입니다. 사용자에게 복잡한 파이프라인의 진행 과정을 신뢰성 있고 안도감 있게 전달하는 것이 핵심입니다.
- **디자인 풍:** 데이터를 명확하고 직관적으로 파악할 수 있는 **대시보드형(Dashboard)** 레이아웃. 산출물을 블록 형태로 구분 짓는 카드형 디자인.
- **시각 원칙:** 
  - 특정 노드의 상태(대기, 승인, 실패)가 한눈에 들어오도록 명료한 포인트 컬러 배색.
  - 마크다운 본문 및 코드 블록 리뷰 환경을 쾌적하게 만들기 위해, 가독성이 높은 모노스페이스 타이포그래피와 넉넉한 패딩(여백) 유지.
- **반응형 원칙:** **모바일 우선(Mobile-first)** 전략을 따르며, 모바일 환경에서 외부 접속 시에도 휴먼 게이트 승인 조작과 산출물 확인에 제약이 없도록 터치 영역과 레이아웃이 유동적으로 재조정되어야 합니다.

## 6. Technology ruleset
- **플랫폼 분류:** web / api
- **web:** React 프레임워크(Vite 환경)를 기반으로 구축합니다. 마크다운 변환은 필수적으로 `marked` 라이브러리와 `DOMPurify`를 사용합니다.
- **api:** FastAPI 기반 백엔드로 구축합니다. 비동기 I/O와 DB 동시성 제어, 분산 환경에서의 스케줄러 락(Distributed Lock) 등 고도화된 워크플로우 상태 제어에 최적화된 아키텍처를 구성합니다. 
- **실행 가이드 포트 제한:** 로컬 환경에서 대시보드 및 웹 서버 구동 시 포트는 충돌을 피하기 위해 가급적 3000번대(예: 3000, 3001)를 기본 포트로 계획합니다.
