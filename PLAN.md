## 2. MVP scope / out-of-scope
**MVP Scope:**
- 기존 구현된 Toast 알림의 치명적 UI 버그 수정 (창 크기 리사이징 미대응, 상태 업데이트 함수 내 부작용 제거)
- 알림 컴포넌트 렌더링에 대한 핵심 단위 테스트 구축
- 동시다발적 웹훅 수신 시 발생하는 큐 상태 경합(Race condition) 방어
- 좁은 해상도에서 텍스트가 잘리는 현상을 보완하는 인라인 텍스트 확장(Expand) UX 제공

**Out-of-scope:**
- 웹훅 데이터 페이로드(`error.detail`)에 대한 프론트엔드/백엔드 차원의 완벽한 보안 사니타이징(Sanitization) 로직 (현재 MVP에서는 UI 안정성 확보에 집중하며, 보안 위협 방어는 별도 백로그로 이관)
- Zustand, Redux 등 외부 전역 상태 관리 라이브러리를 사용하는 아키텍처로의 전환 (현재 React Hook 기반의 내장 상태 관리 유지)
- Toast가 아닌 전체 대시보드 시스템의 레이아웃 재편 및 디자인 개편

## 3. Completion criteria
- 개발 환경(로컬 서버 포트 `3000` 등 사용) 구동 후, 브라우저 폭을 데스크톱 수준에서 모바일 이하로 조절할 때 실시간으로 액션 버튼이 숨겨지고 UI가 알맞게 갱신되어야 함.
- 닫기 버튼 클릭을 통해 강제로 Toast를 닫을 때 무결하게 `dedupeKey`가 정리되는 로직이 단위 테스트를 통해 입증되어야 함.
- 가득 찬 알림 상태(3개)에서 신규 알림이 무작위 타이밍으로 들어오더라도, React 동시성 경고 에러가 콘솔에 남지 않으며 큐 상태가 엉키지 않아야 함.
- 에러 텍스트가 길 경우 컴포넌트를 터치하면 전체 내용을 확인할 수 있도록 말줄임이 해제되며 부드럽게 텍스트 높이가 확장되어야 함.
- 변경된 기능을 포함해 `web/` 하위의 모든 테스트 슈트가 에러 없이 성공(Pass)해야 함.

## 4. Risks and test strategy
- **Risks:** 동적 텍스트 확장 기능을 적용하면서 Toast 컴포넌트의 높이가 갑작스레 변경될 경우, 스택된 다른 알림 위로 겹치거나 화면 레이아웃에서 덜컹거림(Jumping)이 발생하여 시각적 불편을 초래할 수 있음.
- **Test strategy:**
  - **Unit Test:** Jest와 React Testing Library를 활용하여 리사이즈 이벤트 모킹(Mocking) 및 텍스트 확장 토글 상태 변경 시 렌더링을 꼼꼼하게 검증.
  - **Integration Test:** 다수의 웹훅 수신을 흉내 낸 비동기 테스트 함수를 작성하여 경합 상태가 발생하지 않고 최대 3개의 알림만 노출됨을 보장.
  - **E2E Test:** 3000번대 포트(예: `3000` 또는 `3100`) 환경을 통해 동적 브라우저 리사이징 및 모바일 화면에서의 텍스트 확장 UI 시뮬레이션을 수행하여 레이아웃 무결성 점검.

## 5. Design intent and style direction
- **기획 의도:** 시스템 백그라운드에서 실행된 워크플로우 결과를 사용자가 어떤 해상도의 기기에서도 즉각적이고 정확하게 확인할 수 있도록 보장하여 서비스 신뢰도를 높임.
- **디자인 풍:** Agent Hub의 대시보드와 조화를 이루는 모던하고 미니멀한 플로팅 카드(Floating Card) 스타일.
- **시각 원칙:** 
  - **컬러:** 상태(성공, 에러, 정보)를 대변하는 기존의 직관적인 디자인 토큰(Color Token) 유지.
  - **패딩/마진:** 답답함을 주지 않도록 내부 여백을 충분히 주며, 텍스트 확장 시 자연스러운 높이 트랜지션(Transition) 애니메이션 부여.
  - **타이포그래피:** 가독성 높은 산세리프 폰트를 기반으로 메시지 제목과 내용 간의 시각적 위계(Hierarchy) 준수.
- **반응형 원칙:** 모바일 우선(Mobile-first) 환경을 고려하여 좁은 화면에서는 액션 버튼 같은 부차적 요소를 감추고 제한적인 영역 안에서 터치 제스처를 통해 원본 텍스트에 접근 가능하도록 설계.

## 6. Technology ruleset
- **플랫폼 분류:** web
- 앱(web) 특성에 맞추어 **React(Vite)** 프레임워크 기반 하에 TypeScript를 철저히 준수하여 컴포넌트 로직을 전개.
- 창 크기 변경 감지 등 브라우저 네이티브 API 사용 시 반드시 React 훅(`useEffect` 등)의 의존성 및 클린업(Cleanup) 주기를 명시적으로 관리하여 메모리 누수 방지.
- 상태 확인 및 검증용 프리뷰 런타임 환경은 충돌 방지를 위해 로컬의 `3000`번대 포트를 일관성 있게 사용.
