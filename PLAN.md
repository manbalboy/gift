# PLAN

## 1. Task breakdown with priority

- **[P0] `Toast` 컴포넌트 보안 강화 및 렌더링 수정**
  - `web/src/components/Toast.tsx` 파일 내 `dangerouslySetInnerHTML` 및 커스텀 이스케이프 함수 사용 제거.
  - React의 안전한 텍스트 바인딩(`{item.message}`) 방식으로 DOM 렌더링 구조 변경.
- **[P0] 비정상 메시지 값에 대한 예외 처리**
  - `item.message`가 `null` 또는 `undefined`로 전달될 경우 "null", "undefined" 문자열이 렌더링되지 않도록 빈 문자열(`""`) 반환 방어 코드 추가.
- **[P1] 반응형 뷰포트 전환 시 제스처 상태 클린업**
  - `isMobile` 훅 반환값이 `true`에서 `false`로 변경되는 시점에 스와이프 진행 상태(`isSwipingRef`) 및 UI 이동값(`swipeOffsetX`, `swipeOffsetRef.current`)을 `0`으로 초기화하는 `useEffect` 로직 보완.
- **[P1] 전역 알림 대기열(Queue) 단위 테스트 보완**
  - 알림이 최대 노출 개수(3개)를 초과하여 발생했을 때의 대기열 처리 동작 스펙 작성.
  - 고유 키(`dedupeKey`) 기반 중복 필터링 동작을 검증하는 상태 관리자 측 단위 테스트(`Toast.test.tsx` 또는 상태 파일) 추가.
- **[P2] 오버플로우 측정 로직의 엣지 케이스 방어**
  - 렌더링 지연 등으로 대상 노드의 너비가 유효하지 않을 때 잘못된 텍스트 넘침 판정을 내리지 않도록 `clientWidth > 10` 검증 조건 추가.

## 2. MVP scope / out-of-scope

- **MVP Scope**
  - 프론트엔드 알림(`Toast`) 컴포넌트의 잠재적 XSS 취약점 제거 및 예외 상황 렌더링 안정화.
  - 모바일 제스처와 뷰포트 전환(모바일 ↔ 데스크톱) 간의 UI 상태 충돌 및 잔상 버그 해결.
  - 알림 시스템의 큐(Queue) 처리 및 중복 제거 규칙에 대한 단위 테스트 커버리지 확보.
- **Out-of-scope**
  - `Toast` 컴포넌트 이외의 다른 UI/컴포넌트 리팩토링.
  - 백엔드(FastAPI) 로직, 워크플로우 엔진, 에이전트 시스템에 대한 수정.
  - Playwright E2E 프레임워크 자체의 교체나 E2E 테스트 환경 전면 재구성.

## 3. Completion criteria

- `Toast.tsx`에서 `dangerouslySetInnerHTML` 호출이 완벽히 사라지고 테스트를 통과해야 합니다.
- 빈 메시지, `null`, `undefined` 알림이 발생하더라도 화면에 깨진 문자열이 보이지 않아야 합니다.
- 모바일 뷰에서 알림 카드를 스와이프 하는 도중 창 크기를 키워 데스크톱 뷰로 전환할 때, 알림 카드가 스와이프 된 채로 멈춰 있지 않고 제자리로 돌아와야 합니다.
- 텍스트 오버플로우 측정 함수(`measureMessageOverflow`)에서 렌더링 지연 시 오작동하는 조건(너비 10 이하)이 조기에 차단(`early return`)되어야 합니다.
- 글로벌 상태 대기열 큐잉 로직 단위 테스트가 최소 1건 이상 신규 작성되어 성공해야 합니다.

## 4. Risks and test strategy

- **Risks**
  - E2E 환경에서 합성 이벤트(`dispatchEvent`)로 동작하는 스와이프가 실제 모바일 기기의 터치 스크롤이나 브라우저 인터랙션 한계를 완벽히 모사하지 못할 수 있습니다.
  - 뷰포트 상태 변환 시 리렌더링 및 클린업이 빠르게 일어나면서 미세한 애니메이션 끊김 현상이 발생할 수 있습니다.
- **Test Strategy**
  - **단위 테스트 (Unit Test)**: Jest 기반 환경에서 알림 상태 추가/삭제, 중복 키 발생, 4개 이상의 알림 호출 시나리오를 철저히 격리해 검증합니다.
  - **E2E 테스트 (E2E Test)**: Playwright를 사용해 변경된 `Toast` DOM 구조에서의 텍스트 노출 및 모바일 스와이프 제스처를 검증합니다. 충돌 방지를 위해 로컬 실행 시 **`PORT=3100 npx playwright test`** 등 3000번대 포트를 사용하여 테스트를 수행합니다.
  - **수동 테스트**: 크롬 개발자 도구의 디바이스 모드를 활용해 드래그 제스처 중 화면 회전(모바일 ↔ 태블릿/데스크톱) 시나리오를 교차 검증합니다.

## 5. Design intent and style direction

- **기획 의도**: 핵심 시스템의 상태, 성공, 경고, 오류 등의 알림을 사용자에게 일관성 있고 왜곡 없이 전달하며, 환경 변화 시에도 UI 버그 없는 신뢰감을 제공하는 핵심 경험.
- **디자인 풍**: 모던, 대시보드형 스타일. 그림자와 경계선, 여백을 활용해 떠 있는 형태의 미니멀한 알림 카드 레이아웃.
- **시각 원칙**: `<`나 `>` 같은 제어 문자가 포함된 텍스트라도 코드 그대로 안전하게 텍스트만 표시되도록 타이포그래피 정렬을 엄격히 유지합니다. 에러(레드) 및 경고(옐로우) 레벨별 컬러 가이드를 일관되게 적용합니다.
- **반응형 원칙**: 모바일 우선(Mobile-first) 규칙을 적용해 모바일 환경에서는 터치 스와이프로 닫기가 가능하도록 하고, 데스크톱 환경에서는 명시적인 닫기 버튼과 hover 동작 위주로 동작하도록 이원화된 상호작용을 보장합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web
- **기술 스택**: React (Vite 기반 프레임워크), TypeScript
- **스타일링**: 기존 CSS 기반 스타일링 유지 (추가 라이브러리 도입 금지)
- **테스트 규칙**: E2E 로컬 테스트 실행 시 반드시 3000번대 포트(예: 3100)를 사용해 구동해야 합니다.
- **보안 원칙**: 부분적인 HTML 렌더링 허용 요구사항이 없다면 XSS 공격 원천 차단을 위해 DOM 내장 바인딩 기능만을 사용합니다.
