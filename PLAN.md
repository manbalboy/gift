# PLAN

## 1. Task breakdown with priority

**Phase 1: 기능 오류 및 보안 취약점 수정 (최우선순위)**
- **Task 1.1:** `web/src/utils/escapeHtml.ts` 로직을 활용하여 `web/src/components/Toast.tsx` 내 출력 데이터의 XSS 방어 로직 적용 (P0)
- **Task 1.2:** `Toast.tsx` 터치 이벤트(`onTouchStart`, `onTouchMove`)에 모바일 멀티 터치 방어 로직 추가 (`event.touches.length > 1` 무시) (P0)
- **Task 1.3:** `Toast.tsx` 내 폰트 지연 로딩 대응을 위해 `document.fonts.ready` 완료 시점의 텍스트 오버플로우 재평가 및 폴백(Fallback) 함수 적용 (P0)
- **Task 1.4:** 화면 가림 현상 방지를 위해 전역 알림 상태에 다중 알림 큐잉(Queueing) 및 중복 필터링 로직 구현 (P1)

**Phase 2: 테스트 커버리지 확보 및 E2E 테스트 추가 (우선순위 상)**
- **Task 2.1:** XSS 방어, 중복 필터링 및 큐잉 로직 검증을 위한 `web/src/components/Toast.test.tsx` 단위 테스트 작성 (P1)
- **Task 2.2:** `web/tests/e2e/toast-layering.spec.ts` 내에 단일 손가락 스와이프(88px 임계값) 닫기 동작을 검증하는 E2E 테스트 시나리오 추가 (P1)
- **Task 2.3:** 알림 컴포넌트의 접근성(`role="alert"`, `role="status"`, `aria-live="polite"`) 렌더링 검증 E2E 테스트 추가 (P1)

**Phase 3: 고도화 플랜 (현재 구현 연계 추가 기능)**
- **Task 3.1:** 에러 알림 상세 보기 (Expand/Collapse) 기능 추가 (P2)
  - 근거: Workflow 실행 실패 시 반환되는 에러 스택/상세 로그를 좁은 Toast 영역에서 확인하기 어렵기 때문에, 펼쳐보기 기능을 통해 사용자 디버깅 경험을 개선합니다.
- **Task 3.2:** Toast 애니메이션 및 모바일 스와이프 제스처 인터랙션 고도화 (P2)
  - 근거: 스와이프 시 닫히기 전 자연스러운 스프링 트랜지션을 추가하여 네이티브 앱과 유사한 사용자 경험을 제공합니다.

## 2. MVP scope / out-of-scope

**MVP Scope:**
- DevFlow Agent Hub 워크플로우의 상태 변화 및 에러를 안전하게(XSS 방지) 화면에 표시하는 알림 시스템 구현.
- 브라우저 폰트 로드 지연에도 레이아웃이 깨지지 않는 안정적인 텍스트 렌더링.
- 다중 알림 발생 시 최대 노출 개수를 제한하고 큐에 저장하여 순차 표시.
- 모바일 기기에서의 스와이프 제스처 및 터치 오작동 방지.
- 프론트엔드 포트(3100) 기반의 자동화 단위/E2E 테스트 통과.

**Out-of-scope:**
- 시스템 전체 알림 내역을 모아보는 영구적인 히스토리 알림 센터 페이지 구축.
- 슬랙, 이메일 등 외부 메신저 채널 연동.
- 사용자별 맞춤형 알림 설정 및 구독 기능.

## 3. Completion criteria

- `Toast.tsx` 컴포넌트에서 모든 렌더링 텍스트가 안전하게 이스케이프 처리되어 악의적 스크립트가 실행되지 않음.
- 3개 이상의 알림이 동시에 발생하더라도 설정된 제한 개수만큼만 화면에 렌더링되며 나머지는 큐에서 대기함.
- 모바일 시뮬레이터 또는 실기기에서 다중 터치 시 알림이 비정상적으로 닫히거나 스와이프 임계값이 잘못 적용되지 않음.
- 브라우저 폰트 로드 지연 및 실패 상황에서도 `measureMessageOverflow`가 정상 계산되어 텍스트 잘림이나 UI 깨짐이 발생하지 않음.
- `web/src/components/Toast.test.tsx` 단위 테스트가 성공적으로 통과함.
- Playwright를 활용한 E2E 테스트(`PORT=3100 npx playwright test`)가 모든 브라우저 엔진에서 100% 통과하며 필수 접근성(a11y) 속성이 검출됨.
- 네트워크 쓰로틀링 환경 수동 검수 시 Expand 버튼이 적절한 시점에 올바르게 노출됨.

## 4. Risks and test strategy

**Risks:**
- 렌더링 엔진(Chromium, WebKit) 간 터치 이벤트 및 레이아웃 재계산 타이밍의 미세한 차이로 인해 E2E 스와이프 테스트가 간헐적으로 실패(Flaky)할 수 있습니다.
- 구형 모바일 브라우저에서 `document.fonts.ready` API 미지원 시 재평가 로직이 동작하지 않을 수 있습니다.

**Test strategy:**
- **단위 테스트 (Jest):** XSS 이스케이프 유틸리티 함수와 상태 관리 모듈의 큐잉/중복 필터링 로직을 격리하여 테스트합니다.
- **E2E 테스트 (Playwright):** 3100번 포트를 사용하여 로컬 서버를 구동하고, 88px 스와이프 임계값 및 접근성 ARIA 속성을 자동화 스크립트로 검증합니다. 터치 이벤트는 브라우저별 에뮬레이션을 적극 활용합니다.
- **폴백 검증:** `document.fonts.ready` 지원 여부와 관계없이 일정한 타임아웃(`setTimeout`) 기반의 폴백 로직을 구성하고 단위 테스트를 통해 실행을 확인합니다.
- **수동 QA:** 크롬 개발자 도구의 네트워크 속도를 제어(Throttling)하여 폰트 다운로드 지연 시나리오를 재현하고 UI 동작을 육안으로 검수합니다.

## 5. Design intent and style direction

- **기획 의도:** 지속적으로 실행되는 Agent 워크플로우의 결과 및 오류 상태를 사용자 작업 흐름을 방해하지 않고 직관적으로 인지시키는 피드백 경험을 제공합니다.
- **디자인 풍:** 불필요한 장식을 배제한 미니멀하고 모던한 플로팅 카드(Floating Card) 스타일.
- **시각 원칙:**
  - 컬러: 성공은 녹색 계열, 에러는 적색 계열, 정보 표시는 회색 및 푸른색 계열로 테마를 명확히 구분하여 인지성을 높입니다.
  - 패딩/마진: 텍스트 가독성을 위해 내부 여백(Padding)을 12~16px로 넉넉하게 적용하고, 알림 카드 간 외부 여백(Margin)을 주어 독립된 객체임을 강조합니다.
  - 타이포: 시스템 기본 산세리프 폰트를 사용하여 렌더링 속도를 확보하고, 긴 텍스트는 우선 말줄임(Ellipsis) 처리한 뒤 Expand 버튼으로 펼칠 수 있도록 유도합니다.
- **반응형 원칙:** 모바일 우선(Mobile-first) 규칙을 따릅니다. 모바일 환경에서는 뷰포트 하단(Bottom) 전체 너비로 알림이 노출되며 스와이프 닫기 제스처를 적극 지원합니다. 데스크톱 환경에서는 화면 우측 상단이나 하단 모서리에 제한된 너비로 플로팅됩니다.

## 6. Technology ruleset

- **플랫폼 분류:** web, api
- **web:** React 프레임워크(Vite 환경) 기반으로 설계 및 구현합니다. 테스트는 Jest와 Playwright를 사용합니다.
- **api:** 워크플로우 통신 및 데이터 제공 백엔드가 필요한 경우 기존 FastAPI 기반 아키텍처를 따릅니다.
