# PLAN

## 1. Task breakdown with priority
**P0: 시스템 안정성 및 REVIEW.md 필수 반영 (버그 수정)**
- **API 서버 3100번대 포트 충돌 방지 및 예외 처리**
  - 변경 파일 후보: `api/app/main.py`, `scripts/run-api-31xx.sh`
  - 영향 범위 및 근거: `main.py` 서버 구동부 및 셸 스크립트 레벨에서 포트 점유 시 서버가 Crash 되지 않도록 Address already in use 예외를 우아하게 잡아내고, 재시도하거나 안전하게 종료(Graceful Shutdown)하는 로직을 보완하여 인프라 안정성을 확보해야 합니다.
- **대용량 에러 로그 처리 안정성 확보 (UI/메모리 방어)**
  - 변경 파일 후보: `web/src/components/ErrorLogModal.tsx`
  - 영향 범위 및 근거: 10만 자 이상의 거대 로그 다운로드 시 클라이언트 메모리에 `Blob`을 한 번에 올리는 방식을 최적화하여 메모리 부족 및 프리징을 방지해야 합니다. 또한 구형 브라우저에서 `Intl.Segmenter` 미지원 시 Grapheme Cluster(ZWJ 이모지 등)가 깨지지 않도록 Fallback 정규식의 엣지 케이스를 보완합니다.

**P1: 핵심 루프 엔진 검증 및 스트레스 테스트 커버리지 확대**
- **Loop Simulator 엔진 엣지 케이스 단위 테스트 보강**
  - 변경 파일 후보: `api/tests/test_loop_simulator.py`
  - 영향 범위 및 근거: `max_loop_count` 및 `budget_limit` 초과 상황, 음수 파라미터 주입 시 즉시 `stopped` 상태로 전이되는 방어 로직 동작을 보장해야 합니다. 상태 강제 전이 시 비동기로 들어온 큐잉 작업이 락(Lock) 범위 밖에서 누락되는 타이밍 이슈(Race condition) 시나리오 테스트를 구축합니다.
- **에러 로그 렌더링 스트레스 테스트 및 포트 충돌 통합 테스트 추가**
  - 변경 파일 후보: `web/src/components/ErrorLogModal.test.tsx`, 신규 통합 테스트 셸 스크립트
  - 영향 범위 및 근거: ZWJ 결합 이모지가 포함된 대용량(10만 자 이상) 로그 렌더링 벤치마크 테스트를 통해 브라우저 성능 이슈를 사전 차단합니다. 또한 포트 3100번 점유를 강제 시뮬레이션하여 우아한 실패 로직이 동작하는지 검증하는 인프라 레벨 테스트를 추가합니다.

**P2: (인접 기능) 루프 엔진 제어(Control) 엔드포인트 연결**
- **루프 Start/Stop 제어 API 초안 통합**
  - 변경 파일 후보: `api/app/api/loop_engine.py` (기존 라우터 확장 혹은 수정)
  - 영향 범위 및 근거: SPEC.md에 정의된 Self-Improvement Loop 엔진 설계의 필수 요구사항인 제어 명령(Start, Pause, Resume, Stop)을 반영하여, 백엔드 엔진을 안전하게 조작할 수 있는 최소한의 FastAPI 엔드포인트를 구성합니다. P0/P1의 안정화 작업과 자연스럽게 이어지며 MVP 스펙 검증을 완성하는 범위로 제한합니다.

## 2. MVP scope / out-of-scope
**MVP Scope**
- `REVIEW.md`에 명시된 모든 결함 사항의 완벽한 수정 및 누락된 단위/통합 테스트 커버리지 달성.
- API 로컬 실행 시 3100번대 포트 사용 및 충돌 방어 로직 구축.
- Self-Improvement Loop 엔진의 안전한 종료 제어(`max_loop_count`, `budget_limit`) 검증과 기초적인 제어 API 연동.
- 1회 실행 사이클의 결과물은 Docker 기반으로 실행 가능해야 하며, Preview 외부 노출 포트는 7000-7099 범위를 준수.

**Out-of-scope**
- 24시간 완전 무인 동작(Autonomous)을 위한 고도화된 다중 에이전트 협업 체계 구현 (초장기 다음 마일스톤으로 연기).
- 별도의 대규모 DB 설계 변경이나 데이터 마이그레이션을 동반하는 장기 기억 시스템(Memory 시스템)의 완전체 구축.
- 프로젝트 도메인을 벗어난 불필요한 전체 UI 프레임워크 리팩토링 및 광범위한 디자인 개편.

## 3. Completion criteria
- `REVIEW.md` TODO 리스트의 모든 작업 내역 구현 및 테스트 통과.
- 로컬 환경에서 3100 포트가 사전 점유된 상태로 API 시작 시, 치명적 예외 없이 우아하게 재시도하거나 종료 코드를 반환.
- web 브라우저 및 모바일 뷰에서 10만 자 이상의 ZWJ 이모지 포함 로그를 버벅임 없이 렌더링하고 다운로드할 수 있음(메모리 초과 징후 없음).
- `budget_limit` 및 `max_loop_count` 설정의 예외 값/초과 조건 발생 시 루프가 즉각 `stopped`로 전이됨을 테스트로 증명.
- PR 본문에 Docker Preview 배포 정보(대상 컨테이너명, 7000-7099 포트, Preview 호스트 URL)를 명확히 작성.

## 4. Risks and test strategy
- **Risks**
  - 대용량 로그 데이터 클라이언트 분할/렌더링 시 디바이스(특히 모바일 환경)의 메모리 제약으로 인한 OOM(Out of Memory) 및 프리징.
  - 동시성 제어 시 루프 엔진의 강제 종료와 신규 명령 주입 간의 Race Condition 타이밍 문제.
- **Test Strategy**
  - **스트레스 및 UI 벤치마크:** Jest 및 React Testing Library 기반으로 거대 문자열 더미(10만 자 이상)와 복합 이모지를 주입하여 컴포넌트 마운트 시간 및 메모리 누수를 감시.
  - **인프라 통합 테스트:** 셸 환경에서 `nc`나 파이썬 스크립트를 통해 의도적으로 3100 포트를 바인딩한 후 서버 기동 스크립트를 호출하여 `exit` 코드 및 로깅 검증.
  - **예외 상황 집중 단위 테스트:** Pytest를 활용해 루프 시뮬레이터 락(`_lock`) 동작 중과 밖에서 발생하는 상태 전이 요청 타이밍을 모킹(Mocking)하여 검증하고 극단적인 파라미터 주입 테스트 추가.

## 5. Design intent and style direction
- **기획 의도:** 예측 불가능한 시스템 오류와 대규모 로그 출력 속에서도 개발자가 에러의 원인과 루프의 상태를 직관적으로 파악할 수 있는 견고하고 안정적인 디버깅/제어 경험을 제공합니다.
- **디자인 풍:** 불필요한 장식을 배제한 대시보드형의 미니멀, 모던 카드형 스타일.
- **시각 원칙:**
  - **타이포 및 컬러:** 방대한 정보 가독성을 위해 코드 및 로그 영역은 모노스페이스(Monospace) 폰트 유지. 시스템 멈춤 및 에러 등 치명적 상태는 식별이 명확한 고대비 컬러(예: Red 계열)로 표시.
  - **패딩 및 마진:** 복잡한 텍스트 데이터 사이에 여유로운 패딩과 카드형 마진을 배치하여 시각적 압박감을 완화.
- **반응형 원칙:** 모바일 우선(Mobile-first) 구조를 채택. 디스플레이 면적이 좁은 모바일 화면에서도 텍스트 박스 수평 스크롤을 제한하고 자동 줄바꿈 처리하여 로그 가독성을 우선 보장.

## 6. Technology ruleset
- **플랫폼 분류:** web 및 api
- **web:** 현 저장소 구조를 준수하여 React 기반 라이브러리/프레임워크(TSX)로 구현 및 계획.
- **api:** 루프 통제 및 엔진 제어를 위한 백엔드는 FastAPI(Python) 기반으로 계획 및 유지보수. 실행 가이드에 따른 로컬 포트는 3000번대 할당.
