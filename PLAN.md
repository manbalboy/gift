# PLAN

## 1. Task breakdown with priority

**Phase 1: 핵심 루프 엔진 안정화 및 보안 (우선순위: 높음)**
- `api/app/api/loop_engine.py` 내의 모든 제어 라우터(Start, Pause, Resume, Stop, Inject Instruction)에 인증(Authentication) 및 인가(Authorization) 처리를 위한 `Depends` 로직 추가.
- 다중 워커 환경에서 루프 엔진 중복 실행(Race condition)을 방지하기 위해 `api/app/services/loop_simulator.py`에 Redis 기반 분산 락(Distributed Lock) 체계 및 TTL 적용.
- 장기 실행(Long-Running) 중 시스템 로그 및 상태 데이터 급증으로 인한 OOM 방지를 위해, 데이터베이스 및 메모리의 자동 로그 정리(Retention/Windowing) 정책 구현.

**Phase 2: 프론트엔드 모니터링 UI 및 성능 최적화 (우선순위: 중간)**
- 프론트엔드의 `web/src/components/SystemAlertWidget.tsx` (또는 메인 로그 뷰)에 `@tanstack/react-virtual` 라이브러리를 도입하여 대용량 로그 렌더링 시 브라우저 렌더링 지연 및 메모리 누수 방지(가상 스크롤 적용).
- 4개 핵심 엔진(Analyzer, Evaluator, Planner, Executor)의 현재 상태와 Quality Score를 실시간 반영하는 대시보드 UI 연동.
- 지정된 환경(Preview 포트 7000-7099) 및 허용 Origin(`manbalboy.com`, `localhost`, `127.0.0.1`)에 대한 엄격한 CORS 보안 정책 적용.

**Phase 3: 결함 방지를 위한 자동화 테스트 구축 (우선순위: 높음)**
- 분산 락 획득 후 프로세스 비정상 종료 시, TTL 만료에 의해 락이 자동 해제되고 복구되는지 확인하는 통합 테스트(`test_workflow_engine.py` 등) 작성.
- 유효하지 않은 토큰으로 제어 API 접근 시 403 Forbidden 응답 및 다중 워커의 동시 Start 요청 시 동시성 방어 검증 테스트 작성.
- 로컬 포트 3100번 환경에서 대용량 상태 로그 스트리밍을 발생시켜 프론트엔드의 렌더링 성능과 네트워크 단절 복구를 점검하는 E2E 스트레스 테스트 구현.

## 2. MVP scope / out-of-scope

**MVP Scope**
- AI 자율 주행을 위한 기본 루프 제어(Start, Pause, Resume, Stop) API 및 사용자 인증 로직.
- 단일/다중 워커 환경에서도 충돌 없이 실행을 보장하는 Redis 분산 락 기반 컨트롤러 파이프라인.
- 무한 반복 방지 제어(최대 루프 횟수, 동일 수정 반복 감지, 최소 품질 향상 임계치) 기본 로직.
- 1만 건 이상의 상태 로그 스트리밍에도 프레임 드랍 없이 부드럽게 동작하는 프론트엔드 실시간 대시보드.
- 로컬 개발 환경(프론트엔드 포트 3000, 백엔드 API 포트 3100) 구성 및 지정된 Preview용 Docker 환경(포트 7000-7099) 대응.

**Out-of-scope**
- 코드를 실질적으로 변경하고 리팩토링하는 내부 AI 모델(LLM)의 프롬프트 튜닝 (우선적으로 파이프라인과 상태 관리 구조의 안정성 확보에 집중).
- IDE 내부 플러그인 연동 및 복잡한 코드 에디터 양방향 편집 UI.
- 과거 모든 루프의 이력을 수 개월 단위로 영구 보존하는 대규모 데이터 웨어하우스 구축(단기 보존 및 자동 삭제 정책까지만 구현).

## 3. Completion criteria

- 권한이 없는 사용자가 `loop_engine.py`의 제어 API를 호출했을 때 완벽하게 차단되며 적절한 오류 코드가 반환됨.
- 여러 사용자가 동시에 루프 `Start`를 요청하더라도 분산 락에 의해 단 1개의 프로세스만 실행 루프를 점유함.
- 대용량 데이터 유입 상황(스트리밍)에서 프론트엔드의 메모리 점유율이 안정적으로 유지되며 가상 스크롤이 정상 동작함.
- 실행 중인 워커 인스턴스가 강제 종료된 후, 정해진 시간 내에 락이 회수되어 루프 시스템의 중단 없는 재가동이 가능함.
- 프로젝트 전체가 Docker 환경에서 정상 빌드되며, SPEC에 명시된 엄격한 CORS 및 포트 노출 정책을 모두 통과함.

## 4. Risks and test strategy

**Risks**
- **상태 전이 충돌:** 비동기 작업 지연으로 인해 장기 실행 중 Inject Instruction이 들어왔을 때 기존 파이프라인과 충돌하거나 상태가 꼬일 위험.
- **분산 락 교착 상태:** 워커 노드 실패 시 락이 영구적으로 해제되지 않아 시스템 전체가 마비될 위험.
- **스트리밍 단절:** 로컬 및 Preview 환경에서 네트워크 지연이나 끊김으로 인해 프론트엔드에서 로그 유실이나 순서 섞임이 발생할 위험.

**Test Strategy**
- **보안 및 동시성 테스트:** 인증 실패 및 만료 권한에 대한 엣지 케이스 단위 테스트 수행, 다중 스레드를 이용한 API 동시 호출 테스트로 Race condition 방지 로직 검증.
- **통합 락 테스트:** 의도적으로 워커 프로세스를 Kill하고 TTL 이후 자동 복구되는 일련의 과정을 테스트 코드로 증명.
- **스트레스 및 E2E 테스트:** 대규모 로그 모의 발생기(Mock)를 3100번 포트에 띄워 프론트엔드 UI 렌더링 한계 상황 점검 및 네트워크 일시 단절 시의 로그 정합성 회복 테스트.

## 5. Design intent and style direction

- **기획 의도:** 사람이 코드 작성에 개입하지 않고 시스템이 '코드 생성 → 평가 → 개선'을 자율적으로 수행하는 과정을 투명하게 시각화하여 전달해야 합니다. 시스템이 24시간 안정적으로 실행되고 있다는 신뢰감과 통제권을 사용자에게 부여하는 것이 핵심입니다.
- **디자인 풍:** 진행 상태를 한눈에 파악할 수 있는 **모던 대시보드형** 및 **카드형** UI. 불필요한 장식을 배제하고 데이터와 상태의 위계가 명확하게 드러나는 미니멀 스타일.
- **시각 원칙:**
  - **컬러:** 개발자 친화적인 Dark theme을 기본으로 하여 장시간 모니터링 시 눈의 피로를 최소화. 동작 상태(정상 구동: Green, 대기/경고: Amber, 오류: Red)를 명확한 포인트 컬러로 구분.
  - **타이포/여백:** 코드 및 로그 데이터 영역은 Monospace 폰트를 사용하여 가독성을 확보하고, 각 핵심 엔진 지표(Card) 사이의 마진과 패딩을 넉넉히(16px~24px) 두어 답답하지 않은 레이아웃 구성.
- **반응형 원칙:** **모바일 우선(Mobile-First)**으로 설계. 모바일 화면에서는 루프의 현재 상태(Start/Pause) 및 품질 점수 요약이 상단에 배치되어 즉각적인 제어가 가능하게 하고, 상세 스트리밍 로그 뷰는 데스크톱 환경에서 좌우 패널로 확장되도록 유연하게 대응.

## 6. Technology ruleset

- **플랫폼 분류:** web / api
- **web:** React 프레임워크(Vite 환경) 및 TypeScript 기반으로 계획. 대용량 로그 리스트의 성능 최적화를 위해 `@tanstack/react-virtual` 라이브러리를 채택. 로컬 프론트엔드 개발 서버 포트는 3000 사용.
- **api:** FastAPI 프레임워크 기반으로 계획. 다중 워커 상태 관리를 위해 Redis를 사용하고, 실시간 로그 스트리밍을 위한 비동기 처리(SSE) 활용. 로컬 백엔드 서버 구동 포트는 3100 사용.

## 7. 고도화 플랜

현재 도출된 REVIEW.md의 안정화 및 렌더링 성능 최적화(TODO) 작업이 완료된 후, 시스템의 자율성과 모니터링 편의성을 높이기 위한 확장 계획입니다. 기존 구현과 자연스럽게 연결되는 인접 기능으로 제한합니다.

1. **지능형 순환 중단 감지 (Intelligent Loop Breaker)**
   - **근거:** 시스템이 무한 루프나 동일한 코드 수정을 반복하는 현상(Loop Stability 문제)을 단순한 횟수 제한(`max_loop_count`)만으로 막기 어렵습니다. Evaluator의 점수가 연속으로 하락하거나, 동일한 테스트가 3회 이상 실패할 경우 AI가 스스로 '무의미한 루프'로 판단하여 자동으로 Pause 상태로 전환하는 휴리스틱 방어 로직이 필요합니다.
   - **구현 경계:** FastAPI 백엔드의 `Evaluator Engine` 결과 분석 로직에 룰셋 기반의 중단 판별 코드를 추가하고, 프론트엔드 대시보드 시스템 로그에 자동 중단 사유를 명시적으로 출력합니다. (외부 알림 연동 제외)

2. **장기 기억 메트릭 시각화 (Memory Metrics Chart)**
   - **근거:** 24시간 장기 실행되는 엔진의 성과를 직관적으로 확인하려면 단순한 현재 텍스트 점수뿐 아니라 '과거부터 현재까지 코드가 얼마나 개선되었는지' 증명해야 합니다.
   - **구현 경계:** Memory 시스템에 저장된 일별/루프별 Code Complexity, Test Coverage, Quality Score의 변화량을 프론트엔드 React 대시보드에 경량 차트(예: Recharts)를 활용하여 심플한 꺾은선 그래프로 시각화합니다. (다차원 분석이나 복잡한 보고서 뷰 제외)
