# PLAN

## 1. Task breakdown with priority

- **[P0] 로컬 환경 안정화 및 포트 고정 (REVIEW 반영)**
  - 대상 파일: `web/vite.config.ts`
  - 내용: Vite 서버 설정(`server`)에 `strictPort: true`를 추가하여 로컬 구동 시 3100번 포트만 엄격히 준수하도록 강제. 다른 포트로의 임의 할당 방지.
- **[P1] 컴포넌트 고유성 및 상태 충돌 방지 (REVIEW 반영)**
  - 대상 파일: `web/src/App.tsx` (또는 `web/src/components/Toast.tsx`), `web/src/components/WorkflowBuilder.tsx`
  - 내용:
    - Toast 컴포넌트의 ID 발급 로직을 단순 시간/난수 조합에서 글로벌 식별자(Increment ID 또는 UUID 등)로 교체하여 식별자 충돌(Key Conflict) 방지.
    - Fallback 노드 마운트/업데이트 시 중복 경고 알림 방지를 위한 상태 제어(Flag) 추가. React Strict Mode에서의 이중 렌더링에 따른 중복 알림 발생 제어.
- **[P1] 프론트엔드 UI 레이어 및 Z-index 검증 (REVIEW 반영)**
  - 대상 파일: UI/E2E 통합 테스트 검증 스크립트 (신규 추가 검토)
  - 내용: ReactFlow 미니맵, 컨트롤 등 캔버스 위젯과 Toast 팝업 알림 간의 Z-index 계층 충돌 여부를 검증하는 프론트엔드 E2E/시각 테스트 스크립트 작성.
- **[P2] DevFlow 워크플로우 엔진 고도화 기반 작업 (SPEC 반영)**
  - 대상 파일: 백엔드 `api/app/` 하위 워크플로우 로직
  - 내용: 기존 고정형 파이프라인에서 시각적 노드 기반 워크플로우로의 확장을 고려한 `node_runs` 기록 및 저장 스키마 설계 강화.

## 2. MVP scope / out-of-scope

- **MVP Scope**
  - 프론트엔드(React/Vite) 구동 환경의 3100 포트 고정 및 충돌 시 에러 리포팅.
  - Toast 알림 중복 렌더링 현상과 ID 충돌 버그의 완벽한 해결.
  - 모달, 알림창, 캔버스 간의 Z-index 계층 분리 및 시각적 충돌 해소.
- **Out-of-scope**
  - Temporal 등 외부 대규모 오케스트레이션 엔진의 전면 도입 및 백엔드 DB 구조의 완전한 마이그레이션 (우선 프론트엔드 버그 수정 및 워크플로우 UI 안정화에 집중).
  - Github 외 타 외부 시스템 연동 확장.

## 3. Completion criteria

- `web/vite.config.ts` 실행 환경에서 3100번 포트 점유 시 3101로 우회하지 않고 런타임 에러를 뿜으며 구동이 종료됨을 확인.
- 알림 큐에 다수의 Toast가 등록되어도 고유 `id` 충돌이 발생하지 않으며, 추가/삭제 애니메이션이 부드럽게 렌더링됨.
- `WorkflowBuilder.tsx` 화면 진입 시 Fallback 노드로 인한 경고 Toast가 상태 관리 로직에 의해 1회만 정상적으로 노출됨.
- E2E 혹은 수동 시각 테스트를 통해 Z-index가 올바르게 작동하여 ReactFlow의 오버레이 요소들이 알림창을 가리지 않음이 확인됨.

## 4. Risks and test strategy

- **Risks**
  - Toast 중복 방지 로직(Flag 기반) 적용 시 생명주기(Lifecycle) 관리가 어긋나면 알림 자체가 완전히 차단되는 부작용이 발생할 수 있음.
  - 전역 Z-index 토큰값을 조정할 경우 시스템 내 기존 팝업이나 드롭다운 메뉴의 계층 구조에 영향을 미칠 우려가 있음.
- **Test Strategy**
  - **단위 테스트**: `Toast.test.tsx` 등을 업데이트하여 고유 ID 발급 함수의 무결성과 컴포넌트 독립성을 검증.
  - **E2E 및 브라우저 테스트**: 로컬 환경(3100 포트)에서 ReactFlow 화면을 띄워 일부러 Fallback 노드를 주입 후 Toast 노출 횟수(1회)를 검증. Z-index 겹침을 방지하기 위한 시나리오 기반 프론트엔드 자동화 테스트 스크립트 실행.

## 5. Design intent and style direction

- **기획 의도**: 워크플로우 기반 AI 플랫폼 내에서 사용자가 노드 상태와 에러를 혼선 없이 직관적으로 파악할 수 있도록, 예측 가능하고 안정적인 피드백 루프를 제공하는 경험.
- **디자인 풍**: 개발자 친화적인 '모던 대시보드형' 및 상호작용 중심의 '캔버스형'.
- **시각 원칙**: 
  - 상태값(성공, 진행, 실패)별로 의미론적 컬러(Semantic Color)를 적용하여 인지적 명확성을 높임.
  - UI 밀도가 높아지지 않도록 모달과 노드 사이 넉넉한 여백(Margin/Padding)을 유지.
  - 로깅과 에러 메시지 텍스트는 Monospace 폰트를 혼용하여 가독성과 전문성을 강조.
- **반응형 원칙**: 모바일 우선(Mobile-first) 규칙에 따라 작은 화면에서는 핵심 알림과 노드 요약을 세로형 카드로 배치하고, 화면이 커짐에 따라 사이드바와 넓은 캔버스 뷰를 확장하는 방식 채택.

## 6. Technology ruleset

- **플랫폼 분류**: web 및 api
- 프론트엔드(web): **React 기반 (Vite 및 React Flow 활용)** 프레임워크로 계획.
- 백엔드(api): **FastAPI 기반** 구조로 계획.

## 7. 고도화 플랜

REVIEW.md의 버그 수정 사항을 기반으로 플랫폼의 상호작용성을 자연스럽게 확장하는 인접 기능을 다음과 같이 추가합니다.

- **기능 1: Toast 알림의 상호작용 강화 (에러 노드 위치 포커싱)**
  - **근거**: 현재의 단순 텍스트 에러 알림 구조를 개선하여, 알림 발생 시 해당 워크플로우 캔버스 상의 에러 노드를 즉시 탐색할 수 있는 연결고리를 제공함으로써 오류 해결의 UX를 대폭 향상합니다.
  - **구현 경계**: `Toast` 컴포넌트 속성에 선택적 액션 버튼(Action Callback)을 추가하고, Fallback 노드 알림 발생 시 해당 버튼 클릭으로 ReactFlow의 `fitView` 또는 `setCenter` API를 호출하여 문제가 발생한 특정 노드로 화면 포커스를 부드럽게 이동시킵니다.
  
- **기능 2: 로컬 환경 구동 시 포트 점유 사전 점검 스크립트 추가**
  - **근거**: Vite의 `strictPort: true`를 적용하더라도 포트 충돌에 의한 프로세스 다운은 사용자에게 불친절한 경험이 될 수 있습니다.
  - **구현 경계**: `package.json`의 서버 시작 스크립트 전단에 가벼운 포트 검사 유틸리티(CLI 스크립트)를 추가하여, 실행 전 3100 포트가 점유되어 있다면 터미널에 명확한 한국어 안내 메시지("이미 3100 포트를 점유 중인 프로세스가 있습니다.")를 출력하고 스크립트를 조기 종료함으로써 개발자 경험(DX)을 개선합니다.
