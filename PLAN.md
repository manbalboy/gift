# PLAN

## 1. Task breakdown with priority

### 우선순위: P0 (Critical - REVIEW.md 기반 버그 및 취약점 수정)
1. **ReDoS 취약점 대응 및 입력 문자열 제한 (API)**
   - 대상: `api/app/services/system_alerts.py`
   - 내용: `_sanitize_string` 등 텍스트 검열 로직에서 정규표현식을 실행하기 전, 입력 문자열의 길이를 선제적으로 10,000자 이내로 제한(Truncate)하는 방어 로직을 추가합니다.
2. **UI 텍스트 오버플로우 및 레이아웃 붕괴 방지 (Web)**
   - 대상: `web/src/components/SystemAlertWidget.tsx` 및 연관 스타일 시트
   - 내용: 긴 알림 텍스트가 연속될 때 레이아웃 영역을 이탈하지 않도록 텍스트 컨테이너에 `word-break: break-all` 및 `overflow-wrap: break-word` 속성을 강제 적용합니다.
3. **Cursor 기반 페이징 전환 및 복합 인덱스 마이그레이션 (API/DB)**
   - 대상: `api/scripts/migrations/20260305_add_system_alert_created_at_desc_index.sql`, `api/app/api/logs.py`
   - 내용: 기존의 Offset 방식 페이징을 데이터 누락 및 중복 방지에 유리한 Cursor 기반으로 전환합니다. 이를 위해 `(created_at DESC, id DESC)` 구조의 복합 인덱스를 생성하는 마이그레이션 스크립트를 추가하고 조회 쿼리를 개편합니다.
4. **테스트 커버리지 보강 (Tests)**
   - 대상: `api/tests/test_workflow_engine.py`, `web/tests/e2e/system-alert.spec.ts`, `web/scripts/test-port-timeout.sh`
   - 내용: 워크플로우 예산(Budget) 제한 경계값 초과 상황 단위 테스트를 추가하고, 모바일 뷰포트에서의 장문 텍스트 E2E 시각 테스트 시나리오를 보강합니다. 3100번 포트 다중 락 경합 시 타임아웃 해제 검증 시나리오를 통합 스크립트에 추가합니다.

### 우선순위: P1 (High - 시스템 알림 사용성 고도화 및 인접 신규 기능)
1. **시스템 알림 일괄 초기화(Clear All) 기능 구현 (API/Web)**
   - 대상: `api/app/api/logs.py`, `web/src/components/SystemAlertWidget.tsx`
   - 내용: 알림 폭증 시 화면을 빠르게 정리할 수 있도록 백엔드에 알림 숨김/삭제 처리를 위한 엔드포인트를 추가하고, 프론트엔드 대시보드 위젯에 제어 버튼을 연동합니다.
2. **[고도화 추가 기능] 시스템 알림 일괄 내보내기 (Export Logs)**
   - 근거: 오류가 폭증하여 사용자가 'Clear All'을 실행하기 전에 원인 분석용 기록을 남길 수 있도록 돕는 인접 기능입니다. 향후 Artifact 중심의 워크스페이스 확장을 위한 데이터 산출물 생성의 기반이 됩니다.
   - 구현 경계: 서버 부하를 최소화하기 위해 현재 프론트엔드에 로드된 시스템 알림 목록을 클라이언트 사이드에서 즉시 JSON 형식의 파일로 다운로드하는 유틸리티 기능과 버튼을 구현합니다.

### 우선순위: P2 (Medium - SPEC.md 기반 장기 아키텍처 확장 설계 반영)
1. **Autopilot Control Plane 및 엔진 확장 준비**
   - 내용: 장기 실행(Durable Execution) 워크플로우 엔진 기반을 다지기 위해, 알림/로그 시스템이 안정화된 이후 `workflow_id` 기반의 실행 내역 및 `node_runs` 저장 스키마 확장을 순차적으로 진행합니다.

## 2. MVP scope / out-of-scope

### MVP Scope
- `REVIEW.md`에 명시된 서비스 거부(ReDoS) 취약점 방어 및 프론트엔드 뷰포트 레이아웃 붕괴 버그 수정.
- 대량의 로그 데이터 적재 환경에서 일관성 있는 조회를 보장하는 Cursor 기반 페이징 쿼리 변경 및 복합 인덱스 DB 마이그레이션.
- 알림 대시보드의 데이터를 제어할 수 있는 일괄 초기화(Clear All) 및 로그 내보내기(Export) UI 제공.
- 백엔드 워크플로우 반복 제한 단위 테스트, 모바일 UI 레이아웃 E2E 테스트, 쉘 스크립트 동시성 락 경합 시나리오 등 필수 테스트 커버리지 달성.
- 로컬 실행 환경 테스트 시 3000번대 포트(프론트엔드: 3000번, API: 3001번)를 사용하여 충돌을 방지합니다.

### Out-of-scope
- 완전한 형태의 ReactFlow 기반 Visual Workflow Builder UI 및 엔진 개발은 이번 시스템 안정화 MVP 범위에서 제외됩니다.
- Temporal, LangGraph 등 외부 워크플로우 엔진 프레임워크의 직접적인 연동 로직 구현은 장기 목표로 연기합니다.

## 3. Completion criteria

- **보안/안정성:** 길이가 20,000자 이상인 악의적인 텍스트 페이로드가 인입되더라도, 서버의 정규표현식 검열 프로세스가 병목 없이 10,000자로 텍스트를 자른 뒤 200 OK 응답을 500ms 이내에 반환해야 합니다.
- **UI 반응성:** 너비 320px 모바일 뷰포트에서 연속된 문자로 구성된 극단적 길이의 로그를 출력할 때, `SystemAlertWidget` 영역 밖으로 텍스트가 넘치거나 가로 스크롤바가 생기지 않아야 합니다.
- **기능 일관성:** 수십 건의 로그가 동일한 밀리초(ms)에 삽입된 환경에서 커서(Cursor)를 통해 다음 페이지를 요청했을 때, 반환되는 로그 항목에 중복이나 누락이 없음을 데이터베이스 조회 결과와 비교하여 확인해야 합니다. 일괄 초기화 버튼 클릭 즉시 화면의 로그가 비워져야 합니다.
- **테스트 통과:**
  - `pytest`가 예산(Budget) 한계 경계값 테스트에서 성공(Passed)해야 합니다.
  - 모바일 뷰포트 레이아웃을 검증하는 Playwright E2E 테스트가 성공해야 합니다.
  - `test-port-timeout.sh` 스크립트를 백그라운드 멀티 프로세스로 10회 이상 반복 실행해도 데드락이나 포트 무한 점유 현상 없이 정상적으로 락을 반환해야 합니다.

## 4. Risks and test strategy

### Risks
- 페이징 로직 전환 과정에서 기존 프론트엔드의 상태 관리 데이터(Offset 기반)와 호환성 충돌이 발생하여 로그 조회가 멈추거나 무한 스크롤 호출 루프에 빠질 위험이 있습니다.
- 백그라운드 락 스크립트 테스트 시 구동되는 CI 컨테이너 혹은 로컬 PC의 CPU 할당량 차이로 인해 미세한 타이밍이 어긋나 간헐적인 테스트 실패(Flaky Test)가 나타날 수 있습니다.

### Test Strategy
- 페이징 API 전환 시 백엔드 단위 테스트에서 15개의 목업(Mock) 로그를 동시간대로 강제 삽입한 후, `limit=5` 조건으로 커서를 이동하며 데이터 고유 ID 세트가 겹치지 않는지 엄격하게 Assert 합니다.
- 프론트엔드 모바일 UI E2E 테스트는 의도적으로 띄어쓰기가 없는 1,000자의 임의 문자열을 발생시킨 뒤 요소의 `scrollWidth`와 `clientWidth`를 비교하여 레이아웃 초과 여부를 프로그래매틱하게 검증합니다.
- 포트 락 스크립트 테스트는 프로세스 동시 실행 전후로 충분한 허용 Sleep 시간을 두어 OS 환경에 따른 타이밍 차이 리스크를 완화하고, 스크립트가 강제 종료될 때 트랩(Trap)이 정상 작동하는지 여부를 핵심으로 검증합니다.

## 5. Design intent and style direction

- **기획 의도:** "넘쳐나는 시스템 알림 속에서도 길을 잃지 않는, 명확하고 안정적인 컨트롤 패널"
  폭증하는 오류나 긴 로그에도 대시보드가 흔들림 없이 동작하여, 사용자가 스트레스 없이 시스템 상황을 정확히 인지하고 관리 행위(초기화/내보내기)를 즉각 수행할 수 있도록 지원합니다.
- **디자인 풍:** **대시보드형 미니멀리즘 (Dashboard Minimal)**
  불필요한 장식을 배제하고 기계적인 데이터의 가독성을 최우선으로 하는 모던 유틸리티 스타일을 채택합니다.
- **시각 원칙:**
  - 에러(Red), 경고(Yellow), 일반 로그(Gray)의 상태 구분을 위한 포인트 컬러를 유지합니다.
  - 시스템 로그 및 스택 트레이스는 고정폭 폰트(Monospace)를 사용하여 기계적 가독성을 높입니다.
  - 아이템 구분을 위해 면적 중심의 마진 대신 슬림한 하단 1px 보더 라인과 타이트한 패딩(8~12px)을 사용하여 컴팩트한 화면 정보 밀도를 구성합니다.
- **반응형 원칙:** **모바일 우선 규칙 (Mobile-First)**
  최소 지원 너비(320px) 뷰포트에서도 화면을 절대 뚫고 나가지 않도록 줄바꿈을 강제하며, 필수 조작 버튼(Clear All, Export)은 작은 화면에서 아이콘 형태로 축소하여 접근성을 보장합니다.

## 6. Technology ruleset

- **플랫폼 분류:** Web 및 API (대시보드 관리 도구)
- **Frontend (Web):**
  - 프레임워크: React 기반 프레임워크 (Vite + TypeScript)를 사용하여 구축.
  - 실행 포트: 개발서버 기준 `3000`번 고정 사용.
- **Backend (API):**
  - 프레임워크: FastAPI 기반의 고성능 API 서비스.
  - 실행 포트: 개발서버 기준 `3001`번 고정 사용.
  - 데이터베이스: SQLite 엔진 위에서 테스트되며, 복합 인덱스 생성을 포함한 Alembic 마이그레이션 스크립트를 필수적으로 적용합니다.
