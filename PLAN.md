# PLAN

## 1. Task breakdown with priority

**Phase 1: 백엔드 핵심 버그 수정 및 안정화 (P0)**
- `api/app/api/webhooks.py`: 잘못된 `workflow_id` 및 페이로드 수신 시 HTTP 422 상태 코드 반환을 위한 예외 처리 로직 추가
- `api/app/services/workflow_engine.py`:
  - 워크플로우 강제 취소 시 발생하는 `asyncio.CancelledError`의 정상 핸들링 및 SSE 커넥션 비동기 제너레이터 종료 로직 보강 (메모리 누수 방지)
  - DAG 워크플로우 순회 로직 내에서 이전 노드와 연결(edge)이 없는 독립 노드가 병렬 처리 큐에 정상적으로 진입하여 실행되도록 수정

**Phase 2: 보안 인가 처리 및 데이터 무결성 확보 (P1)**
- `api/app/api/workflows.py`: 휴먼 게이트(승인/수정/거절) API 엔드포인트에 Role 기반 인가 시스템(RBAC) 도입 및 권한 부족 시 HTTP 403 에러 반환 구현
- `api/app/models/` 및 데이터베이스 스키마: 휴먼 게이트 결정 기록용 Audit Log 테이블 신설 (결정 주체, 결정 시간, 페이로드 저장) 및 로직 적용

**Phase 3: 프론트엔드 연동 및 사용자 경험 개선 (P1)**
- 포트 고정: 실행 스크립트에서 프론트엔드 포트를 `3100`, API 포트를 `3101`로 명시적 분리 및 고정
- `web/src/services/` 등: 클라이언트의 SSE 커넥션 상태를 모니터링하고, 단절 시 지수적 백오프(Exponential Backoff)를 활용해 서버 부하를 줄이며 자동 재연결하는 기능 구현
- `web/src/components/`: Human Gate 접근 시 403 에러를 핸들링하여 권한 부족을 알리는 친화적인 Fallback 모달 UI 추가
- `web/src/components/`: Human Gate의 과거 의사결정 이력(Audit Log)을 Read-Only 형태로 확인할 수 있는 전용 모달 컴포넌트 추가

**Phase 4: 품질 보증 및 테스트 자동화 (P2)**
- 백엔드 (`api/tests/`): 강제 취소 자원 반환 확인, 422 밸리데이션 예외, 403 인가 에러 등을 검증하는 Pytest 단위 및 통합 테스트 작성
- 프론트엔드 (`web/tests/e2e/`): Playwright를 활용해 `http://localhost:3100` 환경에서 휴먼 게이트 승인/반려 절차 및 에러 발생 흐름을 검증하는 E2E 테스트 스크립트 구축

**Phase 5: 고도화 플랜 (TODO 리뷰 반영) (P3)**
- **동시 승인/반려 충돌(Race Condition) 방지 (근거: 휴먼 게이트 데이터 무결성 보장)**: 다수의 리뷰어가 거의 동시에 같은 게이트에 대해 승인 혹은 반려를 요청할 때, 상태가 의도치 않게 덮어쓰이는 문제를 방지하기 위해 DB 트랜잭션 락(Lock) 및 버전 관리 제어 기법을 도입.
- **재연결 상태 UI 인디케이터 제공 (근거: 백오프 로직 도입에 따른 사용자 경험 보완)**: 지수적 백오프로 재연결을 시도하는 동안 현재 네트워크/연결 상태를 프론트엔드 상단 배너 혹은 인디케이터로 노출하여 사용자의 오조작을 방지.

## 2. MVP scope / out-of-scope

**MVP Scope**
- Workflow Engine v2 및 Human Gate의 기존에 구현된 기능의 치명적인 결함(메모리 누수, 병렬 DAG 오류) 수정.
- HTTP 예외 처리(422, 403) 및 Audit Log DB 기록, 이를 활용한 프론트엔드 UI/UX 대응 모달 적용.
- SSE의 지수적 백오프 처리로 클라이언트 재연결 안정성 확보.
- 상기 기능들을 보증하기 위한 백엔드 및 프론트엔드 통합/E2E 테스트 추가.

**Out-of-scope**
- Visual Workflow Builder (ReactFlow 기반 캔버스)의 전면 개발 및 편집 기능.
- 복잡한 Agent SDK 마켓플레이스의 구축.
- GitHub 이외의 광범위한 CI/CD/Deploy 이벤트 버스 시스템 연동 확장.
- 휴먼 게이트 밖의 복잡한 실시간 채팅형 리뷰 기능 추가.

## 3. Completion criteria

- 잘못된 웹훅 페이로드 전송 시 HTTP 422 예외를 반환하고 있는가?
- 워크플로우 강제 취소 명령이 서버 메모리나 연결된 스트림의 좀비 프로세스 없이 자원을 깨끗하게 해제하는가?
- 의존성 없는 단일 노드가 DAG 엔진에서 누락 없이 정상적으로 큐에 등록되고 실행되는가?
- 로컬 환경 실행 시 프론트엔드는 `3100`, 백엔드 API는 `3101`에서 정상 동작하는가?
- 권한 없는 사용자가 휴먼 게이트 조작 시 403 에러가 반환되며, 프론트엔드에 Fallback 모달이 뜨는가?
- 승인/반려 행위가 주체와 시간 정보를 포함하여 Audit DB에 기록되며, 프론트엔드 UI에서 조회 가능한가?
- 서버 다운 모사 시, 프론트엔드에서 SSE 재연결 간격이 기하급수적으로 늘어나는 지수적 백오프가 작동하는가?
- 신규 작성된 백엔드 Pytest 및 프론트엔드 Playwright E2E 테스트가 100% 통과하는가?

## 4. Risks and test strategy

**Risks**
- **SSE 재연결 집중(Thundering Herd)**: 서버가 복구되었을 때, 다수의 클라이언트가 동시에 재연결을 시도하여 트래픽 부하가 발생할 수 있습니다. 지수적 백오프에 임의의 Jitter(난수 지연 시간)를 더해 요청을 분산시키는 전략이 필요합니다.
- **상태 전이 충돌**: 휴먼 게이트의 동시 접근에 대한 낙관적/비관적 잠금을 적용하지 않을 경우 승인과 동시에 취소되는 등 오염된 데이터가 적재될 위험이 있습니다.

**Test strategy**
- **비동기 취소/자원 관리 테스트**: Mocking을 통해 `asyncio.CancelledError`를 의도적으로 유발한 후 활성 태스크 수 및 메모리 반환 객체를 검증하여 좀비 커넥션이 없음을 확인.
- **권한 기반 API 테스트**: Pytest 인가 테스트 스위트를 구성하여 각 역할군별(Admin, Reviewer, 일반 사용자) API 엔드포인트 응답 상태 코드 검증.
- **E2E 흐름 테스트**: 로컬 프론트엔드 포트(`3100`) 환경을 띄운 상태로 Playwright를 동작시켜 권한이 있는 경우와 없는 경우, 서버 단절 후 복구 등 UI상호작용에 따른 결과 화면을 캡처하고 단언(Assertion) 검증 진행.

## 5. Design intent and style direction

- **기획 의도**: 워크플로우를 관측하고 검토하는 담당자가 신뢰성을 느낄 수 있도록 안정적이고 명확하게 권한, 이력, 에러 상태를 안내하는 것이 목적입니다.
- **디자인 풍**: 불필요한 장식을 배제한 모던한 대시보드형 및 카드형 UI. 기능 중심의 미니멀 스타일로 설계하여 핵심 지표와 상태에만 집중하도록 합니다.
- **시각 원칙**:
  - **컬러**: 중립적인 화이트/그레이 배경 위에 상태를 나타내는 색상(승인: Green, 반려: Red, 대기/경고: Yellow)을 포인트로 적용합니다.
  - **패딩/마진**: 넓은 여백(최소 16px 이상)을 활용하여 Audit Log, 오류 메시지 모달이 화면 내에서 명확한 그룹으로 인지되도록 구성합니다.
  - **타이포**: 시스템 폰트 기반의 가독성 높은 산세리프 폰트를 사용하며, 중요 정보인 상태나 결정 시간은 굵기와 색상으로 명도 대비를 부여합니다.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 적용하여 화면이 좁은 모바일에서는 단일 칼럼 구조의 카드 UI로 표현하고, 태블릿이나 데스크탑 환경에서는 다중 뷰 패널 구조로 확장합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web 및 api
- **web**: React 또는 Vite 기반의 프레임워크 사용. API 통신 및 모달/뷰 컴포넌트를 설계하고 `3100` 포트에서 실행.
- **api**: FastAPI 기반의 비동기 서버 모델 채택. Postgres 및 SQLAlchemy 등 기존 인프라에 맞춰 Audit 로직 적용 후 `3101` 포트에서 실행.
