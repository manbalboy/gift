# PLAN

## 1. Task breakdown with priority

**[P0] 보안 강화 및 스크롤 안정화**
- **XSS 방지 처리**: `SystemAlertWidget` 내 로그 출력 시 악성 스크립트 실행 방지를 위한 텍스트 살균(Sanitization) 또는 엄격한 DOM 이스케이프 적용.
- **스크롤 계산 오차 보정**: 브라우저 줌 및 배율 환경(OS 디스플레이 확대 등)에서 발생하는 소수점 단위 오차 방지를 위해 최하단 판별 로직에 `Math.ceil()` 또는 `Math.floor()` 적용.

**[P1] 필터링 및 엣지 케이스 UX 개선**
- **스크롤 튀김 현상 방지**: 필터 칩(`All`, `Error`, `Warning`) 상태 전환 시 기존 스크롤 위치로 인해 허공을 가리키거나 레이아웃이 깨지는 것을 막기 위해 목록 컨테이너의 `scrollTop` 위치를 `0`으로 명시적 초기화.
- **Empty State UI 구현**: 특정 필터 선택 후 표시할 데이터가 없을 경우 명확한 상태 피드백을 제공하는 빈 화면(Empty State) 마크업 렌더링.
- **백그라운드 탭 동기화 방어**: 브라우저 탭 비활성화 후 복귀 시(`visibilitychange` 이벤트 등) 렌더링 지연으로 인해 어긋나는 스크롤 위치 및 상태(`isAutoScrollPaused`) 재동기화 방어 로직 적용.

**[P2] 테스트 보강 및 환경 격리**
- **포트 충돌 방지**: 로컬 개발 서버 및 Playwright E2E 테스트 재현 포트를 3100번대(`3100`)로 고정 (`playwright.config.ts`, `vite.config.ts`, `package.json` 설정 업데이트).
- **성능 검증 테스트**: 대량의 데이터 상태에서 `useMemo` 기반 필터링 연산 시 브라우저 메인 스레드 블로킹이나 프레임 드랍이 없는지 검증하는 단위 및 성능 테스트 작성.
- **OS별 스크롤 환경 테스트**: 휠 가속도 등 운영체제(Windows/Mac) 간 발생 차이를 고려한 스크롤 이벤트 Throttle 정상 작동 여부 E2E 시나리오 테스트 추가.

- **추가 기능 2: 주요 키워드(문서 URL, 티켓 번호) 하이라이팅 파서**

  - **구현 경계**: 복잡한 전면 마크다운 지원을 배제하고, `http/https` 링크와 사전에 정의된 특정 에러 코드 패턴에 대해서만 안전한 외부 앵커(`<a>`) 태그나 칩 형태로 제한적 변환.

## 2. MVP scope / out-of-scope

**MVP Scope**
- `SystemAlertWidget`의 브라우저 확대/축소 환경 하 스크롤 감지 정확도 보장 및 XSS 보안 취약점 해결.
- 필터 칩 전환 시 스크롤 포지션 즉시 초기화 및 가시적인 빈 화면(Empty State) 피드백 제공.
- 백그라운드 탭 진입 및 복귀 시 스크롤 상태 재동기화 보정 로직 구현.
- E2E 및 로컬 개발 포트를 3100번대(`3100`)로 고정하여 독립적인 테스트 및 실행 환경 격리.

**Out-of-scope**
- 백엔드(FastAPI) 이벤트 브로드캐스트 로직 변경 또는 알림 생성 API의 비즈니스 구조 개편.
- 이메일, 메신저 등 새로운 외부 채널 연동.
- UI/UX 다국어 번역(i18n) 대응 작업.

## 3. Completion criteria

- 임의의 악성 스크립트가 삽입된 로그가 시스템 알림으로 수신되더라도 브라우저에서 실행되지 않고 안전한 텍스트 렌더링으로 방어되어야 함.
- 브라우저 디스플레이 비율이 125%, 150% 등 소수점 픽셀 환경이더라도 스크롤 이벤트의 최하단 감지 오차가 발생하지 않아야 함.
- 목록 필터를 변경했을 때 스크롤 위치가 허공에 머무르거나 튀는 현상 없이 즉시 최상단(`scrollTop: 0`)으로 이동해야 함.
- Playwright E2E 환경 및 로컬 개발 서버가 설정된 `3100` 포트에서 문제없이 독립 실행 및 테스트 통과를 달성해야 함.
- 빈 데이터 배열에서 필터를 조작하더라도 에러가 발생하지 않으며 명시적인 Empty State UI 컴포넌트가 렌더링되어야 함.

## 4. Risks and test strategy

**Risks**
- OS(Windows/Mac)에 따른 스크롤바 영역 차지 비율 및 마우스 휠 가속도 설정 차이로 인하여 스크롤 이벤트 발생 주기가 달라, 특정 기기에서 Throttle 타이밍이 간헐적으로 어긋날 수 있음.
- 백그라운드 탭 전환 시 브라우저 내부적으로 DOM 업데이트 지연을 처리하는 방식이 벤더(Chrome, Safari 등)별로 달라 탭 복귀 시 동기화 타이밍 차이가 발생할 위험.

**Test Strategy**
- **보안 검증 테스트 (Unit Test)**: 렌더링 요소에 XSS 공격 스크립트를 주입하고 브라우저 파서가 이를 철저히 이스케이프 또는 살균했는지 단위 테스트 레벨에서 단언(Assert).
- **성능 한계 테스트 (Unit Test)**: 대규모(10,000건 이상) 가상 데이터를 렌더링 컨테이너에 밀어 넣은 후, 필터 전환 시 연산 시간 및 프레임 지연(Time elapsed)을 측정.
- **크로스 환경 E2E 테스트 (Playwright)**: 브라우저 뷰포트 배율을 강제로 조정하고 스크롤 휠 이벤트를 지속 발생시켜, 자동 스크롤(Auto-scroll) 상태가 오판되지 않고 끝까지 유지되는지 검증.
- **가시성 수명주기 테스트**: 브라우저 탭의 상태를 Playwright API를 이용해 강제로 `hidden` 상태에서 `visible` 상태로 전환하며, `visibilitychange` 이벤트 이후 최신 뷰포트 위치로 DOM 스크롤 상태가 동기화되는지 검증.

## 5. Design intent and style direction

- **기획 의도**: 지속적으로 누적되는 대규모 시스템 알림 속에서도 사용자가 위급한 장애 요소를 놓치지 않고 통제 가능한 형태의 직관적인 모니터링 경험을 제공한다.
- **디자인 풍**: 개발자와 시스템 관리자 친화적인 대시보드 뷰이며, 불필요한 장식을 배제한 컴팩트하고 모던한 콘솔(Console) 스타일.
- **시각 원칙**:
  - **컬러**: 정보의 피로도를 낮추기 위해 무채색 모노톤 배경을 기본으로 채택하고, 심각도를 알리는 `Error`(적색 계열)와 `Warning`(황색 계열) 필터 레이블이나 아이콘에만 명확한 포인트 컬러를 활용한다.
  - **패딩/마진**: 높은 정보 밀도 유지를 위해 콤팩트한 패딩(Dense padding)과 간격을 배치하여, 한눈에 많은 로그 데이터를 파악할 수 있도록 한다.
  - **타이포그래피**: 에러 메시지나 스택 트레이스의 정밀한 가독성을 위해 고정폭(Monospace) 폰트를 일관되게 적용하여 텍스트 포맷과 정렬을 보장한다.
- **반응형 원칙**: 모바일 우선(Mobile-first). 모바일과 같은 좁은 화면에서는 좌우 횡스크롤을 유발하지 않게 긴 텍스트를 자동 줄바꿈(Word-wrap)하고, 데스크톱 등의 대형 화면에서는 대시보드 그리드에 맞추어 유동적(Fluid)으로 너비가 채워지도록 레이아웃을 전개한다.

## 6. Technology ruleset

- **플랫폼 분류**: web
- **프론트엔드 기반 기술**: React (사용자 인터페이스 및 상태 관리)
- **빌드 및 개발 도구**: Vite (빠른 HMR 및 빌드 체인), Playwright (E2E 자동화 테스트 구동)
- **네트워크 및 실행 규칙**: 로컬 개발과 E2E 테스트 환경에서의 네트워크 충돌 방지를 위해 3000번대 포트 중 `3100` (`http://localhost:3100`) 포트를 사용하도록 명시적 고정.
