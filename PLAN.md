  - 경계: 요청 상태가 `approval_pending`일 때만 유효한 단일 엔드포인트(`POST /api/approvals/{id}/cancel`) 구현 및 UI 내 텍스트 버튼 형태까지만 범위를 제한.

## 2. MVP scope / out-of-scope

**MVP Scope**
- 클라이언트 빌드 산출물에서 웹훅 시크릿 환경 변수 참조 완전 제거.
- 휴먼 게이트 승인 중복 요청에 대한 멱등성 보장 로직 구현.
- 단일 노드 워크플로우 상태 전이(`running` -> `done`) 버그 수정.
- Audit Log 조회 API의 Limit/Offset 페이지네이션 및 필터 파라미터 적용.
- Playwright 기반 프론트엔드 오프라인/온라인 상태 E2E 테스트 통합.

**Out-of-scope**
- 정적 토큰 방식을 대체하는 전면적인 사용자 세션 및 JWT 기반 인증 체계 도입 (추후 보안 강화 과제로 분리).
- Audit Log 데이터 보관 주기를 관리하는 백그라운드 아카이빙 배치 작업.
- 오프라인 상태에서의 복잡한 클라이언트 로컬 캐싱 및 데이터 동기화 기능.

## 3. Completion criteria

- 프론트엔드 빌드 후 번들 파일(`dist/` 등)을 텍스트로 검색했을 때 웹훅 시크릿 관련 키워드가 검출되지 않아야 함.
- 휴먼 게이트 승인 API에 동일한 페이로드로 2회 연속 요청을 보냈을 때 모두 HTTP 200 상태 코드를 정상 반환해야 함.
- 엣지가 없는 단일 노드 워크플로우를 실행하고 데이터베이스를 조회했을 때 워크플로우 상태가 즉시 `done`으로 기록되어야 함.
- Audit Log 데이터 100건 생성 후 `limit=10` 파라미터로 요청 시 정확히 10건의 리스트와 `total_count` 메타데이터가 반환되어야 함.
- CLI 환경에서 `npm run test:e2e` 실행 시 오프라인 배너 관련 테스트 케이스가 오류 없이 통과해야 함.

## 4. Risks and test strategy

**Risks**
- 클라이언트 코드에서 환경 변수가 제거되면서 기존 로직의 의존성이 깨져 프론트엔드 빌드 에러나 런타임 통신 불가 상태가 발생할 수 있음.
- 중복 요청 멱등성 처리 시 DB 트랜잭션 충돌이나 Lock 대기로 인한 데드락 및 동시성 문제가 발생할 위험이 있음.

**Test Strategy**
- **정적/빌드 테스트**: 린터와 텍스트 검색을 통해 시크릿 노출 여부를 빌드 파이프라인 레벨에서 검증.
- **단위 테스트(Unit Test)**: 백엔드 `pytest`를 활용하여 단일 노드 워크플로우의 실행 및 종료 트랜잭션을 고립된 환경에서 철저히 검증.
- **통합/동시성 테스트(Integration Test)**: 애플리케이션 레벨의 Lock(`lock_provider`)이 동작하는 환경에서 휴먼 게이트 승인 API를 병렬로 다수 호출하여 데드락 여부와 멱등성을 확인.
- **E2E 테스트(Playwright)**: 네트워크 단절 상황을 강제로 모사하고, UI 상에서 사용자에게 올바른 안내 배너가 렌더링되는지 엔드투엔드로 확인.

## 5. Design intent and style direction

- **기획 의도**: 워크플로우 플랫폼의 기초 보안 결함을 제거하고 성능과 안정성을 높여, 대규모 로그 데이터나 네트워크 예외 상황에서도 사용자가 흔들림 없이 개발 흐름을 관측할 수 있는 신뢰도 높은 경험을 제공한다.
- **디자인 풍**: 모던하고 미니멀한 대시보드형 디자인을 지향. 시각적인 장식을 배제하고 데이터의 가독성과 인지성을 최우선으로 배치한다.
- **시각 원칙**:
  - 오프라인이나 에러 상황을 알리는 배너는 시인성이 높은 컬러(예: Error Red 또는 Warning Amber)를 사용하되, 기존 작업 영역의 레이아웃을 밀어내지 않도록 최상단 플로팅 형태로 구성한다.
  - Audit Log 리스트는 행간과 셀 패딩을 여유롭게 적용하여 텍스트 뭉침을 방지하고, 로그 메시지 영역은 고정폭(Monospace) 타이포그래피를 적용하여 시스템적인 느낌과 가독성을 살린다.
- **반응형 원칙**: 모바일 우선(Mobile-first) 규칙을 적용한다. 복잡한 로그 테이블 뷰는 화면이 좁아질 경우 좌우 스크롤을 허용하거나 카테고리별로 압축된 카드형 리스트로 유연하게 강등(Graceful Degradation)되도록 설계한다.

## 6. Technology ruleset

- **플랫폼 분류**: web 및 api
- **web**: React 기반 프레임워크와 Vite 생태계를 활용하여 계획. 기존 컴포넌트 구조와 상태 관리 방식을 일관되게 유지.
- **api**: FastAPI 기반 백엔드로 계획. 비동기 I/O 및 데이터베이스 세션을 최적화하여 페이지네이션 로직 및 동시성 처리를 통합.
- **환경 구성 규칙**: 로컬 실행 시 충돌을 방지하기 위해 실행 가이드 및 도커 매핑 포트는 반드시 3000번대 포트(예: web은 3000, api는 3001)만 사용한다.
