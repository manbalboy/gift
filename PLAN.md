# PLAN

## 1. Task breakdown with priority

1. **ReDoS 방지 (API)**: `api/app/services/system_alerts.py`의 `_sanitize_string` 함수 내 정규표현식 실행 전 문자열 길이 제한(10,000자 Truncate) 방어 로직 추가.
2. **UI 레이아웃 붕괴 수정 (Web)**: `web/src/components/SystemAlertWidget.tsx` 컨테이너 및 연관 CSS에 `word-break: break-all` 및 `overflow-wrap: break-word` 속성 적용.
3. **페이징 안정성 확보 (API/DB)**: 
   - `api/scripts/migrations/20260305_add_system_alert_created_at_desc_index.sql` 마이그레이션 적용.
   - `api/app/api/logs.py` 페이징 로직을 복합 인덱스`(created_at DESC, id DESC)` 기반 Cursor 방식으로 마이그레이션.

### [P1] 자동화 및 단위/E2E 테스트 보완
1. **무한 루프 제어 테스트 (API)**: `api/tests/test_workflow_engine.py` 내 워크플로우 예산(Budget) 한계 경계값 초과 상황에 대한 테스트 케이스 작성.
2. **모바일 E2E 검증 (Web)**: `web/tests/e2e/system-alert.spec.ts` 내 모바일 해상도(320px) 기준 장문 텍스트 인입 시 레이아웃 무결성(붕괴 여부) 프로그래매틱 검증 추가.
3. **포트 경합 시나리오 보강 (Scripts)**: `web/scripts/test-port-timeout.sh` 내 다중 백그라운드 프로세스 실행 시 락 경합 시나리오(포트 3100 기준) 및 데드락 해제 트랩 검증 코드 보강.

### [P2] 로그 관리 UX 개선
1. **시스템 알림 초기화 UI (Web/API)**: 프론트엔드 대시보드 위젯에 일괄 초기화(Clear All) UI 버튼 추가 및 백엔드 숨김/삭제 API 연동.
2. **로그 내보내기 (Web)**: 클라이언트 사이드에 로드된 시스템 알림 데이터를 즉각적으로 JSON 파일로 다운로드할 수 있는 유틸리티(Export Logs) 구현.

### [P3] 고도화 플랜 (Advanced Plan)
1. **알림 필터링 (Client-side)**
   - **근거**: 쏟아지는 대량의 시스템 알림 속에서 중요한 에러를 인지하기 위해 Clear All 이전에 특정 상태(Error/Warning)만 골라보는 기능이 필요합니다. 기존 시스템 알림 UX 개선(Clear All)과 자연스럽게 이어지는 기능입니다.
   - **구현 경계**: 백엔드 API 추가 없이 클라이언트 측에 로드된 로그 배열 내에서 필터링 칩(Chip) 컴포넌트로 구현합니다.
2. **자동 스크롤 중지 및 재개 (Auto-scroll Pause)**
   - **근거**: 실시간으로 로그가 적재될 때 사용자가 과거 로그를 확인하거나 Export하기 위해 스크롤을 올리면, UI가 강제로 최하단으로 튕기는 현상을 막아 UX 품질을 보장해야 합니다.
   - **구현 경계**: 사용자가 스크롤을 위로 올리면 자동 스크롤 상태를 해제하고, 다시 최하단으로 내릴 때 활성화하는 플래그 로직을 `SystemAlertWidget.tsx`에 추가합니다.

## 2. MVP scope / out-of-scope

### MVP Scope
- 프론트엔드 모바일 뷰포트 환경 내 장문 텍스트 렌더링 시 레이아웃 붕괴 방지 조치.
- 정규표현식 기반 서비스 거부(ReDoS)를 유발할 수 있는 악의적 텍스트 페이로드 사전 차단(Truncate).
- 대량의 시스템 알림 환경에서 로그 조회 시 발생하는 누락/중복 결함을 막기 위한 Cursor 기반 페이징 적용.
- 사용자의 인지 과부하 방지를 위한 즉각적인 화면 초기화(Clear All) 및 오프라인 분석용 로그 내보내기(Export Logs) 제공.
- 프론트엔드, API 백엔드, 로컬 락 경합과 관련된 테스트 커버리지 확장.

### Out-of-Scope
- `SystemAlertWidget` 외 타 대시보드 위젯에 대한 전면 리팩토링.
- 실시간 알림을 위한 WebSocket이나 SSE(Server-Sent Events) 아키텍처 개편(기존 방식 유지).
- Cursor 페이징 외에 타 API 엔드포인트에 대한 대대적인 구조 변경 및 DB 전체 인덱스 튜닝.
- 알림 내보내기 포맷 다각화(CSV, PDF 등 지원 보류, 오직 JSON만 제공).

## 3. Completion criteria
- `system_alerts.py`에 길이가 20,000자인 텍스트가 인입되었을 때, 10,000자로 잘리며 서버 지연 없이 정상 반환되는지 확인 완료.
- 모바일 해상도(320px) 환경에서 띄어쓰기 없는 긴 텍스트 알림 렌더링 시, 가로 스크롤바가 발생하지 않고 다음 줄로 안전하게 넘어가는지 확인 완료.
- 동일한 타임스탬프를 가진 100건의 로그 데이터를 Cursor 페이징으로 반복 조회할 때, 누락된 항목이나 중복 노출 항목이 단 하나도 발견되지 않는지 데이터 정합성 검증 완료.
- "Clear All" 버튼 동작 시 화면 내 알림이 즉각 사라지며, "Export Logs" 동작 시 현재 조회된 로그가 정확한 JSON 포맷으로 브라우저 다운로드 처리되는지 확인 완료.
- 로컬 개발 환경에서 API 서버를 `3101` 포트, Web 프론트엔드를 `3100` 포트로 실행했을 때 포트 충돌 및 락 대기 문제가 발생하지 않는지 확인 완료.

## 4. Risks and test strategy

### Risks
- **DB 페이징 방식 변경**: 기존 Offset 페이징에서 Cursor 기반으로 변경되면서, 기존 API 응답 구조를 기대하는 클라이언트의 사이드 이펙트나 호환성 파괴 발생 가능성.
- **다중 락 경합 이슈**: 개발 및 CI 환경에서 병렬 테스트 실행 시 포트 락 반환 실패로 인한 데드락이나 파이프라인 지연 위험.

### Test Strategy
- **Cursor 페이징 정합성 테스트**: 동일 밀리초(ms) 단위의 시스템 알림 Mock 데이터를 수십 건 동시 주입한 뒤, 페이지 당 제한 건수로 API를 반복 호출해 조회된 ID 목록이 총합 데이터와 100% 일치하는지 백엔드 단위 테스트 수행.
- **보안 (ReDoS) 방어 성능 테스트**: 무의미하게 길게 작성된 텍스트 스트림을 페이로드로 전송하고 API 응답 처리 시간이 1초 이내로 유지되는지 확인하는 스트레스 시나리오 반영.
- **E2E UI 픽셀 검증**: Playwright를 이용해 320px 뷰포트를 강제 세팅한 뒤, 레이아웃 컨테이너의 가로 폭이 화면 폭을 초과하는지 여부를 프로그래매틱하게 Assert.
- **스크립트 동시성 테스트 보강**: `web/scripts/test-port-timeout.sh`를 여러 백그라운드 프로세스로 동시 백그라운드 호출 후, 락이 데드락 없이 안전하게 순차 해제되는지 트랩 검증 테스트 추가.

## 5. Design intent and style direction
- **기획 의도**: 지속적인 워크플로우 에이전트 동작으로 발생하는 대량의 시스템 로그 속에서, 개발자나 운영자가 데이터의 누락 및 브라우저 병목 없이 투명하고 안정적으로 관측할 수 있는 시스템 뷰를 제공합니다.
- **디자인 풍**: 개발자 중심의 모던하고 밀도 높은 대시보드형 (터미널 로그 뷰어와 유사한 정보 구조 지향).
- **시각 원칙**: 불필요한 장식을 배제하고, 성공/에러/경고 등 로그 상태별 직관적인 포인트 컬러만 최소한으로 적용. 코드 및 로그 가독성을 극대화하기 위해 고정폭 폰트(Monospace)와 넉넉한 줄간격을 유지합니다.
- **반응형 원칙**: 모바일 우선(Mobile-first). 모바일 환경의 제한된 너비에서는 텍스트가 어떤 조건에서도 컨테이너를 이탈하지 않도록 엄격한 줄바꿈 규칙을 적용하며, 데스크탑 환경에서는 로그 타임라인이 화면 전체로 쾌적하게 확장됩니다.

## 6. Technology ruleset
- **플랫폼 분류**: web / api
- **web**: React 라이브러리 기반 (Vite 빌드, TypeScript) 및 Playwright E2E 테스트.
- **api**: FastAPI 기반 프레임워크 (Python, Pytest 단위 테스트).
- **실행 및 포트 규칙**: 로컬 실행 시 프론트엔드 Web은 `3100`, 백엔드 API는 `3101` 포트를 명시적으로 할당하여 충돌을 방지하며 실행 가이드를 작성합니다.
