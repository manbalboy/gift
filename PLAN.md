# PLAN

## 1. Task breakdown with priority

**Priority: P0 (Critical Fixes & Core Flow)**
- **보안/인증 픽스**: `VITE_HUMAN_GATE_APPROVER_TOKEN` 하드코딩 제거 및 빌드 산출물 평문 노출 방지를 위한 동적 인증 파이프라인 적용
- **상태 전이 버그 픽스**: 휴먼 게이트 진입 시 워크플로우 런이 `queued`에 머무는 오케스트레이터 버그 수정 (`waiting` 상태로 정상 전이)
- **API 멱등성 확보**: 휴먼 게이트 철회(`POST /api/approvals/{approval_id}/cancel`) 요청 중복 시 예외 처리 및 멱등성 로직 적용 (중복 요청 시 `200 OK` 반환)
- **프론트엔드 메모리 릭 픽스**: `useWorkflowRuns.ts` 등에서 네트워크 플래핑 시 발생하는 SSE(Server-Sent Events) 스트림 다중 생성 문제 해결 (Lock 추가 및 명시적 `EventSource` 언마운트 종료 코드 보강)

**Priority: P1 (Test Coverage Enhancement)**
- **API 필터 검증**: 백엔드 `test_workflow_api.py`에 상태(`status`) 및 기간(`date_range`) 쿼리 파라미터를 활용한 다형성 Audit Log 필터 단위 테스트 케이스 추가
- **E2E 테스트 추가**: 휴먼 게이트 대기 항목의 승인 철회(Cancel) 액션 및 UI 복원 과정을 검증하는 프론트엔드 Playwright E2E 시나리오(`web/tests/e2e/human-gate.spec.ts`) 추가

**Priority: P2 (Workspace & Artifact Integration)**
- **결정 이력 아티팩트화**: 단순 상태 변경을 넘어 휴먼 게이트의 승인/반려 결정 이력 및 사유를 `review.md` 또는 `status.md` 표준 아티팩트로 워크스페이스에 저장하는 기능 연동

## 2. MVP scope / out-of-scope

**MVP scope:**
- 발견된 기능 장애(휴먼 게이트 상태 전이 오류, SSE 누수, 중복 철회 충돌) 완전 해소
- 하드코딩된 보안 토큰의 안전한 제거 및 동적 토큰 기반 인증
- 휴먼 게이트 철회 기능 안정성 확보 및 관련된 백엔드/프론트엔드 테스트 100% 보강
- 휴먼 게이트 이력을 프로젝트 워크스페이스의 아티팩트로 보존

**Out-of-scope:**
- ReactFlow를 활용한 Visual Workflow Builder 신규 편집 캔버스 전면 구현 (본 MVP 범위 밖)
- 기존 내장 FastAPI 워커/오케스트레이터를 대체하는 외부 분산 프레임워크(Temporal 등) 전면 도입
- GitHub CI/Deploy 이벤트 등 복잡한 외부 이벤트 버스의 전면 결합

## 3. Completion criteria

- 휴먼 게이트 상태가 정상적으로 대기(`waiting`)로 전환되는지 백엔드 단위 테스트 케이스 통과
- `POST /api/approvals/{approval_id}/cancel` 엔드포인트에 동시성 요청 시 멱등성이 보장됨을 API 테스트로 증명
- Audit Log 다형성 필터 테스트가 추가되고 성공적으로 통과
- 프론트엔드에서 SSE 재연결 테스트 시 다중 네트워크 소켓이 생성되지 않고 단일 스트림을 유지함 확인
- 프론트엔드 E2E 휴먼 게이트 철회 테스트(Playwright) 성공
- 소스 코드 및 빌드 산출물 내에 `VITE_HUMAN_GATE_APPROVER_TOKEN`가 더 이상 존재하지 않음 확인
- 휴먼 게이트 수행 완료 시 워크스페이스에 `review.md` 또는 연관 아티팩트 파일이 올바르게 생성됨
- 프론트엔드(포트 3100), 백엔드(포트 3101) 로컬 검증 시 모든 동작 및 100% 테스트 패스 확인

## 4. Risks and test strategy

**Risks:**
- 불안정한 네트워크 환경에서 SSE 연결 단절-복구 시 이벤트가 일시적으로 유실되거나 상태 동기화가 어긋날 위험성
- 휴먼 게이트 취소 및 승인 버튼 연타 시 프론트-백엔드 간 레이스 컨디션 발생 가능성

**Test strategy:**
- **동시성 테스트**: 백엔드 API에 다중 스레드로 승인/철회 API를 동시 호출하여 멱등성 및 레이스 컨디션 방어 능력을 엄격하게 검증합니다.
- **네트워크 단절 및 복구 시뮬레이션**: 프론트엔드 로컬 서버(3100번대 포트)에서 브라우저 오프라인 모드를 스위칭하며 SSE 스트림 재생성 락(Lock) 기능 및 메모리 누수 여부를 중점 확인합니다.
- **아티팩트 무결성 테스트**: 워크플로우 런이 휴먼 게이트를 통과한 직후 파일시스템/오브젝트스토어에 지정된 내용의 Markdown 파일이 정상 적재되었는지 백엔드 통합 테스트로 확인합니다.

## 5. Design intent and style direction

- **기획 의도**: 개발자 및 관리자가 중단된 워크플로우의 휴먼 게이트 지점을 명확히 인지하고, 안심하고 승인/수정/거절/철회를 수행할 수 있는 신뢰성 높은 제어 경험 제공. 결정 사유를 아티팩트로 남겨 추적성과 협업을 강화.
- **디자인 풍**: 핵심 데이터에 집중할 수 있는 깔끔한 '모던 대시보드형' 및 '카드형' 레이아웃 적용. 기존 시스템의 간결함을 침해하지 않는 선에서 정보 표현.
- **시각 원칙**:
  - 컬러: 대기(Pending/Waiting)는 경고(Warning) 계열, 승인은 성공(Success), 반려는 위험(Danger) 컬러로 상태를 명확히 대비시킵니다.
  - 패딩/마진: 마우스 및 터치 오클릭 방지를 위해 카드 간 여유로운 패딩과 마진을 두어 안전한 클릭 영역을 확보합니다.
  - 타이포그래피: 에러 상태 및 승인 주체의 이름, 이벤트 시간은 굵기와 색상 대비를 주어 정보의 위계를 명확히 분리합니다.
- **반응형 원칙**: 모바일 우선(Mobile First). 모바일 브라우저에서도 긴 로그 기록이나 휴먼 게이트 액션 버튼이 잘리거나 어긋나지 않도록 카드 스태킹 뷰를 완벽하게 지원합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web / api
- 웹(web) 파트는 **React (Vite 기반)** 환경을 유지하며 구현 및 수정합니다.
- API 파트는 **FastAPI**를 기반으로 설계 및 구현합니다.
- 로컬 실행 및 시뮬레이션 검증 시 포트 충돌 방지를 위해 **3100번대 포트** (UI: 3100, API: 3101 등)를 지정하여 사용합니다.
