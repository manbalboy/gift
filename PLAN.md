# PLAN

## 1. Task breakdown with priority

### 우선순위: 높음 (P0) - 핵심 버그 수정 및 보안 강화
1. **프론트엔드 프리셋 텍스트 파싱 버그 수정**
   - 변경 파일: `web/src/App.tsx`, `web/src/App.test.tsx`
   - 영향 범위: 대시보드 메인 폼 입력 영역
   - 내용: 폼 프리셋 텍스트 연속 입력 시 발생하는 공백 및 개행 누적 버그를 해결하기 위한 정규식 최적화. 관련된 다국어 및 멀티라인 엣지 케이스 단위 테스트 추가.
2. **백엔드 전역 뷰어 토큰(Viewer Token) 검증 미들웨어 추가**
   - 변경 파일: `api/app/api/dependencies.py`
   - 영향 범위: 전체 API 엔드포인트
   - 내용: 인증되지 않은 외부 접근을 차단하기 위한 뷰어 토큰 검증 로직 추가. 로컬 테스트 시 `3100`번 포트 등을 통한 우회 접근 방어.
3. **워크플로우 엔진 무한 루프 및 타임아웃 방어 (Pause 로직)**
   - 변경 파일: `api/app/services/workflow_engine.py`
   - 영향 범위: 백엔드 워크플로우 실행 엔진
   - 내용: 노드 실행 횟수가 임계치를 초과하거나, 단일 노드 실행 시간이 Timeout을 초과할 경우 시스템 자원 고갈을 방지하기 위해 강제로 워크플로우를 일시 정지(Pause)시키는 방어 로직 구현.

### 우선순위: 중 (P1) - UI 유효성 검사 및 에러 핸들링 고도화
4. **워크플로우 빌더 캔버스 클라이언트 검증 로직 추가**
   - 변경 파일: `web/src/components/WorkflowBuilder.tsx`, `web/tests/e2e/WorkflowBuilder.spec.ts`
   - 영향 범위: 워크플로우 편집 UI
   - 내용: 단절된 그래프나 다중 진입점이 존재할 경우 서버 저장 API 호출을 차단하고 사용자에게 피드백을 제공하는 클라이언트 검증 추가 및 관련 E2E 테스트 작성.
5. **백엔드 그래프 유효성 검증 테스트 개편**
   - 변경 파일: `api/tests/test_workflow_engine.py`
   - 영향 범위: 워크플로우 스키마 검증 로직
   - 내용: 서버 단에서도 비정상적인 그래프 데이터 수신 시 단절을 허용하지 않고 명확한 에러(`400` 또는 `422`)를 반환하는지 검증하는 테스트로 개편.
6. **전역 토스트(Toast) UI 컴포넌트 고도화**
   - 변경 파일: `web/src/components/common/Toast.tsx`
   - 영향 범위: 프론트엔드 전역 에러 피드백
   - 내용: 동시다발적인 에러 발생 시 일관된 톤앤매너로 파편화 없이 에러 메시지를 렌더링하도록 토스트 UI 고도화.

### 우선순위: 낮음 (P2) - 고도화 및 인접 기능 추가 (REVIEW 기반 신규)
7. **워크플로우 제어(Pause/Resume) API 및 UI 연동**
   - 변경 파일: `api/app/api/workflows.py` (또는 `runs.py`), `web/src/components/WorkflowControls.tsx` (필요 시 신설)
   - 근거: 무한 루프나 응답 지연으로 인해 엔진이 강제로 워크플로우를 Pause 상태로 변경했을 때, 사용자가 대시보드에서 이를 확인하고 안전하게 재개(Resume)하거나 취소(Cancel)할 수 있는 제어권이 필요함. 구현 범위는 해당 상태를 토글하는 단순 API 엔드포인트와 버튼 UI 제공으로 제한함.

## 2. MVP scope / out-of-scope

### MVP Scope
- 프론트엔드의 폼 입력 버그 해결 및 워크플로우 빌더 캔버스의 무결성 검증(단절 그래프, 다중 진입점 차단).
- 전역 토스트 기반 에러 피드백 시스템 통합 및 안정화.
- 백엔드 워크플로우 엔진의 런타임 안전장치(무한 루프 방지, 실행 타임아웃 강제 Pause 방어 로직).
- API 전체에 대한 뷰어 토큰(Viewer Token) 필수 검증 로직 적용.
- 강제 정지(Pause)된 워크플로우를 대시보드에서 조회하고 재개(Resume)할 수 있는 최소 제어 기능.

### Out-of-scope
- Visual Workflow Builder의 신규 커스텀 노드 타입 추가 및 복잡한 시각적 트랜지션 효과 구현.
- LLM 에이전트 내부 프롬프트 로직 수정 및 새로운 모델 지원 확장.
- 분산 큐(Redis) 기반의 다중 워커 아키텍처 도입 (현재 MVP에서는 단일 엔진의 방어 로직과 안정성에만 집중).
- 권한(Role-Based Access Control) 기반의 복잡한 사용자 계정 관리 시스템.

## 3. Completion criteria
- 대시보드 폼(`web/src/App.tsx`)에서 프리셋 텍스트를 반복 입력해도 화면 레이아웃 깨짐이나 추가 공백/개행 누적이 발생하지 않는다.
- 허가되지 않은 토큰으로 로컬 백엔드(`3100`번 포트 등) 접근 시 `401 Unauthorized` 또는 인증 예외가 일관되게 발생한다.
- 빌더 UI에서 비정상적인 그래프(단절 등)를 구성하고 저장할 경우, API 요청이 차단되고 즉시 토스트 UI로 에러 메시지가 노출된다.
- 백엔드 엔진에서 특정 노드가 설정된 루프 제한 횟수 또는 타임아웃을 초과할 경우 워크플로우 상태가 안전하게 `PAUSED`로 전이된다.
- REVIEW.md의 TODO에 나열된 모든 항목이 단위 테스트 및 E2E 테스트를 성공적으로 통과하며 요구사항을 만족한다.

## 4. Risks and test strategy

### Risks
- 엔진 타임아웃 및 루프 차단 로직 도입 시, 정상적으로 처리 시간이 긴 노드(예: 복잡한 코드 작성 등)가 강제로 중단될 위험이 있음. (초기 타임아웃 임계값의 보수적 설정 필요)
- 엄격한 토큰 인증 미들웨어 추가로 인해 개발 시 내부 Webhook 테스트나 로컬 E2E 테스트가 차단될 수 있음.

### Test Strategy
- **Frontend 단위 테스트 (`web/src/App.test.tsx`)**: 정규식을 통해 파싱되는 텍스트의 다양한 엣지 케이스(국영문, 특수문자, 멀티라인 등)를 파라미터화하여 검증.
- **Frontend E2E 테스트 (`web/tests/e2e/WorkflowBuilder.spec.ts`)**: 브라우저 환경에서 일부러 단절된 그래프를 구성하고 저장 이벤트를 트리거하여 Toast 렌더링 확인 및 네트워크 요청 미발생 검증.
- **Backend 단위/통합 테스트 (`api/tests/test_workflow_engine.py`)**: 비정상적 구조의 그래프 페이로드 주입 시 `400` 또는 `422` 반환 여부를 테스트하고, 무한 루프 상황을 목킹(Mocking)하여 임계치 초과 시 `PAUSED` 상태 전이를 검증. 포트 충돌 방지를 위해 API 테스트 시 `3100`번 포트를 명시적 사용.

## 5. Design intent and style direction
- **기획 의도**: 워크플로우 편집과 실행 전반에 걸쳐 사용자가 시스템을 신뢰할 수 있도록, 오작동을 사전에 방어하고 직관적인 피드백을 제공하여 "안전하고 제어 가능한 자동 개발자 플랫폼" 경험을 전달함.
- **디자인 풍**: 군더더기 없는 모던 대시보드형 디자인. 워크플로우 캔버스는 복잡도를 낮춘 클린한 카드형 스타일로 구현.
- **시각 원칙**: 에러나 경고를 나타내는 토스트 UI는 명확한 포인트 컬러(Red/Yellow/Orange)를 사용하여 시선을 집중시킴. 요소 간 마진과 패딩을 넉넉히 주어 정보 밀도를 낮추고 시스템 폰트를 적극 활용해 가독성을 최우선으로 함.
- **반응형 원칙**: 모바일 우선(Mobile-first) 규칙을 적용하여 화면이 좁아져도 토스트 메시지와 폼 입력 텍스트가 잘리지 않고 유연하게 래핑(Wrapping) 및 플렉스(Flex) 구조를 유지. 캔버스 화면은 모바일 뷰어 모드를 우선 지원.

## 6. Technology ruleset
- **플랫폼 분류**: web, api
- **Web**: React 및 Vite 환경 기반. 상태 관리와 UI 렌더링은 React 함수형 컴포넌트와 훅을 활용. 테스트는 Jest 와 Playwright(E2E)를 사용. 프론트엔드 로컬 개발 서버 포트는 `3000`번 대역을 사용.
- **API**: FastAPI 기반. 비동기 라우팅과 의존성 주입(`Depends`)을 적극 활용하여 뷰어 토큰 인증 미들웨어 처리 로직을 구현. 테스트는 `pytest` 사용. 백엔드 개발 및 API 로컬 실행 포트는 `3100`번을 사용.
