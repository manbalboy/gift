# PLAN

## 1. Task breakdown with priority

### P0 (최고 우선순위 - 핵심 엔진 및 시스템 안정화)
- [ ] **백엔드 실행 엔진 구조 개편**: UI의 API 조회를 통한 스텝 실행 방식에서 벗어나, `workflow_id` 기반 백그라운드 워커 중심의 비동기 실행 엔진으로 코드 전면 리팩토링.
- [ ] **노드 간 전이 로직 수정**: 기존 노드 배열 순서(sequence) 기반 무시 로직을 제거하고, 실제 정의된 노드 간 연결선(edges) 조건을 참조하여 다음 단계로 전이하도록 수정.
- [ ] **실행 강제 취소 리소스 회수**: `api/app/api/workflows.py` 등에서 워크플로우 실행 강제 취소 시, 가동 중인 로컬 스레드 및 활성 SSE 채널을 원자적으로 회수하는 Graceful Shutdown 로직 추가.
- [ ] **실패 노드 자동 재시도 로직 추가 (신규 추가)**: 백그라운드 엔진 전환에 맞춰, 노드 실행 실패 시 설정된 횟수(예: 3회) 내에 백오프를 적용하여 자동 재시도하는 기능을 엔진에 통합 (오류 자가 복구력 확보).

### P1 (높은 우선순위 - UI/UX 개선 및 엣지 케이스 방어)
- [ ] **대용량 아티팩트 렌더링 최적화**: 수십 MB에 달하는 로그나 스크린샷 아티팩트를 브라우저에서 조회할 때 발생하는 크래시를 막기 위해 Chunk Loading 또는 가상화(Virtualization) 기법 적용.
- [ ] **알림(Toast) 컴포넌트 안정화**: `web/src/components/Toast.tsx`에 파라미터 방어(`durationMs` 기본값), 텍스트 레이아웃 이탈 방지(`word-break: break-all`), 다량 이벤트 시 화면 가림을 막는 큐잉 시스템(동시 노출 최대 3개 제한) 도입.
- [ ] **Webhook 수신부 보안 강화**: `api/app/api/webhooks.py`에 수신 IP 대역 검증 로직 및 HMAC 암호화 서명 검증 로직 적용을 통한 인가 검증.
- [ ] **Human Gate 권한 검증**: 승인(Approve) 처리 시 요청 주체의 권한을 확인하는 방어 로직 추가.
- [ ] **Webhook 모니터링 UI 연동 (신규 추가)**: 보안이 강화된 Webhook 엔드포인트에서 발생하는 차단 내역(401/403)을 프론트엔드 대시보드에서 실시간으로 관측할 수 있도록 알림 및 로그 뷰 추가 (보안 가시성 확보).

### P2 (일반 우선순위 - 테스트 커버리지 확보)
- [ ] Webhook HMAC 및 IP 위조 차단을 검증하는 `pytest` 방어망 단위 테스트 작성.
- [ ] Human Gate 대기 상태에서 지속성(Persistence) 유지 및 이행(Resume) 성공을 확인하는 백엔드 통합 테스트 작성.
- [ ] 다중 클라이언트 SSE 연결/종료 반복 시 자원 누수를 검증하는 Python 스트림 부하 테스트 스크립트 작성.
- [ ] `http://localhost:3100` 대상 Workflow Builder 에디터(ReactFlow) 순환 참조 및 편집 에러 방어용 Playwright E2E 테스트 스크립트 작성 및 통과 확인.

## 2. MVP scope / out-of-scope

### MVP scope
- `workflow_id` 기반 비동기 백그라운드 실행 엔진으로의 전환 및 Edges 기반 전이 로직 활성화.
- ReactFlow 기반 워크플로우 에디터의 안정성 확보 및 대용량 아티팩트 조회 처리.
- 핵심 보안 결함(Webhook HMAC, 승인 권한, SSE 리소스 반환) 해결.
- 알림 UI의 렌더링 오류 수정 및 큐 제한 로직 반영.
- 위 변경 사항을 뒷받침하는 단위, 통합, E2E 테스트 구현 및 통과.

### Out-of-scope
- 분산 시스템(예: Kubernetes, Temporal 등) 수준의 거대 워크플로우 오케스트레이션 인프라 도입.
- 외부 SaaS 툴(예: Slack, Jira 등)로의 신규 이벤트 버스 연동 (기존 Webhook 수신부 보안에만 집중).
- 모바일 전용 앱 플랫폼 출시 (웹의 반응형 뷰 지원으로 갈음).

## 3. Completion criteria
- 저장소 내 `api` 앱의 테스트 스위트(`pytest`)가 모두 통과하며, 새로 추가된 Webhook 방어 및 상태 이행(Resume) 통합 테스트가 Success를 반환해야 함.
- Web UI E2E 테스트(`playwright`) 실행 시 `http://localhost:3100` 환경에서 노드 순환 참조 예외 처리 및 에디터 저장 시나리오가 100% 통과해야 함.
- 파이썬 기반 스트림 부하 테스트 도구를 실행했을 때 활성 SSE 채널 누수로 인한 리소스 초과 현상이 없음을 증명해야 함.
- 프론트엔드 상에서 에러 이벤트를 강제로 10개 이상 발생시켰을 때 Toast 알림이 화면에 최대 3개까지만 동시 노출되고 순차적으로 처리되는 것을 시각적으로 확인해야 함.
- API를 통한 워크플로우 취소 호출 시, 활성화된 워커 스레드가 Orphan 상태로 남지 않고 안전하게 회수(Graceful Shutdown)됨을 백엔드 로그로 증명해야 함.

## 4. Risks and test strategy

### Risks
- **구조 전환에 따른 상태 비동기화 위험**: API 호출 트리거에 의존하던 프론트엔드 상태 갱신 구조를 백그라운드 엔진으로 완전히 분리함에 따라, SSE 동기화 과정에서 클라이언트와 서버 간 일시적인 이벤트 누락이나 지연이 발생할 수 있는 위험.
- **포트 충돌 가능성**: UI 프리뷰 및 테스트 환경 구축 시 `3100` 포트를 고정으로 할당하므로, CI/CD 환경이나 로컬 환경에서 포트 점유 충돌이 발생할 수 있음.

### Test strategy
- 백엔드 비즈니스 로직(검증, 상태 전이, 보안 인가)은 `pytest`를 활용하여 CI 연동 가능한 수준의 단위 및 통합 테스트를 철저히 검증.
- 프론트엔드 시각화 및 노드 조작 로직은 `playwright`를 통해 UI 인터랙션 흐름을 `http://localhost:3100` 환경에서 자동화하여 End-to-End로 테스트.
- 다중 SSE 커넥션과 같은 동시성 엣지 케이스는 가상 유저를 생성하여 커넥션 연결/해제 스트레스를 집중적으로 가하는 파이썬 기반의 독립된 부하 테스트 스크립트로 검증.

## 5. Design intent and style direction
- **기획 의도**: 복잡한 노드 기반 자동화 파이프라인(SDLC)을 개발자가 직관적으로 설계하고 모니터링할 수 있는 '안정적이고 관측 가능한 개발 지휘소' 경험을 제공.
- **디자인 풍**: 모던 대시보드형 (Modern Dashboard). 정보 밀도가 높은 로그와 노드를 다뤄야 하므로 눈의 피로도를 낮추고 집중도를 높일 수 있는 어두운 테마(Dark mode) 친화적 인터페이스 권장.
- **시각 원칙**:
  - 컬러: 노드의 상태(성공, 실패, 대기)를 즉각 인지할 수 있도록 채도 높은 포인트 컬러(Green, Red, Yellow)를 사용하되, 배경과 패널은 무채색 기반의 톤다운 컬러 배색.
  - 패딩/마진: 정보와 계층의 명확한 구분을 위해 섹션 간 마진은 16px 이상으로 넓게 가져가고, 카드 및 모달 내부는 12px 이상의 일관된 패딩을 적용.
  - 타이포그래피: 코드 조각이나 로그, 상태 코드 영역에는 고정폭 폰트(Monospace)를 적용하고, 일반 안내 및 정보 텍스트는 가독성이 우수한 산세리프체(Sans-serif)를 사용.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 적용하여, 런타임 타임라인과 리포트 내역 등이 모바일 기기에서도 세로 스크롤로 가독성 있게 표현되도록 레이아웃 재배치(Stacking)를 지원. 단, 워크플로우 캔버스 에디터 화면의 경우 모바일에서는 '조회(Read-only)' 위주로 기능을 제한하여 제공.

## 6. Technology ruleset
- **플랫폼 분류**: web, api
- **Frontend (web)**: React 라이브러리 (Vite 빌드 기반) 구조를 채택. 시각화 코어는 ReactFlow를 활용하며 UI 테스트는 Playwright를 기반으로 계획.
- **Backend (api)**: FastAPI를 기반으로 비동기 API 및 백그라운드 워커 엔진 로직을 설계. 데이터 모델 검증 및 백엔드 전반의 테스트는 pytest 기반으로 계획.
