# PLAN

## 1. Task breakdown with priority

### 우선순위: 높음 (REVIEW.md 피드백 우선 반영)
- **보안 버그 수정 (`web/src/utils/security.ts`)**: `sanitizeAlertText` 함수 내부의 XSS 필터링 로직 전면 수정. DOMPurify가 악성 태그를 먼저 정상적으로 제거하게 한 후, 제네릭(`<T>`) 등 정상적 문법을 화면에 안전하게 출력할 수 있도록 복원 로직 개선.
- **초과 루프 카운트 표기 UI 버그 수정**: 루프 엔진 오류로 제한 횟수가 초과 실행될 경우 `LoopMonitorWidget`에서 '0'으로만 표기되는 문제를 해결. 초과 상태를 명확히 인지할 수 있도록 시각적 경고(예: 붉은색 텍스트 표기 및 초과 횟수 노출) 반영.
- **프론트엔드 상태 갱신 쓰로틀링/디바운싱 적용**: 잦은 SSE 이벤트 유입에 대비하여 프론트엔드의 상태 갱신 로직을 디바운스(Debounce) 또는 쓰로틀링(Throttle) 처리해 브라우저 렌더링 병목 및 UI 멈춤 현상 방지.

### 우선순위: 중간 (모니터링 UI 및 제어 시각화 고도화 - 인접 기능)
- **수동 루프 개입(제어) 상태 피드백 강화**: 루프 제어(Start, Pause, Stop) 시 백엔드의 처리 상태가 프론트엔드 화면에 반영되기 전까지 로딩 스피너 및 버튼 비활성화 상태를 주입하여 중복 요청을 방지하고 사용자 경험 안정화. (근거: 실시간 지연에 대한 UX 혼란 방지)
- **에러 로그 상세 보기 모달 추가**: 큐 오버플로우나 루프 중단 오류 발생 시 단순 토스트 알림 외에, 에러 상세 정보와 디버그 트레이스를 확인할 수 있는 전용 로그 모달 창 추가. (근거: 개발 자동화 시스템 특성상 단순 오류 인지를 넘어 즉각적인 원인 파악이 필요함)

### 우선순위: 낮음 (테스트 커버리지 보강)
- **`LoopMonitorWidget` 단위 테스트 추가 (`web/src/components/LoopMonitorWidget.test.tsx`)**: 대시보드 컴포넌트의 마운트, 색상 톤(`qualityTone`), 수치 변화 등 렌더링 검증.
- **큐 오버플로우 토스트 알림 검증 테스트 추가 (`web/src/App.test.tsx`)**: API 응답 지시사항 상태가 `dropped`(`queue_overflow`)로 수신되었을 때 화면에 경고 UI(Toast)가 노출되는지 통합 프론트엔드 환경에서 검증.

## 2. MVP scope / out-of-scope

### MVP Scope
- 프론트엔드 대시보드를 통한 Self-Improvement Loop 엔진 상태(Quality Score, 실행 횟수 등) 실시간 모니터링 보완 및 브라우저 성능 최적화.
- 프론트엔드 XSS 보안 취약점의 근본적 제거 및 안전한 메시지 노출 체계 확보.
- 엔진 제어 동작에 대한 직관적인 시각적 피드백 제공 및 오류 알림/디버그 모달 연동.
- 프론트엔드 주요 위젯 및 이벤트 수신에 대한 검증 테스트 코드 작성.

### Out-of-Scope
- 루프 엔진(Analyzer, Evaluator 등)의 신규 핵심 평가 로직 추가나 프롬프트 구조 전면 재설계 (백엔드 코어 모델 변경 제외).
- 다중 사용자 계정 기반 권한 관리(RBAC) 및 사용자별 맞춤형 대시보드 편집 기능.
- 1차 MVP 범위를 벗어난 전역적인 UI 컴포넌트 라이브러리 교체.

## 3. Completion criteria
- `web/src/utils/security.ts`의 `sanitizeAlertText` 로직이 XSS 공격 스크립트를 완벽히 제거하면서도 일반 텍스트 내 꺾쇠 기호를 정상 보존하여 보안 테스트 케이스를 통과해야 함.
- 빈번한 엔진 갱신 상황(다중 SSE 이벤트 유입)에서도 렌더링 스파이크가 발생하지 않고 화면 지표가 지연 없이 부드럽게 업데이트되어야 함.
- 루프 횟수 초과나 큐 오버플로우 등 에러 엣지 케이스 발생 시, UI에서 경고 상태와 초과 수치, 상세 에러 로그 모달이 명확하게 표출되어야 함.
- 새로 작성된 모든 단위/통합 테스트(`LoopMonitorWidget.test.tsx`, `App.test.tsx` 테스트 코드)가 정상적으로 통과되어야 함.
- 로컬 개발 시 웹 프론트엔드는 3000번대 포트에서, Docker 컨테이너 프리뷰 배포 시 `7000-7099` 포트 범위 내에서 원활하게 서빙 및 연동되어야 함.

## 4. Risks and test strategy

### Risks
- 성능 최적화(쓰로틀링/디바운싱)로 인해 UI에 표시되는 상태 지표와 실제 백엔드 엔진 상태 간 약간의 시간차가 생겨 수동 제어 시 체감 반응성이 떨어질 수 있음.
- 기존 XSS 방어 로직 전면 수정 과정에서 DOMPurify 설정 오류로 인한 신규 렌더링 버그나 또 다른 보안 우회 경로가 열릴 수 있음.

### Test Strategy
- **보안 단위 테스트 강도 강화**: `<script>`, `<img onerror=...>` 등의 악성 태그 패턴과 `<T>` 같은 정상 제네릭 문자열 조합을 다수 입력하여 필터링과 보존의 정확성을 엄격히 테스트.
- **성능 부하 시뮬레이션 테스트**: 짧은 시간에 다량의 Mock SSE 이벤트를 프론트엔드로 주입하고, 쓰로틀링 로직 적용 전후의 컴포넌트 재렌더링 횟수를 측정하여 성능 방어를 확인.
- **자동화 UI 엣지 테스트**: React Testing Library(RTL)를 활용하여 특정 오류 상태(`dropped`, 루프 횟수 초과)일 때 토스트 노출, 컴포넌트 붉은색 강조, 상세 모달 열기 등의 렌더링 분기를 철저히 검증.

## 5. Design intent and style direction
- **기획 의도**: 지속적으로 스스로 코드를 생성, 테스트, 개선하는 Self-Improvement Loop 엔진의 방대한 상태와 지표를 개발자가 투명하고 안전하게 파악하며 즉각적으로 개입(제어)할 수 있도록 신뢰감을 주는 모니터링 환경 제공.
- **디자인 풍**: 모던 대시보드형 뷰. 기술적인 시스템 정보를 직관적인 위젯 카드 형태로 분리하여 시각화.
- **시각 원칙**:
  - 데이터의 상태(정상, 경고, 오류)에 따라 색상 톤(`qualityTone`)을 확실히 구분(예: 안정-녹색, 주의-주황색, 위험-붉은색)하여 빠른 인지 도모.
  - 컴포넌트 여백(Margin/Padding)은 8px 배수 시스템을 일관되게 적용하여 정돈된 그리드 레이아웃을 형성.
  - 품질 점수, 로그 코드, 실행 횟수 등 시스템 관련 핵심 수치에는 고정폭(Monospace) 타이포그래피를 적용하여 개발 도구 특유의 정교함을 전달.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 적용하되, 한 화면에 복합적인 시스템 지표를 담아야 하는 대시보드 특성상 데스크탑/태블릿 환경에서 다단 그리드 뷰 확장을 적극적으로 활용.

## 6. Technology ruleset
- **플랫폼 분류**: web / api
- **프론트엔드 (web)**: React 기반 (현행 Vite 빌드 체계와 기존 프론트엔드 생태계 유지). 상태 관리는 기존의 구조나 Context API를 활용.
- **백엔드 (api)**: FastAPI 기반 (제공된 구조에 포함된 기존 제어 라우터 및 SSE 스트리밍 체계 활용).
- **포트 규정**: 로컬 프론트엔드 개발 서버 실행 시 3000번대 포트만 사용. 프리뷰 및 배포 환경에서는 요구사항에 따라 7000-7099 범위만 할당.
