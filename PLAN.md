# PLAN

## 1. Task breakdown with priority

### P0 (최우선 과제: 핵심 엔진 및 안정성 확보)
- **Workflow Engine v2 전환**: `workflow_id` 기반 정의 실행기 구현 및 `node_runs` 저장 체계 구축.
- **Human Gate + Resume 도입**: 휴먼 인더루프를 통한 승인/수정/거절 처리 및 LangGraph 기반 상태 대기/재개 흐름 완성.

### P1 (주요 과제: 산출물 및 에이전트 확장)
- **Artifact-first Workspace**: 텍스트, 스크린샷 등 노드 산출물의 아카이빙, 재현 및 검색 메타데이터 시스템 구축.
- **Agent SDK & Marketplace 기반 마련**: CLI 템플릿의 Agent Spec 추상화 및 실행 어댑터 표준화.

### P2 (부가 과제: 시각화 및 통합)
- **Visual Workflow Builder**: ReactFlow 기반 워크플로우 노드/엣지 UI 편집, 상태 검증 및 시뮬레이션 구동.
- **Dev Integrations 이벤트 연동**: GitHub PR/CI 이벤트 수집 및 워크플로우 자동 트리거 구성.

## 2. 고도화 플랜 (Enhancement Plan)
본 플랜은 기존 코드 리뷰(`REVIEW.md`)의 피드백을 수용하여, 사용자 경험과 시스템 안정성을 향상시키는 것을 목표로 합니다.

- **SSE 통신 장애 대응 UI 강화**
  - **근거**: 실시간 로그 스트리밍 시 백엔드 단절 현상(예: 프록시 오류)이 발생하면 무한 로딩 상태에 빠져 사용자가 장애를 인지하기 어렵습니다.
  - **구현 경계**: 지수 백오프(Exponential Backoff) 기반의 최대 재연결 시도 횟수를 초과할 경우 재시도를 멈추고, '서버 통신 실패' 상태를 명시하는 Graceful Fallback UI로 전환하는 흐름까지만 한정합니다.
- **대용량 Artifact 뷰어 렌더링 및 UX 편의성 개선**
  - **근거**: 50MB 이상의 대규모 텍스트 렌더링 시 브라우저 OOM을 방지하고, 휴먼 게이트 승인/거절 시 사용자 입력 경험의 불편함(프리셋 덮어쓰기 오차)을 보완해야 합니다.
  - **구현 경계**: 가상 스크롤 내 텍스트 청크 교체 시 검색 하이라이팅 깜빡임 방지 및 좌표 오차 보정 기능을 구현하고, 거절 사유 프리셋 버튼 클릭 시 기존 작성된 텍스트 뒤에 내용을 덧붙이도록(Append) 폼 로직을 수정합니다.

## 3. MVP scope / out-of-scope

### MVP Scope
- 워크플로우 JSON 기반 단일/다중 노드 실행, 재시도, 실패 처리 파이프라인.
- 승인/거절 상태를 관측하고 처리할 수 있는 대시보드와 Human Gate 인프라.
- 안정적인 대용량 파일 청크 렌더링을 지원하는 Artifact 뷰어 및 통신 장애 Fallback UI.
- React 기반 사용자 대시보드 및 FastAPI를 활용한 비동기 백엔드 워커 연동.

### Out-of-scope
- 다중 클러스터 노드 기반의 오토 스케일링 인프라.
- Visual Builder 내에서의 실시간 양방향 코드 디버거 기능.

## 4. Risks and test strategy

### Risks
- 대용량 데이터 로딩으로 인한 프론트엔드 성능 병목 및 브라우저 힙 메모리 고갈.
- 장기 실행 워크플로우의 부분적 통신 단절로 인한 무한 대기 상태 노출.

### Test Strategy
- **통합 검증**: `test_workflow_engine.py` 등 API 테스트를 통해 Human Gate 상태 전이(Pending -> Resume) 데이터 정합성을 검증합니다.
- **E2E 성능/결함 테스트 (REVIEW 기반 추가)**: 
  - 50MB 이상 대규모 산출물 더미 데이터를 로드하여 OOM 발생 여부와 메모리 힙 추이를 확인하는 E2E 성능 테스트 스크립트를 도입합니다.
  - 로컬 환경의 백엔드/프록시 서비스 포트(3100번대)를 인위적으로 강제 종료하여 프론트엔드의 백오프 중단 및 Graceful Fallback 작동을 검증하는 네트워크 결함 테스트 시나리오를 구성합니다.

## 5. Design intent and style direction

- **기획 의도**: 개발 및 운영 조직이 복잡한 AI 에이전트의 워크플로우 실행 상태를 직관적으로 파악하고 산출물 결과를 검토하여, 병목 없이 빠르고 정확한 의사결정(승인/거절)을 내릴 수 있는 투명성을 제공합니다.
- **디자인 풍**: 시스템 모니터링에 적합한 대시보드형(Dashboard) 뷰와 데이터의 응집성을 높인 카드형 레이아웃 스타일.
- **시각 원칙**:
  - 컬러: 무채색(Dark/Light Gray)을 메인으로 사용하며, 상태값 구분(Success-Green, Failure-Red, Pending-Yellow)에만 직관적인 포인트 컬러를 부여합니다.
  - 패딩/마진: 정보 영역별로 16px~24px의 여백 규격을 일관되게 사용하여 안정적인 구조를 설계합니다.
  - 타이포: 정보 전달력을 높이기 위해 시스템 기본 산세리프 폰트를 주력으로 사용하며, 로그와 코드가 출력되는 뷰어는 가독성 높은 고정폭(Monospace) 폰트를 혼용합니다.
- **반응형 원칙**: 상세 정보 열람 및 스키마 검증은 화면이 넓은 데스크탑 중심(Desktop-First)으로 구성하며, 휴먼 게이트 승인/거절 등 즉각적 조작이 필요한 화면은 유동적인 1단 컬럼으로 축소되어 모바일에서도 대응 가능하게 처리합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web 및 api
- **web**: React (TypeScript) 기반 라이브러리로 계획하여 반응성이 뛰어난 클라이언트 아키텍처 및 ReactFlow 노드 기반 UI를 구성합니다.
- **api**: FastAPI 기반으로 계획하여 워크플로우 실행 엔진 제어, Human Gate 데이터 검증 및 비동기 스트리밍(SSE)을 전담합니다.
- **실행 및 프리뷰 포트 원칙**: 모든 로컬 실행 가이드, 컨테이너 프리뷰, 프록시 포트는 충돌 없는 일관된 구동을 위해 가이드라인에 따라 3000번대 포트(예: 3000, 3108)를 사용하도록 제한합니다.
