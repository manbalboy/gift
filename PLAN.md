# PLAN

## 1. Task breakdown with priority

### Priority 1: Security & Core Stability (API)
- `config.py`의 `DEVFLOW_LOCALHOST_SPOOF_GUARD_PORTS` 파싱 로직 안정화: 유효하지 않은 입력이나 특수문자 입력 시 파서 크래시를 방지하고 안전한 기본값으로 폴백(Fallback)하도록 예외 처리.
- 워크스페이스 아티팩트 접근 시 파일 시스템 권한 및 OS Lock 예외 방어: 장기 실행 워크플로우 런타임 중 I/O 오류로 인한 프로세스 크래시 및 자원 누수 방지를 위한 포괄적 `try-except` 핸들링 추가.

### Priority 2: Functional Fixes (API)
- `api/app/api/workflows.py` 내 `resume` 엔드포인트 멱등성(Idempotency) 보장: 동시 요청 시 락 경합으로 인한 `400 Bad Request` 에러를 수정하여, 상태에 따라 `409 Conflict` 또는 `200 OK`를 일관성 있게 반환.
- 워크플로우 실패 상태(`failed`) 전이 로직 리팩토링: 아티팩트 유실 등으로 실패 시 시스템 예외를 발생시키지 않고, 안전하게 갱신된 `Run` 상태 객체를 반환하도록 수정.

### Priority 3: UI Rendering Fixes (Web)
- React 대시보드의 실패 상태 렌더링 불일치 해결: 단순히 에러 알림(Toast 등)으로만 처리되던 워크플로우 실패 상태를 노드 다이어그램에 정확하게 Failed 상태(Red)로 시각적 렌더링하도록 보완.

### Priority 4: Test Coverage Expansion (Test)
- `config.py` 포트 파싱 로직에 대한 경계값 및 예외 상황 단위 테스트 작성.
- 비동기 다중 `resume` API 호출 시 멱등성을 검증하는 통합 테스트 작성.
- Graceful Failure 발생 시 `Run` 및 `NodeRun` DB 테이블에 `failed` 상태가 정확히 커밋되었는지 검증하는 통합 테스트 추가.
- 고의로 아티팩트를 손상시킨 뒤 `resume`을 수행하여 UI상 Failed 노드가 렌더링되는지 확인하는 E2E 테스트 시나리오 구현.

## 2. MVP scope / out-of-scope

### MVP scope
- `REVIEW.md`에 나열된 치명적인 기능 결함 및 보안 취약점의 완벽한 해결.
- 시스템 다운이나 데드락 없이 실패 상태를 우아하게(Graceful) DB에 기록하고 API로 응답하는 멱등성 로직 구현.
- 사용자가 직관적으로 워크플로우 장애를 인지할 수 있도록 프론트엔드 실패 UI 상태 연동 및 테스트 셋업 완성.

### Out-of-scope
- 새로운 워크플로우 노드 타입의 추가 구현이나 설계 변경.
- 프론트엔드 ReactFlow 다이어그램 구조의 전면적인 재설계 (실패 상태 색상 표기에만 한정).
- 데이터베이스 스키마 확장 (현재 `status` 컬럼만을 이용해 처리).

## 3. Completion criteria
- 다중 스레드/비동기 동시 `resume` 요청을 수행해도 시스템이 크래시되지 않으며 올바른 HTTP 상태 코드가 반환됨.
- 환경변수에 잘못된 값을 할당하고 실행해도 런타임 초기화 과정에서 서버가 다운되지 않음.
- 워크스페이스의 필수 파일을 임의로 삭제한 후 워크플로우를 진행할 시, DB의 해당 워크플로우 상태가 `failed`로 변경되고 UI상 노드가 붉은색으로 표출됨.
- 신규 작성된 주요 단위/통합/E2E 테스트가 CI 파이프라인(또는 로컬 환경)에서 100% 통과함.

## 4. Risks and test strategy

### Risks
- 다수의 요청이 동시 다발적으로 유입될 때 멱등성을 확보하기 위한 잠금 로직(Lock)이 기존 트랜잭션의 데드락을 유발할 수 있음.
- 파일 I/O 로직 전체에 도입되는 `try-except`가 예상치 못한 시스템 에러까지 무음(Silent) 처리하여 디버깅을 어렵게 만들 가능성이 있음 (반드시 구조화된 에러 로깅 병행).

### Test strategy
- **단위 테스트 (Unit Test)**: `_parse_ports_csv`와 같은 파싱 유틸리티 함수에 극한의 입력값(특수기호, 빈 문자열, 매우 큰 수)을 주입해 기본값이 잘 매핑되는지 검증.
- **통합 테스트 (Integration Test)**: `pytest-asyncio`를 활용해 수십 개의 비동기 클라이언트가 한 번에 `resume`을 호출하도록 구성한 뒤, 단 1건만 성공 또는 모두 적절한 예외 처리로 끝나는지 락 경합 상태를 검증.
- **E2E 테스트 (E2E Test)**: Playwright 또는 Puppeteer를 통해 UI에서 실패 상황이 연출되었을 때 DOM 요소 내의 노드 상태 클래스가 올바르게 주입되어 렌더링되는지 확인.

## 5. Design intent and style direction

- **기획 의도**: 개발자 및 사용자가 예측할 수 없는 런타임 에러나 동시성 충돌을 겪더라도 시스템을 신뢰할 수 있도록, 투명하고 일관성 있는 실패 피드백 루프를 제공합니다.
- **디자인 풍**: 모던 대시보드형 디자인을 유지하여 정보의 우선순위가 명확하게 전달되도록 구성합니다.
- **시각 원칙**: 실패(Failed) 및 에러 상태는 즉시 인지가 가능한 강렬한 붉은색 계열을 활용하며, 패딩과 마진은 넉넉하게 주어 다른 워크플로우 시각 요소들과 간섭을 최소화합니다. 텍스트 가독성을 위해 명확한 시스템 타겟 타이포그래피를 적용합니다.
- **반응형 원칙**: 모바일 환경에서도 실패 상태에 대한 시각적 알림과 노드 다이어그램이 화면 영역을 벗어나지 않고 잘 정렬되도록 모바일 우선(Mobile-First) 규칙을 바탕으로 구현합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web / api
- **web**: React 라이브러리 기반 프레임워크를 사용하여 프론트엔드 대시보드의 상태 변경 및 렌더링을 구현합니다.
- **api**: FastAPI 기반 프레임워크를 통해 백엔드 엔드포인트 멱등성 보장 및 안정성을 확보하며, 실행 가이드에서 요구될 경우 3000번대 포트를 기준으로 할당 및 충돌 회피 규칙을 준수합니다.
