# PLAN

## 1. Task breakdown with priority

*   **[P0] UI 버그 수정**: `web/src/components/WorkflowBuilder.tsx` 내 모바일 뷰 안내 문구 오타("모 니터링을" -> "모니터링을") 수정.
*   **[P0] API 보안 및 예외 처리 강화**:
    *   `api/app/api/webhooks.py` 내 IP 파싱 로직 수정: `X-Forwarded-For` 헤더 변조(Spoofing)를 방지하기 위해 신뢰할 수 있는 프록시 설정 또는 엄격한 파싱 로직 적용.
    *   `api/app/api/webhooks.py` 내 `workflow_id` 타입 파싱 실패(`isdigit()` 실패 등) 시 원인을 추적할 수 있도록 Error/Warning 로거 추가.
*   **[P1] 프론트엔드 테스트 환경 정비**: `web/` 디렉터리의 단위 테스트(`jest`) 환경을 점검하고, 정상적으로 테스트 스크립트가 실행되도록 `package.json` 및 설정 보완.
*   **[P1] Rate Limiter 설계 한계 명시화**: `api/app/services/rate_limiter.py` 등 관련 파일에 다중 워커 환경에서의 로컬 메모리 동시성 경합 문제와 향후 분산 캐시(Redis 등) 도입 필요성에 대한 구조적 주석 및 문서화 추가.
*   **[P2] Visual Builder 노드 속성 패널 기초 구현 (고도화)**:
    *   **근거**: `WorkflowBuilder.tsx` 유지보수 및 프론트엔드 테스트 정비 시점에, 향후 워크플로우 편집기 완성을 위해 필수적인 '노드 상세 보기' 기능의 뼈대를 마련하기 위함입니다.
    *   **구현 경계**: React Flow 상에서 특정 노드를 클릭했을 때, 우측 또는 하단에 선택된 노드의 기본 정보(ID, 타입)만을 표시하는 읽기 전용 UI 패널까지만 구현합니다(속성 편집 기능은 제외).

## 2. MVP scope / out-of-scope

*   **MVP Scope**:
    *   `REVIEW.md`에서 식별된 프론트엔드 텍스트 오타 및 API 웹훅 엔드포인트의 보안 취약점/예외 처리 누락 수정.
    *   프론트엔드 UI 컴포넌트에 대한 테스트 실행 환경 복구.
    *   현재 구조에서 해결하기 힘든 Rate Limiter 다중 워커 이슈의 구조적 한계 문서화.
    *   워크플로우 빌더의 노드 상태 확인을 위한 최소한의 UI 확장(속성 패널).
*   **Out-of-scope**:
    *   Temporal 또는 LangGraph 기반의 전면적인 Workflow Engine 아키텍처 재작성.
    *   다중 워커 환경에서의 Rate Limiter 완벽한 동기화 및 분산 처리 구현(문서화로 대체).
    *   Visual Builder 노드의 속성을 실시간으로 수정하고 저장하는 복잡한 상태 관리 기능.

## 3. Completion criteria

*   `WorkflowBuilder.tsx` 화면에서 "모 니터링을" 오타가 사라지고 정상적으로 "모니터링을"로 렌더링되어야 합니다.
*   `X-Forwarded-For` 헤더가 조작된 악의적인 웹훅 요청이 들어와도 서버가 안전하게 IP를 처리하거나 차단해야 합니다.
*   웹훅 페이로드에 잘못된 타입의 `workflow_id`가 포함되었을 때, 무시되기 전 시스템 로그에 해당 사실이 명확히 기록되어야 합니다.
*   `web/` 디렉터리에서 프론트엔드 테스트 스크립트 실행 시(예: `npm run test`) 에러 없이 모든 구성된 테스트가 통과해야 합니다.
*   워크플로우 에디터 화면에서 노드를 클릭했을 때 해당 노드의 식별 정보가 패널에 정상적으로 표시되어야 합니다.

## 4. Risks and test strategy

*   **Risks**: 웹훅 API의 IP 검증 및 헤더 파싱 로직을 엄격하게 수정함에 따라, 기존에 정상적으로 들어오던 외부 연동 서비스(GitHub 등)의 요청이 차단될 위험이 있습니다.
*   **Test Strategy**:
    *   **API 계층**: `pytest`를 활용하여 `X-Forwarded-For` 헤더가 조작된 경우, 없는 경우, 정상적인 프록시 체인을 거친 경우 등 다양한 엣지 케이스에 대한 단위 테스트(`test_webhooks_api.py`)를 추가 작성하여 검증합니다. 잘못된 페이로드 요청을 보내 로그가 정상적으로 출력되는지 캡처하여 확인합니다.
    *   **웹 계층**: 보완된 Jest 테스트 환경을 통해 수정된 컴포넌트(`WorkflowBuilder.tsx`)와 신규 패널의 렌더링이 깨지지 않는지 단위 테스트로 확인합니다.

## 5. Design intent and style direction

*   **기획 의도**: 워크플로우를 구성하고 실행 상태를 확인하는 과정에서 개발자 및 관리자가 직관적으로 상황을 파악하고 디버깅할 수 있도록 돕는 투명한 인터페이스 제공.
*   **디자인 풍**: 불필요한 장식을 배제한 모던하고 실용적인 대시보드형 및 카드형 디자인.
*   **시각 원칙**: 다크 테마를 기반으로 하여 장시간 작업하는 개발자의 눈 피로를 줄이고, 상태를 나타내는 색상(성공, 실패, 대기)을 명확한 고대비 포인트 컬러로 사용하여 가시성을 높입니다. 여백과 패딩을 일관되게 적용하여 정보의 위계를 나눕니다.
*   **반응형 원칙**: 모바일 우선(Mobile-first) 규칙을 적용합니다. 좁은 모바일 화면에서는 복잡한 그래프 대신 리스트 형태나 요약된 카드 뷰로 정보를 제공하며, 화면이 넓어질수록 상세 패널과 시각적 워크플로우 빌더 영역을 확장하여 보여줍니다.

## 6. Technology ruleset

*   **플랫폼 분류**: web 및 api
*   **프론트엔드 (web)**: React 기반(Vite 생태계)으로 컴포넌트를 구성하며, 스타일링과 UI 로직을 구현합니다.
*   **백엔드 (api)**: FastAPI(Python) 기반으로 구축된 기존 아키텍처를 유지하며, HTTP 통신 및 워크플로우 오케스트레이션을 담당합니다.
*   **실행 가이드**: 로컬 환경 실행 시 API 서버는 **3000** 포트를, 웹 클라이언트 개발 서버는 **3001** 포트를 사용하는 등 3000번대 포트만을 사용하여 포트 충돌을 방지합니다.
