# PLAN

## 1. Task breakdown with priority
- **[P0] 보안/버그 수정**: `web/src/utils/security.ts`의 XSS 필터링 로직 수정. `<T>` 형태의 제네릭 문법 텍스트가 삭제되는 문제를 해결하고 악성 스크립트만 차단되도록 방어 로직 고도화.
- **[P0] UI/버그 수정**: `web/src/components/LoopMonitorWidget.tsx`에서 루프 횟수가 최대치를 초과했을 때 남은 횟수가 '0'으로 표기되는 시각적 버그 수정. 오버런(Overrun) 횟수를 정확히 표기하고 붉은색 경고 스타일 반영.
- **[P1] 성능 최적화**: 백엔드에서 유입되는 잦은 SSE 이벤트에 대응하기 위해 상태 갱신 로직에 쓰로틀링(Throttling) 기법을 적용하여 브라우저 렌더링 병목 예방.
- **[P1] UX 개선**: 루프 엔진 제어 버튼(Start/Pause/Stop) 조작 시 백엔드 응답을 기다리는 동안 버튼 비활성화 및 로딩 스피너를 추가하여 중복 조작 방지.
- **[P1] 신규 컴포넌트**: 시스템 큐 오버플로우나 엔진 예외 종료 시 상세 에러 원인을 확인할 수 있는 '에러 로그 모달(Error Log Modal)' 컴포넌트 구현.
- **[P2] 테스트 코드 보강**: `LoopMonitorWidget.test.tsx` 컴포넌트 렌더링 단위 테스트, `App.test.tsx` 토스트 알림 통합 테스트, 보안 엣지 케이스 단위 테스트 추가 작성.
- **[P3] 고도화 기능 추가 (1): 에러 로그 클립보드 복사 기능**
  - **근거**: 개발자/사용자가 에러 로그를 외부 이슈 트래커나 디버깅 툴에 쉽게 전달할 수 있도록 편의성 제공.
  - **구현 경계**: 에러 로그 모달 창 내부에 `navigator.clipboard` API를 활용한 단순 텍스트 복사 버튼 배치.
- **[P3] 고도화 기능 추가 (2): SSE 연결 상태 알림 뱃지 UI**
  - **근거**: REVIEW.md에 명시된 엣지 케이스 반영. 장기 실행 특성상 네트워크 단절 시 엔진이 멈춘 것인지 네트워크 문제인지 직관적으로 구분하기 위함.
  - **구현 경계**: 헤더 영역 또는 위젯 상단에 'Reconnecting...' 등 현재 소켓 연결 상태를 작게 표시하는 수준으로 제한 (복잡한 오프라인 데이터 동기화 기능은 제외).

## 2. MVP scope / out-of-scope
**MVP Scope**
- `security.ts` 내 정상 제네릭 텍스트 보존 및 XSS 필터링 정상 작동 보장.
- 루프 오버런 횟수의 정확한 수치 노출 및 시각적 경고 스타일링 변경.
- 프론트엔드 상태 업데이트 주기를 조절하여(예: 200ms) 과도한 UI 렌더링 방어.
- 제어 액션 클릭 후 로딩 상태 시각화.
- 화면 내 에러 발생 내역을 상세히 열람할 수 있는 팝업/모달 제공.
- 프론트엔드 위주의 버그 픽스와 UX 개선 및 관련 테스트 코드 확보.

**Out-of-Scope**
- 백엔드 Loop Engine 아키텍처나 실제 루프 평가 알고리즘(Evaluator/Analyzer)의 구조적 변경.
- 프론트엔드 상태 쓰로틀링 시 중간에 생략된 과거 상태 내역을 강제로 모두 보관하고 재생하는 복잡한 Time-travel 디버깅 뷰 구현.
- 신규 데이터베이스 테이블이나 백엔드 마이그레이션을 요구하는 대규모 로그 저장 시스템.

## 3. Completion criteria
- `<T>`가 포함된 텍스트 로그가 웹 화면상에서 유실 없이 렌더링되며, `<script>` 등의 악성 코드는 제거됨을 검증.
- `LoopMonitorWidget`에서 설정된 `max_loop_count`를 초과할 경우 초과 수치(예: +5)가 화면에 올바르게 노출되고 글색이 붉게 렌더링됨을 검증.
- 엔진의 제어 버튼 클릭 시 즉각적으로 비활성화(Disabled) 처리되며 로딩 인디케이터가 표출됨을 검증.
- 네트워크 단절 모의 시 화면에 재연결 중임을 알리는 뱃지가 나타남을 검증.
- 추가된 단위/통합 테스트 스위트가 CI 환경에서 100% Pass 상태로 완료됨을 확인.
- `npm run dev -- --port 3100` 명령어를 통해 로컬 3100번 포트에서 프론트엔드가 오류 없이 실행됨을 확인.

## 4. Risks and test strategy
**Risks**
- XSS 필터 개선 과정에서 정규 표현식 엣지 케이스로 인해 의도치 않은 HTML 속성이 허용되거나 기존 정상 렌더링 로직이 망가질 위험성.
- SSE 렌더링 쓰로틀링 주기가 너무 길게 설정될 경우, 사용자가 느끼기에 엔진 상태 갱신이 체감상 멈춰 있는 것처럼 답답하게 느껴질 위험.
- 상태 갱신 지연으로 인해 '진행 중인 루프 수치'와 사용자가 'Pause' 버튼을 누른 시점 간의 미세한 Race condition 발생 가능성.

**Test Strategy**
- **단위 테스트 (Unit Test)**: `web/src/utils/security.test.ts`에 XSS 공격 페이로드 샘플과 정상적인 제네릭 문장(`<T>`, `<any>`)을 다수 추가하여 필터링 우회 여부를 엄격히 검증.
- **컴포넌트 테스트 (Component Test)**: `web/src/components/LoopMonitorWidget.test.tsx`에서 루프 초과 상태(`cycleCount > maxLoopCount`) 객체를 Props로 주입 후 렌더링된 요소의 텍스트와 적용된 클래스명 단언.
- **통합/동작 테스트**: `web/src/App.test.tsx`를 통해 `dropped` 에러 상태 이벤트를 모킹하여 모달과 토스트가 순차적으로 올바르게 렌더링되는지 통합 검증.

## 5. Design intent and style direction
- **기획 의도**: 개발자 및 AI 관리자가 24시간 돌아가는 Self-Improvement Loop 엔진의 상태를 한눈에 신뢰성 있게 모니터링하고, 돌발 에러 상황에서 직관적으로 개입(수동 중단, 에러 복사 등)할 수 있는 안전한 대시보드 경험을 제공.
- **디자인 풍**: 불필요한 장식을 덜어내고 데이터 가독성에 집중한 "모던 대시보드형 (Modern Dashboard)" 및 "카드형 (Card-based)" UI. 개발자에게 친숙한 터미널/콘솔 느낌을 곁들임.
- **시각 원칙**:
  - **컬러**: 정보의 위계에 따라 무채색(기본 텍스트 및 테두리)을 베이스로 사용하되, 상태에 따라 차분한 블루/그린(정상 작동), 옐로우(대기/지연), 강렬한 레드(초과/에러)를 포인트 컬러로 활용하여 즉각적인 시각적 피드백 제공.
  - **패딩/마진**: 카드 요소 내외부에 일관성 있고 넉넉한 여백(1rem ~ 1.5rem)을 주어 정보가 빽빽해 보이지 않도록 함.
  - **타이포그래피**: 루프 횟수, ID, 상태 코드 등 명확한 분별이 필요한 수치 데이터는 `Monospace` (고정폭) 글꼴을 사용하여 가독성과 개발 도구 특유의 전문성을 강조.
- **반응형 원칙**: 모바일 우선(Mobile-first) 접근. 좁은 모바일 화면에서는 카드 위젯들이 1열로 수직 배치되며 가로 스크롤 없이 주요 수치를 확인할 수 있게 설계하고, 태블릿 이상에서는 2~3열 다단 그리드 형태로 확장됨.

## 6. Technology ruleset
- **플랫폼 분류**: web
- **프레임워크**: React, TypeScript, Vite 기반 환경으로 계획 및 구현.
- **상태 및 통신**: 브라우저 기본 `EventSource` 및 React Hooks(`useSSE`)를 활용하여 상태 업데이트 반영.
- **보안/필터링**: 기존 설치된 `DOMPurify` 라이브러리의 옵션 확장을 통해 XSS 방지 처리.
- **실행 환경**: 구현 및 확인 시 로컬 구동 포트는 3100번대 할당 권장 (`npm run dev -- --port 3100`).
