# PLAN

## 1. Task breakdown with priority

### [P0] 버그 수정: 자동 스크롤 제어 로직 구현
- **상태 관리 추가**: `SystemAlertWidget.tsx` 컴포넌트 내에 사용자 스크롤 개입 여부를 추적하는 `isAutoScrollPaused` (boolean) React State 구현.
- **이벤트 핸들러 구현**: 로그 리스트 컨테이너에 `onScroll` 이벤트 리스너를 부착.
  - 사용자가 스크롤을 위로 올리면 `isAutoScrollPaused` 플래그를 `true`로 설정하여 새 알림 수신 시 강제 최하단 이동을 차단.
  - 스크롤이 다시 최하단(미세 오차 범위 내)에 도달하면 플래그를 `false`로 해제하고 자동 스크롤을 재개.
- **성능 최적화**: 과도한 스크롤 이벤트 트리거를 방지하기 위해 Throttle 또는 Debounce(예: `lodash/throttle`)를 적용.

### [P0] 신규 기능: 시스템 알림 필터 칩(Chip) UI 및 필터링 구현
- **필터 UI 마크업**: `All`, `Error`, `Warning` 상태를 선택할 수 있는 칩(Chip) 형태의 버튼 그룹을 툴바 영역(`.system-alert-toolbar`)에 추가.
- **클라이언트 필터링**: 선택된 칩의 상태에 따라 현재 메모리에 적재된 `alerts` 배열을 렌더링 시점에 동적으로 필터링.

### [P1] UX 개선: 모바일 해상도(320px) 레이아웃 최적화
- **반응형 CSS 적용**: 기존 [CLR], [EXP] 액션 버튼과 새로 추가될 필터 칩이 모바일 환경에서 겹치거나 가로 오버플로우를 유발하지 않도록 `flex-wrap: wrap` 및 적절한 간격(gap/margin) 속성 부여.

### [P1] 품질 보증: 테스트 코드 작성 및 보강
- **컴포넌트 테스트**: React Testing Library를 사용하여 필터 칩 클릭에 따른 상태 변경 및 필터링된 목록 노출 여부를 검증하는 단위 테스트 추가.
- **E2E 테스트**: Playwright를 사용하여 다음 시나리오 검증.
  - 320px 모바일 에뮬레이터 환경에서 UI 오버플로우 및 겹침 미발생 확인.
  - 마우스/터치로 스크롤을 올린 상태에서 새 로그 삽입 시 `scrollTop` 위치가 고정되는지 확인.

### [P2] [고도화 플랜] 필터 결과 Empty State UI 제공
- **근거**: 리뷰 문서(Edge cases)에서 제기된 상태 불일치 버그 가능성 방지. 결과 목록이 0개일 때 `hasMore` 상태만 남아 사용자가 혼란을 겪는 UX 개선.
- **구현 경계**: 클라이언트 필터링 결과 배열 길이가 0일 경우, "해당 조건의 알림이 없습니다"와 같은 명확한 안내 텍스트나 아이콘을 화면 중앙에 노출.

### [P2] [고도화 플랜] 대량 데이터 렌더링 병목 방지 최적화
- **근거**: 알림 폭주 시 배열 재연산 및 DOM 렌더링에 의한 화면 프리징(브라우저 병목) 우려 해소. 기존 로직의 확장 기능.
- **구현 경계**: 알림 목록 필터링 연산에 `useMemo`를 적용해 불필요한 재계산을 방지하고, 렌더링되는 DOM 개수가 극단적으로 많아질 경우 가상 스크롤(Virtual Scroll) 라이브러리 도입을 준비할 수 있는 구조적 래퍼 컴포넌트 구성.

## 2. MVP scope / out-of-scope

- **MVP Scope**
  - 사용자의 스크롤 조작에 따른 자동 스크롤 일시 정지 및 재개 기능.
  - `All`, `Error`, `Warning` 3가지 필터 칩을 통한 클라이언트 사이드 로그 필터링 기능.
  - 320px 모바일 뷰포트에서 가로 스크롤바가 발생하지 않는 반응형 레이아웃 구성.
  - 신규 추가된 로직에 대한 컴포넌트 단위 테스트 및 E2E 테스트 통과.
- **Out-of-Scope**
  - 클라이언트 단을 넘어선 서버(API) 측의 로그 필터링 처리 및 전문 검색 기능.
  - 다른 대시보드 위젯 컴포넌트의 레이아웃 개편 및 전역 UI 구조 변경.

---

## 3. Completion criteria

- [ ] 시스템 알림 목록에서 과거 로그를 보기 위해 스크롤을 위로 올린 상태라면, 새로운 로그가 추가되어도 스크롤 위치가 유지되어야 한다.
- [ ] 스크롤을 다시 최하단으로 내리면 자동으로 새로운 로그를 쫓아가는 자동 스크롤 상태로 복귀해야 한다.
- [ ] `Error` 또는 `Warning` 칩 클릭 시 해당 레벨의 알림 항목만 목록에 노출되어야 한다.
- [ ] 브라우저 창 너비를 320px로 줄였을 때, 필터 칩과 기존 액션 버튼 간 겹침이 발생하지 않고 가로 스크롤바가 생성되지 않아야 한다.
- [ ] 필터 동작 및 목록 렌더링 개수 변경을 검증하는 컴포넌트 단위 테스트가 통과해야 한다.
- [ ] Playwright를 통한 모바일 해상도 렌더링 검증 및 자동 스크롤 동작 검증 E2E 테스트가 로컬 환경(예: `http://localhost:3000`)에서 정상 통과해야 한다.

## 4. Risks and test strategy

- **Risks**
  - **성능 저하**: 빈번한 `onScroll` 이벤트 발생과 그에 따른 상태 업데이트가 렌더링 프레임 드랍을 유발할 수 있음.
  - **경계 조건 모호성**: 스크롤이 완전한 최하단이 아닌 1~3px 정도 미세하게 위에 걸쳐 있을 때, 이를 '일시 정지'로 판단할지 '자동 스크롤 활성화'로 판단할지에 대한 모호함.
- **Test Strategy**
  - 스크롤 이벤트 핸들러에 지연(Throttle) 처리가 올바르게 동작하는지 확인하여 성능 영향을 최소화한다.
  - 스크롤 최하단 판별 수식에 임계값(Threshold, 예: 오차 5px 이내)을 두어 모호한 경계 조건에서도 안정적으로 동작하도록 테스트한다.
  - React Testing Library를 활용해 DOM 변경과 상태 변화를 고립시켜 테스트하고, Playwright 기반 E2E 테스트를 통해 실제 브라우저 엔진에서 모바일 뷰포트 크기와 DOM 크기 계산이 정확히 일치하는지 통합 검증한다. (E2E 테스트 실행 시 지정된 3000번대 포트를 사용한다.)

## 5. Design intent and style direction

- **기획 의도**: 서버/에이전트로부터 지속해서 쏟아지는 로그 알림 속에서도 사용자가 페이스를 잃지 않고 중요한 경고/에러 내역을 추적, 탐색할 수 있는 제어권을 부여하는 것을 핵심 경험으로 삼는다.
- **디자인 풍**: 대시보드형의 미니멀하고 모던한 스타일. 부가적인 장식을 최대한 배제하여 사용자의 시선이 텍스트 데이터(로그)에 온전히 집중되도록 구성한다.
- **시각 원칙**:
  - **컬러**: 정보의 심각성을 직관적으로 인지할 수 있도록 기존에 정의된 로그 레벨별 시스템 컬러(Error=Red 계열, Warning=Yellow/Orange 계열)를 필터 칩의 활성화 상태에도 일관성 있게 적용한다.
  - **패딩/마진**: 필터 칩과 버튼 그룹은 좁은 화면에서도 오터치를 방지할 수 있도록 충분한 내부 여백(Padding 최소 8px 이상)과 요소 간 간격(Gap)을 보장한다.
  - **타이포**: 로그 출력부는 기존의 고정폭(Monospace) 폰트를 유지하여 코드 가독성을 보존하고, UI 요소는 시스템 기본 폰트로 명확하게 구분한다.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 준수한다. 작은 화면(320px) 기준에서 세로 정렬이나 줄 바꿈(`wrap`)이 자연스럽게 일어나게 한 뒤, 가로 너비가 넓어질 때 툴바 영역이 가로로 확장되도록 레이아웃을 구현한다.

## 6. Technology ruleset

- **플랫폼 분류**: web
- **기술 스택**: 현재 저장소에 구성된 React 기반 기술(Vite/TypeScript) 및 테스트 프레임워크(React Testing Library, Playwright)를 그대로 활용.
- **변경 파일 후보 및 영향 범위**:
  - `web/src/components/SystemAlertWidget.tsx`: 스크롤 이벤트 제어 및 필터 상태 관리, UI 요소 추가로 인한 직접적인 수정. 컴포넌트 내부로 영향 범위 한정.
  - `web/src/components/SystemAlertWidget.test.tsx`: 새로 추가된 기능 검증을 위한 컴포넌트 단위 테스트 코드 추가.
  - `web/tests/e2e/system-alert.spec.ts` (또는 기존 E2E 테스트 파일): 320px 모바일 화면 및 스크롤 고정 시나리오 검증용 테스트 코드 추가. (실행 포트는 `3000`번대 지정)
  - `web/src/components/SystemAlertWidget.css` (또는 연관 스타일 파일): 툴바 `flex-wrap` 처리 및 미디어 쿼리 조정 시 영향.
