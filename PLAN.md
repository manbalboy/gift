## 2. MVP scope / out-of-scope

**MVP scope:**
- `check-port.mjs`의 재시도 지연 시간 증가 및 동시성(Race condition) 방어 코드 적용
- Playwright 기반의 E2E 시각적 테스트 코드 작성 (SystemAlertWidget 뷰포트 오버플로우 검증)
- API 서버의 로그 목록 조회 시 쿼리 정렬 인덱스(`created_at`) 최적화
- 프론트엔드 UI의 세로 오버플로우 방어를 위한 스크롤 CSS 추가
- 포트 고갈 시뮬레이션 BASH 테스트 스크립트 작성

**Out-of-scope:**
- 초장기 실행으로 인해 발생하는 전체 노드 런타임의 DB I/O 지연 및 로그 페이징 처리 등 아키텍처 레벨의 성능 튜닝
- 지정된 프론트엔드 알림 위젯 이외의 전체 UI 컴포넌트 리팩토링
- Workflow Engine 루프 예산(budget) 제한 등 백엔드 코어 비즈니스 로직 수정

## 3. Completion criteria

- 여러 인스턴스가 동시에 `check-port.mjs`를 실행할 때, 포트 경합으로 인한 실패 없이 3100번대 포트를 정상적으로 각각 할당받아 반환해야 합니다.
- 새롭게 추가된 Playwright E2E 테스트가 모든 뷰포트(데스크톱 및 모바일 해상도) 환경에서 정상 통과하며 레이아웃 붕괴가 없음을 입증해야 합니다.
- 백엔드 `/api/logs` 호출 시 `created_at` 역순 스캔 쿼리가 DB의 인덱스를 타고 효율적으로 실행됨이 코드(스키마)로 확인되어야 합니다.
- 임의의 프로세스가 3100번대 포트를 모두 점유하고 있을 때, 작성한 검증 스크립트를 통해 `check-port.mjs`가 적절히 타임아웃 에러를 발생시키며 종료됨을 입증해야 합니다.

## 4. Risks and test strategy

- **Risks:** 동시성 처리 로직 변경 시 포트 반환 타이밍과 시스템의 네트워크 상태에 따라 CI/CD 환경에서 간헐적 테스트 실패(Flaky Test)가 발생할 수 있습니다.
- **Test strategy:**
  - **단위/통합 테스트:** DB 인덱스 적용 후 로그 조회 API 쿼리가 의도대로 매핑되는지 API 테스트 코드를 통해 응답 및 성능(타임아웃 여부)을 점검합니다.
  - **E2E UI 테스트:** Playwright를 통해 텍스트 노드가 비정상적으로 길게 렌더링되었을 때 부모 컨테이너(`Bounding Box`)를 넘어서는지 스크린샷과 상태 값을 비교해 검증합니다.
  - **스크립트 검증:** 실행 가이드에 따라 `nc` 혹은 로컬 웹 서버를 사용해 3000번대(3100-3199) 포트를 일시적으로 막아둔 뒤 `check-port.mjs`의 타임아웃 실패 동작을 검증합니다.

## 5. Design intent and style direction

- **기획 의도:** 시스템 장기 실행 과정에서 발생하는 에러 및 경고를 사용자가 어떤 디바이스 환경에서도 누락 없이, 그리고 레이아웃의 불편함 없이 파악할 수 있도록 신뢰감을 주는 경험을 전달합니다.
- **디자인 풍:** 대시보드형 뷰 내부에 이질감 없이 녹아드는 직관적이고 모던한 에러 알림 위젯 컴포넌트 형태입니다.
- **시각 원칙:** 에러와 경고를 색상으로 명확히 구분하며, 내부 텍스트가 아무리 길어도 레이아웃 밖으로 벗어나지 않도록 강한 Containment(컨테인먼트) 원칙과 넉넉한 내부 패딩을 유지합니다.
- **반응형 원칙:** 모바일 우선(Mobile-first) 규칙을 적용해, 좁은 너비의 모바일 화면에서는 위젯이 화면의 최대 가로를 채우고 정보가 길면 내부 스크롤을 유도하여 뷰포트 이탈을 방어합니다.

## 6. Technology ruleset

- **플랫폼 분류:** web / api
- **web:** React 프레임워크(Vite 환경) 기반으로 설계 및 구현하며, 스타일 수정과 Playwright E2E 검증 테스트에 활용합니다.
- **api:** FastAPI 기반 로직으로 설계하며, 데이터베이스 최적화 스키마(SQLAlchemy)를 중점으로 둡니다.
- **실행 포트 정책:** 로컬 환경 구동 시 테스트 스크립트 등에서 별도의 임시 서버 포트가 필요한 경우, 시스템 간섭을 피하기 위해 3000번대 포트(예: 3000, 3100-3199)만 고정적으로 사용하도록 계획합니다.
