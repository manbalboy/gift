# PLAN

## 1. Task breakdown with priority

### 우선순위: 높음 (P0) - 코어 엔진 안정화 및 보안
- **API 호출 의존성 제거 및 비동기 워커 전환**: 클라이언트 의존적인 폴링/호출 구조를 탈피하고, `workflow_id` 기반 백그라운드 워커 중심 비동기 실행 엔진으로 전면 리팩토링.
- **노드 전이 로직 정상화**: 단순 배열(Sequence) 방식에서 실제 연결선(Edges) 기반의 조건부 전이 로직 활성화 및 순환 참조(DAG 위반) 방어 로직 적용.
- **Webhook 보안 및 인가 검증**: `api/app/api/webhooks.py` 내 Webhook 수신 엔드포인트에 HMAC 서명 암호화 검증 및 허용 IP 대역 인가 확인 로직 구현. (인가되지 않은 차단 내역은 프론트엔드 대시보드 모니터링 연동)
- **Human Gate 권한 검증**: 노드 승인(Approve)/반려(Reject) 요청 단계에서 파이프라인 조작을 시도하는 요청 주체의 실제 권한 유효성을 검증하는 인가 로직 추가.

### 우선순위: 중 (P1) - 안정성 확보 및 UX 개선
- **자원 누수 방지 (Graceful Shutdown)**: 워크플로우 강제 취소 시 백엔드 활성 스레드 및 연결된 SSE 채널을 안전하게 회수하여 메모리 누수를 방지하는 Shutdown 프로세스 도입.
- **자동 재시도 (Backoff Retry)**: 노드 실행 실패 시 즉시 Failed 처리하지 않고, 최대 지정 횟수(예: 3회) 내에 점진적으로 대기 시간을 늘려 재시도하는 자가 복구 로직 엔진 레벨 적용.
- **Toast 컴포넌트 최적화 및 엣지 케이스 방어**: `web/src/components/Toast.tsx` 다중 에러 발생 시 동시 노출 최대 3개 제한(큐잉 시스템 적용), 자동 소거 주기(`durationMs`) 방어 로직 및 텍스트 줄바꿈(`word-break: break-all`) 스타일 적용.
- **대용량 아티팩트 렌더링 방어**: 수십 MB 크기의 로그나 스크린샷 뷰어 호출 시 브라우저 응답성 보호를 위해 Chunk Loading 및 가상화(Virtualization) 렌더링 기법 적용.

### 우선순위: 낮음 (P2) - 테스트 인프라 및 고도화 
- **백엔드 검증망 강화**: Webhook HMAC/IP 방어 로직 통합 테스트(pytest) 작성, 휴먼 게이트에 의한 워크플로우 Resume(상태 이행) 테스트 로직 작성, 다중 SSE 커넥션 생성 및 해제를 통한 스트레스/자원 누수 검증.
- **프론트엔드 UI E2E 커버리지 확대**: `http://localhost:3100` 환경 기준 ReactFlow 캔버스의 노드 연결, 순환 참조 에러, 예외 처리 등을 검증하는 Playwright 자동화 테스트 스크립트 추가.
- **[고도화] 알림(Notification) 센터 구축**: 일시적인 Toast 컴포넌트를 넘어, 우측 상단 패널에 알림 이력을 누적하여 부재 중 발생한 웹훅 차단 및 워크플로우 상태 변경(에러/승인 대기) 내역을 사후 확인할 수 있도록 UI 확장.
- **[고도화] 상태 기반 워커 헬스체크 및 DLQ 재할당**: 장애가 발생하여 응답 없는 워커 스레드의 태스크를 Dead Letter Queue(DLQ) 처리하고 다른 가용 워커에 재할당하는 모니터링/복원 체계 내장.

## 2. MVP scope / out-of-scope

### MVP Scope
- `workflow_id`를 기준으로 백엔드에서 온전히 사이클을 제어하는 비동기 워커 오케스트레이션 엔진 구축.
- ReactFlow를 이용한 노드/엣지 연결, 조건 전이, 유효성(DAG) 검증, 프리뷰 실행 기능 안정화.
- Webhook HMAC 서명 검증 및 휴먼 게이트 인가 제어를 통한 파이프라인 무결성 확보.
- 대용량 로그 렌더링 크래시 방어 및 다량의 Toast 알림 폭주를 방어하는 UI 응답성 개선.
- SSE 채널 누수 방지 및 실패 노드 자동 재시도를 통한 장기 가동 프로세스의 안정성 확보.

### Out-of-Scope
- 멀티 클러스터 환경에서의 글로벌 분산 큐 시스템(RabbitMQ/Kafka 등) 도입 (현재는 단일 애플리케이션 및 Redis 수준으로 제한).
- 실시간 동시 편집 기능을 갖춘 완전한 멀티플레이어 에디터 구축 (현재는 단일 사용자 저장/검증 수준).
- 자체적인 CI/CD 컨테이너 프로비저닝 시스템 내재화 (외부 시스템 트리거 연동까지만 한정).

## 3. Completion criteria
- 모든 Node 전이가 배열 순서가 아닌 Edge 설정 조건에 따라 정확하게 100% 이행되어 순환 참조 없이 종료되어야 함.
- 비정상 Webhook 페이로드 및 미승인 IP 주입 시 백엔드에서 401/403을 반환하고 대시보드 차단 로그 패널에 1초 이내에 반영되어야 함.
- 50MB 이상의 텍스트 로그 및 이미지 로드 시 브라우저 프리징 현상 없이 1초 내에 첫 화면을 렌더링(Chunking/가상화) 해야 함.
- 진행 중인 워크플로우를 강제로 취소할 때 백엔드 활성 워커 스레드 및 파생된 파이썬 SSE 커넥션이 100% 반환(Orphan 커넥션 0개)되어야 함.
- 임의 노드 실패 에러 주입 시 설정된 재시도(Backoff) 횟수만큼 동작한 후 실패 상태로 안전하게 전환되어야 함.
- 작성된 신규 E2E 테스트(Playwright) 및 백엔드 단위/통합 테스트(pytest) 커버리지가 CI 상에서 모두 패스되어야 함.

## 4. Risks and test strategy

### Risks
- 엔진 리팩토링 시 비동기 큐/워커 병목: API 호출 의존성을 제거하고 백그라운드 워커로 전환함에 따라 로컬 단일 환경에서 데드락 혹은 병목이 발생할 수 있음.
- SSE 채널 동시 연결 제한: 다중 사용자가 대시보드 관측 시 서버의 파일 디스크립터(FD) 한계를 넘거나 채널 관리가 미흡해 메모리 릭이 가속화될 위험.

### Test Strategy
- **단위/통합 테스트 (pytest)**: 
  - 서명(HMAC) 위조 및 IP 제한 검증 로직 통합 방어 테스트.
  - 휴먼 게이트 일시 정지(Interrupt) 및 조건별 Resume 상태 전환 이행 테스트.
  - 가상의 워커 스레드 환경에서 Graceful Shutdown 시 잔류 스레드 정리 상태 검증.
- **서버 부하/스트레스 테스트**: 무작위 다중 SSE 커넥션 생성 후 강제 종료를 반복하는 파이썬 스트레스 스크립트를 통해 서버 메모리 프로파일링 및 잔류(Orphan) 커넥션 검수.
- **E2E UI 테스트 (Playwright)**: 
  - `http://localhost:3100` 프리뷰 환경 대상으로 ReactFlow UI 에디터의 사이클 생성 시도 시 서버 무한 루프 차단 및 프론트 경고 노출 검증.
  - 동시에 10개 이상의 오류가 Toast 컴포넌트에 주입될 때 3개 초과분이 정상 큐잉되어 순차 노출되는지 검사.

## 5. Design intent and style direction
- **기획 의도**: 복잡한 개발 자동화 워크플로우(SDLC)의 흐름과 각 단계 산출물을 시각적으로 투명하게 파악하고, 안정적인 제어(승인/반려/중단) 및 알림 경험을 지원하는 개발 관제 플랫폼 UX 제공.
- **디자인 풍**: 모던 테크 대시보드 형 (노드 그래프 기반 시각화, 어두운 백그라운드 영역에 상태를 명확히 나타내는 채도 높은 포인트 컬러 적용).
- **시각 원칙**: 
  - **Color**: Success(Green), Failed(Red), Pending(Amber), Running(Blue) 등 노드/로그 상태 정보를 직관적이고 일관된 시스템 컬러로 매핑.
  - **Spacing**: 대용량 로그 및 컴포넌트 정보가 화면에 밀집되지 않도록 여유 있는 패딩(Padding)과 마진(Margin)을 사용하여 시각적 분리 강화. 
  - **Typography**: 아티팩트/로그/코드 등 개발 데이터는 가독성이 높은 Monospace 폰트를 적용하고, 대시보드 지표 및 상태값은 시인성이 좋은 산세리프 폰트로 배치.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 기본으로 적용하되, 기능의 성격상 워크플로우 캔버스(ReactFlow)와 복합 대시보드 뷰어는 데스크탑 및 태블릿 가로 모드에 최적화된 유동형 레이아웃(Fluid Layout)을 중점적으로 제공하며 모바일 환경에서는 조회(View) 위주로 기능을 단순화.

## 6. Technology ruleset
- **플랫폼 분류**: `api` (백엔드 인프라 및 워커) / `web` (프론트엔드 대시보드 및 에디터)
- **web 규칙**: React + Vite 프레임워크 기반 구성 (ReactFlow, UI 렌더링). 로컬 E2E 테스트 및 프리뷰 서빙 포트는 `3100`을 기본값으로 사용.
- **api 규칙**: Python FastAPI 프레임워크 기반 구성 (비동기 워커 엔진, Webhook 처리, SSE 스트리밍). 로컬 실행 및 프리뷰 포트는 충돌을 피하기 위해 `3000`을 기본값으로 사용.
