# PLAN

## 1. Task breakdown with priority

### P0: 워크플로우 엔진 고도화 및 버그 수정 (Workflow Engine v2 & Fixes)
- `api/app/services/workflow_engine.py` 로직 수정: 연결(엣지)이 없는 독립 노드가 강제로 순차 실행되는 현상(DAG Fallback 예외)을 방지하고, 병렬 처리 큐로 올바르게 진입하도록 로직 보완.
- `api/app/api/workflows.py` 로직 수정: Human Gate 승인 엔드포인트에 사용자 Role(예: `reviewer`, `admin`) 기반 인가(RBAC) 확인 로직 도입. 단순 토큰 유무를 넘어 권한에 따른 거절(403) 처리 적용.

### P0: 웹훅 및 실시간 상태 관리 보강 (Webhooks & SSE Management)
- `api/app/api/webhooks.py` 로직 수정: 웹훅 수신 시 잘못된 포맷이나 유효하지 않은 `workflow_id`에 대해 명확하게 HTTP `422 Unprocessable Entity`를 반환하도록 에러 핸들링 추가.
- `api/app/services/workflow_engine.py` (또는 스트리밍 관련 핸들러): 워크플로우 실행 강제 취소 API 호출 시, 활성화된 SSE(Server-Sent Events) 커넥션을 즉시 해제하여 좀비 커넥션 및 메모리 누수를 방지하는 자원 관리 로직 구현.

### P1: UI 연동 고도화 및 엣지 케이스 대응 (Frontend Enhancements)
- **인가 실패 시나리오 Fallback**: 프론트엔드(`web/src/components` 등)에서 403 에러 발생 시 콘솔 에러로 끝내지 않고, "필요 권한 안내 및 관리자 문의"를 유도하는 모달 컴포넌트를 출력하여 UX 저하 방지.
- **실시간 스트리밍 인디케이터**: GNB 상단에 SSE 커넥션 상태(연결 중/연결됨/단절됨)를 시각적으로 보여주는 상태 표시등 인디케이터 컴포넌트 추가.
- **실행 환경 포트 고정**: `web` 환경 변수 및 구동 스크립트를 정비하여 프론트엔드는 `3100` 포트, API는 `3101` 포트를 고정적으로 사용하도록 설정.

### P2: 테스트 커버리지 확보 (Testing)
- **프론트엔드 E2E 테스트**: `web/tests/e2e/` 경로에 Playwright 기반 통합 테스트 스크립트 작성. Human Gate의 라이프사이클(대기 -> 승인/반려 -> 워크플로우 재개) 시나리오 검증.
- **백엔드 예외 및 권한 단위/API 테스트**: 
  - 워크플로우 취소(Cancel) 시 `asyncio.CancelledError`가 정상적으로 포착되고 안전하게 종료되는지 검증하는 단위 테스트 작성.
  - 인가(Role 기반 검증) 및 유효성(422 반환)을 입증하는 API 통합 테스트 케이스 추가.

### P3: 고도화 플랜 (REVIEW.md 연계 인접 기능)
- **기능 1: SSE 커넥션 재연결(Auto-Reconnect) 및 백오프 정책 추가**
  - **근거**: 네트워크 단절 시 커넥션 상태 표시등이 붉은색(단절)으로 변경될 때, 사용자가 수동 새로고침을 하지 않아도 시스템이 점진적 백오프 정책(Exponential Backoff)을 통해 재연결을 시도하도록 하여 연결 안정성을 확보합니다.
  - **구현 경계**: 클라이언트의 EventSource 연결 관리 훅(Hook)을 업데이트하고, 재연결 시도 횟수를 제한합니다.
- **기능 2: Human Gate 승인 이력 감사 로그(Audit Log) 추가**
  - **근거**: 권한 검증 기능이 고도화됨에 따라, 어떤 `reviewer`나 `admin`이 승인 혹은 반려를 처리했는지 이력이 남아야 조직형 SDLC에 부합합니다.
  - **구현 경계**: 결정을 내린 주체(`decided_by`), 결정 일시(`decided_at`), 페이로드(`payload`)를 DB(`approval_requests` 또는 신규 `audit_logs` 테이블)에 기록하고 승인 UI에 타임라인 형태로 짧게 노출합니다.

## 2. MVP scope / out-of-scope

- **MVP Scope**:
  - `REVIEW.md`에서 식별된 DAG Fallback 버그, 잘못된 웹훅에 대한 422 상태 코드 반환, SSE 자원 누수 버그 픽스.
  - Human Gate 승인 시 Role 기반 권한 확인(백엔드) 및 권한 부족 시 모달 안내(프론트엔드).
  - SSE 연결 상태 시각화 컴포넌트 추가 및 백엔드 취소 이벤트의 안전한 종료.
  - 로컬 구동 시 프론트엔드 UI `3100`, API `3101` 포트 고정 적용 및 테스트.
  - Playwright E2E 통합 테스트(Human Gate 흐름) 및 백엔드 보강 테스트.
  - 인접 기능으로 추가된 프론트엔드 자동 재연결 로직 및 백엔드 승인 이력 기록 기능.
- **Out-of-scope**:
  - ReactFlow를 활용한 Visual Workflow Builder 전체 구현(다음 Phase로 이월).
  - Agent Marketplace 및 외부 도구(SDK)와의 연동/추상화 로직 확충.
  - AWS 등의 클라우드 인프라 프로비저닝 스크립트 작성.

## 3. Completion criteria

- 독립 노드가 존재하는 워크플로우가 멈추거나 순차 실행되지 않고 정상적으로 병렬 큐를 거쳐 실행 완료되어야 한다.
- 존재하지 않는 `workflow_id`로 웹훅 트리거 시 `422 Unprocessable Entity` 응답이 와야 한다.
- `reviewer`나 `admin` Role이 없는 사용자가 승인 요청 API를 호출할 경우 HTTP 403을 응답하고, 프론트엔드는 이 에러를 잡아 즉시 사용자 친화적인 모달을 띄워야 한다.
- 실행 중인 워크플로우를 강제로 취소할 시, 서버 상에 남아있는 SSE 커넥션이 완전히 해제되며 콘솔에 어떠한 좀비 프로세스 관련 에러도 남지 않아야 한다. (관련 Pytest 단위 테스트 패스)
- API 서비스가 `localhost:3101`, Web UI 서비스가 `localhost:3100`으로 고정 노출되어야 한다.
- E2E Playwright 테스트가 고정된 포트에서 구동되어 100% 성공(Pass) 상태를 유지해야 한다.

## 4. Risks and test strategy

- **Risks**: 
  - 취소 시그널에 따른 비동기 제너레이터(SSE) 강제 종료 로직이 불안정할 경우 서버의 비동기 이벤트 루프가 블로킹되거나 무한 대기에 빠질 리스크가 존재.
  - 자동 재연결 로직의 백오프 정책 설계가 잘못될 경우 서버에 과도한 연결 재시도 부하가 생길 수 있음.
- **Test Strategy**:
  - **Backend**: 취소/강제 종료 시나리오를 모킹하여 `asyncio.CancelledError`가 정상 트리거되고 컨텍스트 매니저가 메모리를 해제하는지 검증하는 단위 테스트(`pytest`) 강화. 권한/웹훅 엣지 케이스는 API 통합 테스트로 확인.
  - **Frontend**: API를 `3101` 포트에 띄운 후, 프론트엔드 `3100` 포트에서 `web/tests/e2e/` Playwright 스크립트를 실행. 브라우저 환경에서 SSE 연결 상태등이 올바르게 점등되는지, 휴먼 게이트 모달 플로우가 동작하는지 시각적/기능적 단계를 검사.

## 5. Design intent and style direction

- **기획 의도**: 개발/배포 자동화 파이프라인 안에서 사용자의 확인이 필요한 순간(Human Gate)을 명확하게 가시화하여, 조작 실수를 방지하고 권한에 맞는 안전한 검수 경험을 제공한다. 실시간 상태(SSE)의 투명한 공개를 통해 시스템 신뢰도를 높인다.
- **디자인 풍**: **모던 대시보드형 및 카드형 UI**. 복잡한 로그와 워크플로우 노드 데이터를 구역별 카드로 나누어 정보를 덩어리(Chunk)로 인지하기 쉽도록 구성.
- **시각 원칙**:
  - **컬러**: 시스템의 상태를 색상만으로 인지할 수 있도록 시맨틱 컬러(대기/Gate: 주황색, 통과/성공: 녹색, 오류/차단: 붉은색, 진행 중: 파란색)를 적극 사용. 권한 없음 모달 등은 경고색으로 강조.
  - **패딩/마진**: 데이터가 빽빽하지 않도록 카드 내부 패딩(예: `1rem`~`1.5rem`)과 둥근 모서리(`border-radius`)를 주어 부드럽고 여유로운 컴포넌트 여백을 유지.
  - **타이포**: 장문의 로그나 아티팩트 내용은 가독성을 위해 모던 산세리프(San-serif) 폰트의 Regular 웨이트를 사용하고, 각 단계의 제목/상태는 Bold 웨이트로 대비 부여.
- **반응형 원칙**: **모바일 우선(Mobile-First)** 기반으로 설계. 태블릿 및 모바일 기기에서도 승인/반려 버튼 오클릭이 없도록 터치 타겟 사이즈를 충분히 확보하고, 모달 및 로그 영역은 잘림 없이 스크롤할 수 있도록 반응형 레이아웃 구성.

## 6. Technology ruleset

- **플랫폼 분류**: web, api
- **web**: React (Vite 기반, TypeScript 적용) 프론트엔드 아키텍처로 컴포넌트 단위 구현 (포트 3100). Playwright 이용 E2E 테스트 구성.
- **api**: FastAPI 기반의 비동기 백엔드 서버 로직 구현 (포트 3101). 비동기 스트리밍(SSE), 의존성 주입(Role 기반 권한 확인 로직 포함)을 철저히 활용하며 Pytest 이용 단위/통합 테스트 구성.
