# PLAN

## 1. Task breakdown with priority

### 우선순위: 높음 (P0) - 코어 엔진 안정성 보완 및 고도화
- **T1. Workflow Engine v2 안정화**: `workflow_id` 기반 정의 실행 및 `node_runs` 저장 로직이 정상 동작하는지 점검. `default_linear_v1` 폴백 처리 검증. (백엔드)
- **T2. 리버스 프록시 환경 SSE 타임아웃 대응 (REVIEW TODO)**: 로컬 3100번대 포트 테스트 외에, Nginx 등 리버스 프록시 환경에서 발생하는 SSE 장기 연결 타임아웃 엣지 케이스 방어. 백엔드 Keep-Alive 주기 최적화 및 프론트엔드 자동 재연결 로직 고도화.
- **T3. Human Gate 재개(Resume) 멱등성 보장**: 휴먼 게이트 승인/수정/거절 액션 시 중복 요청에 대한 멱등성 보장 로직 유지 및 상태 충돌 완전 방지.

### 우선순위: 중간 (P1) - 관측성 및 알림 기능 확장
- **T4. 대시보드 Audit Log 검색 기능 구현 (REVIEW TODO)**: 각 워크스페이스에 저장된 `status.md` 아티팩트 데이터를 추출/인덱싱하여, 대시보드 UI에서 과거 휴먼 게이트 결정 이력과 상태를 조회 및 검색(기간, 상태 필터링)할 수 있는 기능 추가. (프론트엔드 및 백엔드 API 연동)
- **T5. 장기 미처리 휴먼 게이트 알림 정책 적용 (REVIEW TODO)**: 대기 상태(`waiting` 또는 `approval_pending`)로 일정 시간(예: 24시간) 이상 지연된 휴먼 게이트 건을 스케줄러가 탐지하여 관리자에게 만료 경고 및 재개(Resume) 촉구 알림을 발송하는 로직 기획 및 구현.

### 우선순위: 낮음 (P2) - 차기 인프라 사전 작업
- **T6. Artifact-first 구조 통합**: `code_change_summary.md`, 테스트 리포트 등 산출물들을 `status.md` 뷰어와 유사한 방식으로 대시보드 런 타임라인에 연동하기 위한 백엔드 스키마 준비.

## 2. MVP scope / out-of-scope

### MVP Scope
- 기존 휴먼 게이트 기능의 통신 안정성 극대화 (특히 리버스 프록시 환경의 SSE 연결).
- 프론트엔드 대시보드에 `status.md` 데이터를 활용한 Audit Log 검색 뷰어 페이지 추가.
- 백엔드에 장기 대기 워크플로우를 스캐닝하여 알림 이벤트를 생성하는 스케줄러 API 탑재.
- API 및 Web 서버 포트는 3100번대 할당 유지 (Web: 3100, API: 3101).

### Out-of-scope
- ReactFlow 기반의 완전한 Visual Workflow Builder 구현 (이후 Phase에서 진행).
- 실제 이메일 발송 서버 연동 (현 단계에서는 시스템 내부 알림 및 로그 이벤트 기록으로 한정).
- PR, CI, 배포 서버 등과의 외부 웹훅 전체 통합 (이벤트 버스 구축 이후 진행).

## 3. Completion criteria
- 테스트용 Nginx 컨테이너를 거쳐 프론트엔드(포트 3100)와 백엔드(포트 3101) 간 SSE 통신 시, 강제 지연 및 유휴 상태에서도 연결이 끊기지 않거나 즉각 복구되어야 함.
- 대시보드 Audit Log 탭에서 날짜 범위(`date_range`) 및 승인/거절 상태(`status`) 쿼리를 통해 특정 워크플로우 런의 `status.md` 요약 내용을 오차 없이 렌더링해야 함.
- 의도적으로 휴먼 게이트를 24시간 전으로 조작한 테스트 워크플로우를 생성했을 때, 백엔드 스케줄러 훅이 이를 정확히 식별하고 알림 대상 리스트에 포함하는지 백엔드 단위 테스트를 통해 입증되어야 함.

## 4. Risks and test strategy

### Risks
- **네트워크 타임아웃 엣지 케이스**: 프로덕션 수준의 웹 서버 환경에서 인프라 레벨의 TCP 타임아웃 설정과 애플리케이션 계층의 SSE 유지 로직 간 충돌 가능성.
- **아티팩트 파싱 성능 저하**: 수많은 워크스페이스의 `status.md` 파일을 실시간으로 읽고 파싱할 경우, 대시보드 로딩 지연 발생 우려.

### Test strategy
- **백엔드 (FastAPI)**: Nginx 리버스 프록시 모의 환경을 도커로 구성하여 장시간 스트리밍 부하 테스트를 수행. 스케줄러의 타임 모킹(Time Mocking) 단위 테스트를 작성해 장기 미처리 알림 로직 검증.
- **프론트엔드 (React)**: Playwright 기반 E2E 테스트를 확장하여, Audit Log 필터 기능 클릭, 날짜 선택, 결과 리스트 확인 과정을 자동화된 시나리오로 검증.

## 5. Design intent and style direction

- **기획 의도**: 워크플로우 진행 상태와 사람의 개입(Human Gate) 이력을 명료하게 시각화하여, 개발 조율 과정의 투명성을 높이고 병목 지점을 빠르게 식별할 수 있는 관측 경험 제공.
- **디자인 풍**: 군더더기 없는 모던 대시보드형 및 정보 밀도가 높은 데이터 테이블/카드형 스타일 혼합 적용.
- **시각 원칙**:
  - 상태에 따른 명확한 시맨틱 컬러 사용: 승인 및 정상(Green), 대기 및 경고(Yellow/Amber), 거절 및 실패(Red).
  - 배경은 가독성을 극대화하는 화이트 및 라이트 그레이 톤으로 구성하고, 로그 및 아티팩트 본문은 Monospace 폰트로 렌더링하여 개발자 친화적 가독성 확보.
  - 여유 있는 패딩(16px~24px)을 통해 정보 간의 시각적 분리 도모.
- **반응형 원칙**: 모바일 우선(Mobile-first) 원칙 준수. 뷰포트 너비가 좁은 모바일 기기에서는 각 Audit Log 항목을 독립적인 카드(Card) 형태로 배치하고, 태블릿 및 데스크탑 환경에서는 컬럼 기반의 넓은 그리드/테이블(Table) 포맷으로 매끄럽게 확장되도록 구현.

## 6. Technology ruleset

- **플랫폼 분류**: web 및 api
- **web**: React 및 Vite 기반 생태계 (현행 레포지토리 구조 유지, TypeScript 적용)
- **api**: FastAPI 기반 (비동기 처리, Pydantic 모델링, Python 3.10+ 권장)
- 포트 및 실행 컨텍스트: 프론트엔드는 3100, 백엔드는 3101 등 3000번대 포트를 기준으로 실무형 개발/테스트 가이드 작성.
