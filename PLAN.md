# PLAN

## 1. Task breakdown with priority

### High (P0) - 기능 결함 및 안정성 수정 (REVIEW 반영)
- **`resume_run` 멱등성 및 응답 일관성 확보**: `api/app/api/workflows.py` 내 `resume` 엔드포인트에서 동시 요청 시 멱등성을 보장하도록 `ValueError`(`400`) 반환을 `409 Conflict` 또는 `200 OK`로 수정.
- **Graceful Failure 응답 규격 개선**: 런타임 아티팩트 유실 등으로 인한 워크플로우 실패 시, 상태가 `failed`로 정상 커밋되었다면 `400 Bad Request` 에러를 던지지 않고 업데이트된 상태 모델과 함께 정상 응답을 반환하도록 리팩토링.
- **아티팩트 접근 OS 권한(Permission) 예외 처리**: 장기 실행 워크플로우 재개 시 파일 시스템 잠금 문제로 인한 크래시를 방지하기 위해, Workspace 디렉토리 아티팩트 검증 로직에 포괄적인 `try-except` 오류 핸들링 추가.

### Medium (P1) - 엔진 고도화 기반 및 테스트 보강 (SPEC & REVIEW 반영)
- **환경변수 파싱 안정성 개선 및 테스트**: 잘못된 `DEVFLOW_LOCALHOST_SPOOF_GUARD_PORTS` 환경변수 주입 시 예외 크래시 대신 명확히 종료되거나 안전한 기본값을 쓰도록 파서를 개선하고 경계값 테스트 케이스 추가.
- **에러 응답 DB 상태 검증 자동화**: Graceful Failure 시나리오 등에서 API 응답만 체크하던 기존 테스트에 DB 상의 `Run`/`NodeRun` 상태(`failed`)가 정확히 커밋되었는지 검증하는 Assertion 추가.
- **Workflow Engine v2 기초 연동**: `workflow_id` 기반 그래프 실행을 위한 초기 모델 설정 및 내구성 있는 체크포인트 로직 뼈대 작업.

### 고도화 플랜 (REVIEW TODO 연계 추가 기능)
- **Run-level Audit Log (실행 감사 로그) 연동**
  - **근거(왜 필요한지)**: Graceful Failure로 인한 `failed` 전이 시 클라이언트에 정상 모델을 내려주더라도, 세부 원인(OS Lock, Workspace Missing 등)의 명확한 사후 추적이 필요합니다. 이를 위해 상태 전이와 연관된 감사 로그를 남기면 디버깅 및 사용자 관측성이 대폭 향상됩니다.
  - **구현 경계**: 상태가 변경되는 시점에 예외 및 상태 정보를 DB 테이블(예: `RunAuditLog`)에 적재하고, 대시보드에서 조회할 수 있는 단일 API 엔드포인트(`/api/runs/{id}/audit`)를 추가 구현합니다. 범위 밖의 외부 시스템 파이프라인 연동은 제외합니다.

## 2. MVP scope / out-of-scope

### MVP Scope
- 기존 API 서버(`FastAPI`)의 워크플로우 제어(시작/중단/재개) 로직 중 동시성 이슈와 HTTP 응답 불일치 버그 해결.
- 시스템 환경/OS 차원의 권한 예외 발생 시 워커 프로세스 크래시를 방지하는 파일 I/O 안정성 보완.
- 누락된 설정 파서(`DEVFLOW_LOCALHOST_SPOOF_GUARD_PORTS`) 단위 테스트 및 상태 전이 검증 통합 테스트 보강.
- 변경된 API 응답(에러 대신 `failed` 객체 반환 등)을 프론트엔드 대시보드(React)에서 올바르게 렌더링하도록 클라이언트 연동 로직 수정.

### Out-of-scope
- 이번 페이즈에서는 Temporal이나 LangGraph 같은 외부 분산 내구 실행 엔진의 완전한 도입은 제외하고, 현재의 자체 Engine 기반 안정화에 집중합니다.
- 시각적 워크플로우 빌더(ReactFlow) 기능 전체 구현 및 CI/CD 이벤트 버스 통합 등은 차기 마일스톤(Phase 2)으로 이관합니다.

## 3. Completion criteria

- 다중 `resume` 동시 요청(동일 조건) 시 락 경합이 발생해도 `400` 내부 에러가 터지지 않고 `200` 또는 `409`로 일관성 있게 응답한다.
- 아티팩트나 런타임 데이터 유실로 워크플로우가 중단(Failed)될 경우, 클라이언트는 `400 Bad Request`가 아닌 갱신된 `Run` 상태 객체를 응답받는다.
- Workspace 디렉토리의 파일 읽기/쓰기에 OS Lock(권한 에러)이 걸려있을 때 런타임이 크래시되지 않고 로직이 이를 핸들링(Graceful Fail)한다.
- 관련 환경변수 파싱 및 롤백 상태 검증을 포함하는 모든 테스트 코드(`pytest`)가 통과한다.
- 로컬 Preview 환경 구성 시 컨테이너 포트 `7000~7099` 대역을 통해 정상적으로 서비스 접속 및 검증이 가능하다.

## 4. Risks and test strategy

### Risks
- **클라이언트 하위 호환성 파손**: 기존 `400` 에러 코드로 실패를 감지하던 프론트엔드 UI(React) 로직이, `200` 정상 응답과 `failed` 상태 프로퍼티를 받도록 변경됨에 따라 상태 처리 불일치로 인한 UI 버그가 발생할 수 있습니다.
- **초장기 실행의 좀비 프로세스화**: OS Permission 예외를 삼키기만 하고 프로세스를 정리하지 않으면 시스템 자원 누수가 발생할 수 있습니다.

### Test strategy
- **단위 테스트 (Unit Test)**: `config.py`의 `spoof_guard_ports` 파싱 로직에 대해 유효/무효/특수문자 문자열 등의 경계값 테스트 케이스를 집중 추가합니다.
- **통합 테스트 (Integration Test)**: `pytest`의 비동기 모킹을 이용해 `POST /api/runs/{id}/resume` 동시 호출을 발생시키고 멱등성 응답 코드를 검증합니다. 예외 발생 시나리오 연출 후 DB를 조회해 상태 커밋 정합성을 판단하는 Assertion을 필수로 추가합니다.
- **E2E 테스트 (UI)**: React 대시보드에서 고의적으로 워크플로우 아티팩트를 손상시킨 뒤 재개 버튼을 클릭했을 때 에러 알림이 아닌, 노드 다이어그램이 정상적으로 `Failed` 상태 색상으로 렌더링되는지 검증합니다.

## 5. Design intent and style direction

- **기획 의도**: 개발자 및 관리자가 24시간 자율 동작하는 워크플로우 상태를 통제할 때, 파일 잠금이나 예외 상황 등 예측 불가능한 장애가 발생하더라도 "시스템의 붕괴(Crash)"가 아닌 "통제된 실패(Graceful Failure)"로 명확히 안내받아 높은 관측성과 신뢰를 경험하도록 합니다.
- **디자인 풍**: 모던하고 심플한 형태의 **대시보드형 뷰**를 지향하며, 복잡한 워크플로우를 간결하게 모니터링할 수 있는 카드형 컴포넌트 배치를 사용합니다.
- **시각 원칙**:
  - **컬러**: 상태에 따른 인지가 즉각적이도록 시맨틱 컬러를 강하게 적용합니다 (Success: Green, Failed: Red, Paused: Orange, Running: Blue).
  - **패딩/마진**: 시스템 로그와 노드 상태 정보 밀도가 높으므로 각 패널 간 최소 `16px~24px`의 넉넉한 여백을 두어 정보 피로도를 낮춥니다.
  - **타이포**: 상태 요약 및 메뉴 등 기본 텍스트는 가독성 높은 산세리프(Sans-serif)를 사용하고, 터미널 출력 및 Audit Log와 같은 디버그 정보는 고정폭 폰트(Monospace)를 적용합니다.
- **반응형 원칙**: 모바일 환경에서도 상태 모니터링이 원활하도록 **모바일 우선(Mobile-First)**으로 카드형 뷰를 배열하되, 데스크탑 스크린에서는 로그와 다이어그램을 분할해서 볼 수 있는 다단 레이아웃(Split-view)으로 부드럽게 확장됩니다.

## 6. Technology ruleset

- **플랫폼 분류**: `web`, `api`
- **web 규칙**: 사용자 대시보드 및 워크플로우 시각화는 **React 기반**의 프레임워크와 도구(Vite, TypeScript 환경)를 사용하여 계획 및 구현합니다.
- **api 규칙**: 런타임 제어 및 상태 저장을 위한 백엔드는 **FastAPI 기반**으로 설계하며, 데이터베이스 연동(SQLAlchemy) 및 의존성 주입을 활용해 구조화합니다. (로컬 실행 포트는 가이드라인에 따라 충돌을 방지하며 Preview 포트는 `7000`번대를 채택합니다.)
