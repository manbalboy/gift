# PLAN

## 1. Task breakdown with priority

### P0 (Highest Priority) - Critical Bug Fixes & Security
1. **[Bug Fix] 포트 할당 레이스 컨디션 방지 및 타임아웃 검증**
   - **대상**: `web/scripts/check-port.mjs`, `web/scripts/test-port-timeout.sh`
   - **영향 범위**: 다중 워커의 로컬 구동 스크립트 및 테스트 파이프라인
   - **내용**: 3100번대 포트 점유 시 다중 워커 간 경합을 방지하기 위해 점유 유예 시간 설정 및 재시도 로직을 보완합니다. 이와 함께 포트 고갈 상황을 시뮬레이션하고 명시적으로 타임아웃 처리가 동작하는지 검증하는 테스트 스크립트를 작성합니다.

2. **[Security] 시스템 알림 로그 내 민감 정보 마스킹 (Log Sanitization)**
   - **대상**: `api/app/services/system_alerts.py` (또는 로깅 파이프라인)
   - **영향 범위**: 알림 로그 저장 로직 및 프론트엔드 데이터 전달 구간
   - **내용**: 시스템 절대 경로(예: `/home/docker/`) 및 인증 토큰 등의 민감 정보가 DB에 저장되거나 클라이언트에 노출되지 않도록 정규식을 활용하여 `***[MASKED]***`로 치환하는 필터링 로직을 반영합니다.

### P1 (High Priority) - UI/UX & Reliability Tests
3. **[UI/UX] SystemAlertWidget 레이아웃 오버플로우 수정**
   - **대상**: `web/src/components/SystemAlertWidget.tsx` 및 관련 스타일시트
   - **영향 범위**: 대시보드의 시스템 알림 위젯 UI 컴포넌트
   - **내용**: 많은 양의 텍스트가 렌더링될 때 발생하는 레이아웃 깨짐을 해결하기 위해 `overflow-y: auto`, `max-height`를 추가하고, 긴 단어의 경우 `word-break: break-all` 등의 줄바꿈 처리를 적용합니다.

4. **[Test] 뷰포트 교차 검증을 위한 시각적 E2E 테스트 신설**
   - **대상**: `web/tests/e2e/system-alert.spec.ts`
   - **영향 범위**: 프론트엔드 E2E 테스트 스위트
   - **내용**: Playwright를 이용해 데스크톱과 모바일 뷰포트에서 `SystemAlertWidget` 컴포넌트의 시각적 안정성을 보장하는 테스트를 추가합니다.

5. **[Test] 백엔드 무한 루프 차단 (Budget limit) 단위 테스트**
   - **대상**: `api/tests/test_workflow_engine.py` (또는 워크플로우 테스트 파일)
   - **영향 범위**: 워크플로우 엔진 런타임 및 재시도 제어 로직
   - **내용**: 에이전트 워크플로우가 재시도 횟수나 시간 예산을 초과했을 때 엔진이 정상적으로 개입하여 노드를 차단(Blocked)하는지 검증하는 단위 테스트를 작성합니다.

6. **[DB] 알림 로그 쿼리 성능 최적화 마이그레이션**
   - **대상**: `api/app/db/system_alert_model.py` 및 Alembic 마이그레이션 스크립트
   - **영향 범위**: `system_alert_logs` 테이블 스키마 및 로그 조회 성능
   - **내용**: 대시보드 로그 역순 조회 쿼리의 성능 지연을 막기 위해 `created_at` 컬럼 기준의 역순 정렬 스캔용 인덱스를 마이그레이션 파일로 생성하여 반영합니다.

### P2 (Medium Priority) - 확장 기능 추가 (인접 기능 고도화)
7. **[Feature] 워크플로우 제어 API 인가(Authorization) 적용**
   - **대상**: `api/app/api/workflows.py` 및 인증 의존성 컴포넌트
   - **근거**: 리뷰 문서에 제기된 외부 지시 주입 및 제어 API 접근의 취약점을 선제적으로 방어하기 위해 필요합니다.
   - **구현 경계**: 기존에 구성된 서명 검증(HMAC) 또는 Role 기반 토큰 검증 미들웨어를 워크플로우 중단/재개 및 지시 주입 API 엔드포인트에 최소 단위로 적용합니다.

8. **[Feature] Agent 루프 위험도(Risk Score) 지표 알림 연동**
   - **대상**: `api/app/services/workflow_engine.py`, `web/src/components/SystemAlertWidget.tsx`
   - **근거**: 극단적 로컬 포트 고갈이나 LLM 반복 루프로 인한 비용 폭주를 실시간으로 모니터링하고 관측성을 높이기 위해 필요합니다.
   - **구현 경계**: 노드 실행 중 동일한 오류가 연속 발생 시 내부적인 위험도 점수를 산정하여, 기준치 초과 시 알림 로그에 Warning 타입으로 강제 노출하는 로직을 엔진 쪽에 추가합니다.

## 2. MVP scope / out-of-scope

### MVP Scope
- 다중 워커 구동 시 3100번대 포트(3100~3199) 내에서 포트 할당 경합이 없도록 동시성 제어.
- 대시보드의 알림 로그 컴포넌트 레이아웃 최적화 및 E2E 테스트 확보.
- 로그 저장 시 민감 정보 마스킹 처리를 통한 보안성 확보.
- 워크플로우 무한 루프 진입 시 예산에 따른 자동 차단 로직 테스트 검증.
- 알림 데이터 역순 조회 시 성능 이슈 해결을 위한 DB 인덱스 마이그레이션 완료.

### Out-of-scope
- 대규모 OAuth 연동 등 기존 구조를 크게 벗어나는 새로운 인증 아키텍처 도입.
- 3100번대를 초과하는 동적 포트 할당 및 프록시 네트워크 전면 개편.
- 전체 데이터베이스에 대한 성능 튜닝 및 인프라 구조 변경.

## 3. Completion criteria
- `check-port.mjs` 및 다중 워커 실행 환경에서 로컬 3100번대 포트 할당이 정상 동작하며, 포트 고갈 시나리오 실행 시 명시적 타임아웃 종료를 검증해야 합니다.
- 프론트엔드 모바일/데스크톱 해상도에서 `SystemAlertWidget`의 스크롤 및 줄바꿈이 정상 동작함을 Playwright E2E 테스트로 증명해야 합니다.
- 민감 정보 토큰이나 절대 경로 문자열이 포함된 로그 입력 시, API 반환 값과 DB 저장 값 모두 `***[MASKED]***` 처리가 성공해야 합니다.
- 워크플로우 엔진이 설정된 실행 횟수/시간 예산을 초과하면, 노드의 상태가 정상적으로 차단(`Blocked` 등)되는지 확인하는 백엔드 단위 테스트가 통과해야 합니다.
- DB 마이그레이션 적용 후 `system_alert_logs` 테이블의 조회 쿼리가 내림차순 인덱스를 사용하여 응답 지연 없이 데이터를 반환해야 합니다.

## 4. Risks and test strategy

### Risks
- 대용량 로그 치환을 위한 정규식 연산이 백엔드에 CPU 오버헤드를 발생시킬 수 있습니다.
- `check-port.mjs` 작동 중 프로세스 강제 종료로 인해 남겨진 락(Lock) 파일이 영구적인 포트 고갈을 유발할 위험이 있습니다.
- 로컬 개발 환경이나 Preview 배포 환경에서 3100번대 포트가 외부 프로그램에 의해 이미 점유되어 있어 테스트가 간헐적으로 실패할 우려가 있습니다.

### Test Strategy
- 로그 마스킹 정규식 필터링 함수에 대해 복합적인 패턴의 문자열을 주입하는 단위 테스트를 작성하여 안정성을 검증합니다.
- 포트 락 스텔스 상황 및 고갈 상황을 재현하는 `test-port-timeout.sh` 시뮬레이션 테스트를 통해, 락 해제 및 타임아웃 로직의 견고함을 증명합니다.
- 모바일 및 데스크톱 뷰포트를 고정시킨 상태에서 시각적 E2E 회귀 테스트를 수행하여 UI 컴포넌트의 안정성을 보장합니다.

## 5. Design intent and style direction

- **기획 의도**: 개발자 및 시스템 운영자가 24시간 자율 실행되는 에이전트 환경에서 시스템 병목(Lock 경합) 및 보안 경고를 즉각적이고 투명하게 인지할 수 있는 핵심 모니터링 경험을 제공합니다.
- **디자인 풍**: 개발자 친화적인 모던 대시보드형 (Modern Dashboard) 및 정보 밀집도가 높은 카드형 스타일.
- **시각 원칙**: 
  - 타이포그래피: 로그 및 코드 영역은 가독성을 극대화하기 위해 모노스페이스(Monospace) 글꼴을 사용합니다.
  - 컬러/여백: 에러는 옅은 붉은색, 경고는 황색을 보조 색상으로 미니멀하게 사용하며, 좁고 일관된 패딩 마진(8px~16px)을 적용해 컴팩트한 레이아웃을 구성합니다.
- **반응형 원칙**: 모바일 우선(Mobile-First) 규칙을 적용합니다. 모바일에서는 카드 위젯이 가로 공간을 100% 점유하고 내부 콘텐츠만 스크롤 되도록 하며, 화면이 커짐에 따라 그리드 기반의 타일 레이아웃으로 확장되도록 유도합니다.

## 6. Technology ruleset

- **플랫폼 분류**: web, api
- **web (프론트엔드)**: React 프레임워크 기반으로 계획.
- **api (백엔드)**: FastAPI 기반으로 계획.
- **포트 정책**: 로컬 실행 가이드 등 포트가 필요한 환경 구성 시 오직 3100번대 포트만 사용합니다.
