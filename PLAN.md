# PLAN

## 1. Task breakdown with priority

### 우선순위 높음 (P0) - 코어 버그 수정 및 안정성 강화 (REVIEW 반영)
- **프론트엔드 폼 프리셋 텍스트 개행/공백 누적 버그 수정**
  - 대상 파일: `web/src/App.tsx`
  - 영향 범위: 사용자가 대시보드에서 `handleRejectReasonPreset` 등을 통해 프리셋 문구를 폼에 입력할 때의 텍스트 파싱 로직. 정규식을 최적화하여 텍스트 누적 시 발생하는 불필요한 여백 생성을 막습니다.
- **프론트엔드 폼 텍스트 단위 테스트 추가**
  - 대상 파일: `web/src/App.test.tsx`
  - 영향 범위: 컴포넌트 내 프리셋 입력 함수의 결과 문자열 검증 로직 추가.
- **백엔드 뷰어 토큰(Viewer Token) 인증 로직 강화**
  - 대상 파일: `api/app/api/dependencies.py` 등 API 미들웨어/라우팅 파일
  - 영향 범위: 전역 API 라우팅 단계에 토큰 검증 로직을 추가하여 3000번대 로컬 포트로 직접 접근(프록시 우회)하려는 시도를 차단합니다.
- **백엔드 엔진 무한 루프 감지 및 강제 일시 정지(Pause) 로직 구현**
  - 대상 파일: `api/app/services/workflow_engine.py`
  - 영향 범위: 워크플로우 실행 엔진 라이프사이클 내. 노드 실행 횟수를 카운팅하고 임계치 초과 시 즉시 실행 상태를 Pause로 전환하여 예산 고갈을 방지합니다.

### 우선순위 높음 (P0) - 워크플로우 검증 기능 강화 (REVIEW 반영)
- **워크플로우 빌더 UI 클라이언트 검증 로직 구현**
  - 대상 파일: `web/src/components/WorkflowBuilder.tsx` (혹은 관련 빌더 컴포넌트)
  - 영향 범위: 단절된 그래프(Disconnected graph)나 다중 진입점이 있는 상태에서 저장 버튼 클릭 시, API 요청을 보내지 않고 사전에 차단하는 밸리데이션 로직.
- **빌더 UI 사전 유효성 검사 E2E 테스트 추가**
  - 대상 파일: `web/tests/e2e/WorkflowBuilder.spec.ts`
  - 영향 범위: 비정상적인 그래프 데이터 구성 및 저장 시도 시 프론트엔드에서 에러 피드백을 올바르게 노출하는지 검증합니다.
- **백엔드 워크플로우 그래프 유효성 검증 테스트 전면 개편**
  - 대상 파일: `api/tests/test_workflow_engine.py`
  - 영향 범위: 이전 사양인 단절 그래프 허용 코드를 제거하고, 비정상 그래프가 수신되었을 때 `400` 혹은 `422` 상태 코드를 올바르게 반환하는지 테스트 코드를 수정합니다.

### 우선순위 보통 (P1) - 자연스러운 인접 기능 고도화 (추가 구현)
- **프론트엔드 에러 피드백(Toast) 공통 모듈화 고도화**
  - 대상 파일: `web/src/components/common/Toast.tsx` 및 관련 훅(Hook)
  - 근거: REVIEW에서 요구한 "빌더 UI 에러 피드백" 추가에 맞추어, 향후 다양한 화면(무한 루프 경고, 권한 에러 등)에서 일관된 UX로 에러를 노출하기 위해 기존 파편화된 토스트 UI 상태 관리를 재정비합니다.
  - 구현 경계: 디자인 시스템 톤앤매너에 맞춘 토스트 UI 컴포넌트 추상화 및 전역 컨텍스트 기반 알림 트리거 도입.
- **워크플로우 엔진 노드별 타임아웃(Timeout) 방어 로직 추가**
  - 대상 파일: `api/app/services/workflow_engine.py`
  - 근거: REVIEW의 무한 반복 카운팅(Agentic Loop 방지)과 결을 같이하는 안정성 강화 조치입니다. 루프 횟수 외에도 단일 노드가 응답 지연으로 시스템 스레드를 영구 점유하는 것을 막기 위해 최대 실행 시간 초과 시 강제 Pause 되도록 합니다.
  - 구현 경계: 엔진 런타임 내 타이머 데몬 적용 혹은 비동기 태스크 타임아웃 처리.

## 2. MVP scope / out-of-scope

- **MVP Scope**
  - 폼 프리셋 텍스트의 정규식 처리 최적화 및 테스트 코드 작성.
  - 시각적 워크플로우 캔버스(ReactFlow 기반)의 단절 그래프 및 다중 진입점 클라이언트 측 사전 차단 로직 도입.
  - 워크플로우 엔진의 무한 반복 횟수 초과 및 타임아웃 감지에 따른 Pause 상태 전환 제어.
  - API 백엔드 접근 시 뷰어 토큰 유효성 필수 검증.
  - 공통 토스트 UI 모듈화를 통한 일관된 에러 메시지 렌더링.
- **Out-of-scope**
  - 전체 사용자/그룹 권한 관리(RBAC) 등 거대 보안 체계 (현재는 단일 뷰어 토큰 기반 강화로 제한).
  - 캔버스 내 노드 드래깅 성능 및 Z-index 최적화 등 코어 렌더링 라이브러리 자체의 퍼포먼스 튜닝.
  - Temporal, LangGraph 등 외부 오케스트레이션 엔진으로의 즉각적인 마이그레이션 (자체 Python 엔진 안정화 집중).

## 3. Completion criteria

- `web/src/App.tsx` 최적화 후 폼 프리셋 기능을 반복 클릭하더라도 후행 공백 및 개행이 누적되지 않아야 하며, `App.test.tsx`가 성공적으로 통과해야 합니다.
- 워크플로우 빌더 UI에서 비정상 그래프 구성 시 서버 저장 요청이 발생하지 않고, 즉시 공통 모듈화된 에러 토스트 피드백이 노출되며 해당 E2E 테스트가 통과해야 합니다.
- 3000번대 로컬 포트 직접 호출을 통한 백엔드 라우팅 우회 접근 시 API에서 `401` 또는 `403` 에러가 명확하게 반환되어야 합니다.
- 엔진 런타임 내에서 임의의 노드가 지정된 반복 카운트 임계치 혹은 타임아웃 임계치를 초과할 시 즉시 해당 노드의 상태가 Pause 처리되는지 확인해야 합니다.
- `test_workflow_engine.py` 실행 시 단절 노드가 포함된 요청은 `400` 또는 `422` 코드를 응답해야 합니다.

## 4. Risks and test strategy

- **Risk: 엔진 제약 강화가 정상적인 롱-러닝(Long-running) 태스크를 중단시킬 위험성**
  - **Strategy:** 최대 반복 횟수(max_loops)와 타임아웃 시간(timeout)의 기본값을 시스템 메타데이터나 환경변수를 통해 유연하게 주입할 수 있도록 설계합니다. 테스트 단계에서는 Mocking을 이용해 극단적인 작은 값을 설정하여 올바르게 중단 로직이 작동하는지 먼저 검증합니다.
- **Risk: 폼 문자열 정규식 최적화에 따른 타 언어 및 특수 기호 파싱 오류**
  - **Strategy:** 영문, 국문, 멀티라인 형태 등 다양한 엣지 케이스를 포함한 파라미터화된 단위 테스트(Parameterized test)를 `web/src/App.test.tsx`에 작성하여, 정규식이 불필요한 공백만 제거하고 핵심 내용은 훼손하지 않음을 보장합니다.

## 5. Design intent and style direction

- **기획 의도:** 사용자가 아이디어를 던지면 24시간 자율적으로 동작하지만, 폭주(무한 루프)나 오류를 시스템이 먼저 감지하여 안전하게 통제 가능한 "신뢰할 수 있는 자동 개발자 플랫폼"을 제공하는 데 있습니다. 사용자는 시스템을 안심하고 제어할 수 있어야 합니다.
- **디자인 풍:** 전문적이고 신뢰감을 주는 모던 대시보드형 및 시각적 노드 캔버스.
- **시각 원칙:**
  - 컬러: 안정감을 주는 Neutral 톤(회색/흰색) 베이스를 유지하되, 오류나 무한 루프 경고 시 확실한 인지를 위한 Error Color(Red/Orange)를 토스트 피드백에 명확하게 사용합니다.
  - 패딩/마진: 조작 실수를 줄이기 위해 각 컴포넌트(입력 폼, 캔버스 노드 등) 간 넉넉한 여백(Padding)을 확보하여 시각적 분리감을 제공합니다.
  - 타이포그래피: 가독성을 우선시하는 산세리프 폰트를 사용하며, 에러 및 경고 메시지는 굵게 처리하여 한눈에 파악할 수 있도록 합니다.
- **반응형 원칙:** 복잡한 워크플로우 제어는 데스크톱 관리를 기본(Desktop-first)으로 합니다. 단, 모바일 기기를 통해서도 실행 중인 태스크 모니터링과 긴급 중단(Pause) 버튼을 누를 수 있도록, 핵심 카드 컴포넌트 위주로 부드럽게 재배치되는 모바일 우선 반응형 규칙을 보조적으로 혼용합니다.

## 6. Technology ruleset

- **플랫폼 분류:** web / api
- **프론트엔드 (web):** React 라이브러리 기반으로 UI(ReactFlow, Vite) 구축. 테스트는 Jest(Unit) 및 Playwright(E2E)를 사용합니다.
- **백엔드 (api):** FastAPI 기반으로 워크플로우 엔진과 라우팅을 구성하며, 테스트는 Pytest를 사용합니다.
- **실행 가이드:** 개발 환경 및 리뷰를 위한 로컬 구동 시 포트 충돌을 피하기 위해 프론트엔드/웹은 3000번, 백엔드는 3100번 등 오직 3000번대 포트만을 사용하도록 설정합니다.
