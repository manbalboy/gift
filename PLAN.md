# PLAN

## 1. Task breakdown with priority

### P0 (Critical - System Stability & Core UI)
- **API 동시성 제어 (`api/app/api/workflows.py`)**: `/resume` 엔드포인트에 동시성 제어 락을 적용합니다. 중복 요청 발생 시 `409 Conflict` 또는 `200 OK`를 상황에 맞게 반환하도록 처리하여 상태 경합을 방지합니다.
- **예외 처리 및 상태 커밋 (`api/app/services/workflow_engine.py`)**: 워크플로우 엔진 구동 중 발생하는 파일 I/O 및 시스템 예외 처리 블록을 추가합니다. 백그라운드 프로세스의 크래시를 방지하고, 에러 발생 시 DB의 `Run` 및 `NodeRun` 상태가 `failed`로 안전하게 커밋되도록 보장합니다.
- **실패 상태 UI 및 재시도 기능 (`web/src/` ReactFlow 컴포넌트)**: 실패(`failed`) 상태로 전이된 노드에 대해 명시적인 붉은색 시각적 렌더링을 적용하고, 대시보드 상에서 즉각적인 복구가 가능하도록 "Retry Node" 액션 버튼 UI를 구현합니다.

### P1 (High - Security & Robustness)
- **환경 변수 파싱 안정화 (`api/app/core/config.py`)**: `DEVFLOW_LOCALHOST_SPOOF_GUARD_PORTS` 등 설정 파싱 시 빈 문자열, 특수문자 주입 같은 엣지 케이스를 방어하고, 파싱 실패 시 안전한 기본값(Fallback)을 반환하는 로직을 적용합니다.
- **OS Lock 방어 및 로깅 (`api/app/services/workspace.py`)**: 워크스페이스 내 파일 읽기/쓰기 등 핸들링 시 OS Lock 획득 경합에 대비한 방어 로직을 적용하고, 보안 감사에 용이하도록 예외 발생 시 구조화된 에러 로깅(`logging.error`)을 추가합니다.

### P2 (Medium - Testing & Adjacent Features)
- **테스트 커버리지 보강 (`api/tests/`, `web/tests/`)**:
  - `config.py`의 설정 파싱 방어 코드가 정상 작동하는지 확인하는 Unit Test 작성.
  - 포트 `3100` 환경에서 50개의 비동기 동시 요청을 인가하는 `/resume` Integration Test 작성 및 크래시 방어 검증.
  - 고의적인 파일 권한 예외 유발 시 프로세스가 죽지 않고 `failed` 상태가 정상 기록되는지 확인하는 E2E/Integration Test 추가.
  - 로컬 실행/프리뷰용 포트를 `3000`번대(예: 3100~3199)에서 동적으로 충돌 없이 할당받도록 테스트 및 구동 스크립트 수정.
- **고도화 기능 (인접 기능 추가)**:
  - **에러 로그 스니펫 툴팁 표시**: "Retry Node" 버튼 주변이나 노드 툴팁에 최근 에러 로그 40줄 스니펫을 간략히 노출합니다. (근거: 재시도를 수행하기 전 대시보드 내에서 즉각적으로 실패 원인을 파악할 수 있도록 돕는 필수 인접 기능. 구현 경계: 기존 DB에 적재된 로그 데이터를 UI 컴포넌트로만 매핑하여 표시)
  - **잔여 재시도 횟수 시각화**: ReactFlow 노드에 남은 자동/수동 재시도 기회(예: `Attempt 2/3`)를 표시합니다. (근거: 무한 재시도 클릭을 방지하고 시스템 상태 가시성을 높임. 구현 경계: 기존 노드 메타데이터의 attempt 속성을 노드 UI에 추가 렌더링)

## 2. MVP scope / out-of-scope

**Scope (포함 대상):**
- `/resume` 엔드포인트의 락 메커니즘을 통한 동시성 문제 해결 및 멱등성 확보.
- 워크플로우 엔진과 워크스페이스 파일 핸들링 간 발생하는 I/O 예외를 캐치하여 시스템 다운을 막고 상태를 안전하게 DB에 보존.
- ReactFlow 기반 대시보드에서 실패 노드의 시각적 렌더링(붉은색) 및 "Retry Node" 액션 UI 처리.
- 환경 변수 파싱 예외 처리 및 OS Lock 방어 로직.
- 로컬 테스트 및 프리뷰 환경에서 3000번대 포트 동적 할당 스크립트 구축.
- 위 명시된 버그 및 취약점 수정을 증명하는 단위/통합/E2E 테스트 코드 작성.

**Out-of-scope (제외 대상):**
- 워크플로우 엔진의 아키텍처 전면 개편 (예: Temporal, LangGraph 등 외부 내구 실행 런타임 도입은 향후 과제로 유지).
- 대규모 RBAC(역할 기반 접근 제어) 또는 복잡한 사용자 관리 시스템 신규 구축.
- 기존 요구사항에 명시되지 않은 완전히 새로운 타입의 AI 에이전트 실행 노드 추가.

## 3. Completion criteria
- 다수의 클라이언트가 동시에 `/resume` API 호출 시 크래시나 데드락 없이 `409 Conflict` 또는 `200 OK`를 상황에 맞게 안정적으로 반환해야 합니다.
- 파일 I/O 에러가 발생하더라도 백그라운드 프로세스가 종료되지 않고, DB의 해당 NodeRun과 Run 상태가 `failed`로 온전히 기록되어야 합니다.
- 비정상적인 환경 변수 주입 시에도 애플리케이션이 안전한 기본값으로 정상 기동해야 합니다.
- 대시보드의 ReactFlow 캔버스에서 실패한 노드가 붉은색으로 명확하게 표시되며, 에러 로그 스니펫 확인 및 "Retry Node" 버튼 클릭을 통해 재시도가 정상 트리거되어야 합니다.
- 모든 단위, 통합, E2E 테스트가 3000번대 로컬 포트 환경에서 충돌 없이 성공적으로 통과해야 합니다.

## 4. Risks and test strategy
- **Risk 1 (동시성 데드락 및 경합)**: `/resume` API에 동시성 락을 부적절하게 적용할 경우 무한 대기나 트랜잭션 데드락이 발생할 위험이 있습니다.
  - **Strategy**: 3100번 포트 테스트 환경에서 50개 이상의 비동기 요청을 동시에 발송하는 부하 테스트 스크립트를 Integration Test로 작성하여 동시성 제어의 안정성을 검증합니다.
- **Risk 2 (데이터 상태 불일치)**: 파일 I/O 및 OS Lock 권한 예외 발생 시 처리가 누락되면, 프로세스는 실패했으나 DB에는 상태가 반영되지 않는 불일치가 발생합니다.
  - **Strategy**: 시스템 권한 예외를 고의로 모사(Mocking)하여, 예외 상황에서도 워크플로우 엔진이 오류를 캐치하고 `failed` 상태로 올바르게 롤백/업데이트하는지 검증하는 테스트를 작성합니다.
- **Risk 3 (포트 충돌로 인한 CI 실패)**: 로컬 혹은 CI 환경에서 테스트 병렬 실행 시 포트 점유 문제가 발생할 수 있습니다.
  - **Strategy**: 테스트 구동 시 3000번대 대역(예: 3100~3199) 내에서 가용한 포트를 동적으로 탐색하고 바인딩하는 스크립트를 적용하여 격리성을 확보합니다.

## 5. Design intent and style direction
- **기획 의도**: 워크플로우 실행 중 불가피하게 발생하는 시스템 오류와 동시성 경합 상태를 사용자에게 투명하게 보여주고, 클릭 한 번으로 안전하게 상황을 복구할 수 있는 '신뢰성 높은 자동 개발자 플랫폼' 경험을 제공합니다.
- **디자인 풍**: 대시보드형의 모던하고 기술 지향적인 노드 베이스 스타일로, 군더더기 없이 실행 상태와 정보 밀도에 집중합니다.
- **시각 원칙**:
  - **컬러**: 실패(Error/Failed) 상태는 시각적으로 즉시 인지할 수 있는 강렬한 붉은색(Red 계열)을 사용하며, 배경과 명도 대비를 뚜렷하게 주어 접근성을 높입니다.
  - **패딩/마진**: 시스템 텍스트의 정보 밀도는 높이되, "Retry Node" 버튼과 같은 인터랙션 영역은 충분한 여백(Margin/Padding)을 두어 사용자의 오클릭을 방지합니다.
  - **타이포**: 로그 스니펫과 코드 경로 등은 가독성을 위해 모노스페이스(Monospace) 폰트를 적용하고, 일반적인 대시보드 UI 텍스트는 깔끔한 산세리프(Sans-serif)를 사용합니다.
- **반응형 원칙**: 모바일 우선(Mobile-first) 규칙을 따르되, 워크플로우 노드 특성상 데스크탑 등 넓은 화면에서 캔버스를 100% 활용할 수 있도록 반응형 레이아웃을 유연하게 구성합니다.

## 6. Technology ruleset
- **플랫폼 분류**: web, api
- **web**: React 프레임워크 기반(Vite 빌드 환경)으로 계획하며, 시각적 다이어그램은 ReactFlow 라이브러리를 활용하여 캔버스와 커스텀 노드를 구현합니다.
- **api**: FastAPI 기반의 비동기 백엔드 구조로 계획하여, Workflow Engine, Webhook 수신, 락(Lock) 기반의 동시성 제어를 구현합니다.
