# PLAN

## 1. Task breakdown with priority

- **[P0] `item.message` 타입 예외 처리 보완**
  - 대상 파일: `web/src/components/Toast.tsx`
  - 내용: `message` 프로퍼티가 `number` 등 `string` 외의 타입으로 전달될 경우 빈 문자열로 렌더링되는 버그 수정. `String(item.message || '')` 등으로 안전한 타입 캐스팅을 적용.
- **[P0] 제스처 취소 시 복귀 애니메이션(Transition) 추가**
  - 대상 파일: `web/src/components/Toast.tsx`, `web/src/styles/app.css`
  - 내용: 모바일 기기에서 스와이프를 시도하다 임계값(`SWIPE_DISMISS_THRESHOLD`)을 넘지 못해 취소될 때, 알림이 원위치로 뚝 끊기듯 돌아가는 현상 수정. 부드러운 복귀를 위한 CSS `transition` (예: `transform 0.3s ease-out`) 클래스를 제스처 종료 시점에 부여.
- **[P1] 타이머 일시 정지(Pause on hover/focus) 기능 도입**
  - 대상 파일: `web/src/components/Toast.tsx`
  - 내용: 사용자가 긴 텍스트를 읽기 위해 알림 카드에 마우스를 올리거나(Hover) 키보드 포커스(Focus)를 주었을 때 자동 만료 타이머(`durationMs`)를 일시 정지하고, 벗어나면 재개되도록 React 상태 및 생명주기 제어 로직 추가.
- **[P1] 키보드 접근성 대응 및 테스트**
  - 대상 파일: `web/src/components/Toast.test.tsx`, `web/tests/e2e/toast-layering.spec.ts`
  - 내용: Tab 키를 이용해 알림 내 버튼(펼치기/접기, 닫기)에 접근 및 상호작용할 수 있는지 단위/E2E 테스트 케이스 추가 및 구조 보완.
- **[P2] 전역 알림 접근성 가이드(Aria-live) 보완 (고도화 플랜)**
  - 대상 파일: `web/src/components/Toast.tsx`
  - 내용: 스크린 리더 사용자가 알림 발생을 즉시 인지할 수 있도록, Toast 컨테이너에 `aria-live="polite"` 또는 `assertive` 속성을 일관되게 적용하여 포괄적인 접근성(A11y) 경험을 고도화. 기존의 시각적 접근성(호버/포커스 타이머 정지) 및 키보드 접근성(Tab 이동) 요구사항과 자연스럽게 연결되는 인접 기능.

## 2. MVP scope / out-of-scope

**MVP Scope**
- 숫자 등 비문자열 타입의 알림 메시지를 누락 없이 정상적으로 텍스트로 렌더링.
- 스와이프 취소 시 제자리로 복귀하는 부드러운 CSS 트랜지션 애니메이션 지원.
- 마우스 호버 및 키보드 포커스 상태에서 알림 자동 닫힘 타이머를 일시 정지(Pause).
- 키보드(Tab)를 통한 알림 카드 내부 액션 버튼 포커스 이동 및 조작 보장.
- 스크린 리더를 위한 `aria-live` 속성 보강으로 시각/비시각적 접근성 동시 개선.

**Out-of-scope**
- Toast 알림 디자인(레이아웃, 컬러셋 등)의 전면 개편.
- 커스텀 애니메이션 라이브러리(Framer Motion 등)의 신규 도입 (기존 바닐라 CSS 트랜지션 및 React 상태만 활용).
- REVIEW에 언급되지 않은 대시보드 및 Workflow Builder 등 타 영역 컴포넌트의 기능 수정.

## 3. Completion criteria

- `Toast` 컴포넌트 호출 시 숫자 `123` 등 다양한 타입의 값을 `message`로 전달했을 때 UI 화면에 텍스트가 정확히 출력되어야 함.
- 뷰포트 환경에서 알림을 살짝 스와이프했다가 놓았을 때, 딱딱하게 끊기지 않고 부드럽게 원래 위치로 돌아가야 함.
- 마우스를 알림 위에 올려두거나 Tab 키로 포커스를 맞추었을 때, 지정된 시간이 지나도 알림이 사라지지 않고 화면에 유지되어야 함.
- Jest 기반 단위 테스트 및 Playwright 기반 E2E 테스트 통과. 특히 Hover/Focus에 의한 타이머 일시 정지와 Tab 키 포커스 진입/조작에 대한 테스트 검증 코드가 반영되어야 함.
- `npm run test` 및 `npm run test:e2e` 등 자동화 테스트 실행 시 에러 없이 통과해야 하며, 로컬 실행 포트는 `3100` 등 3000번대 범위를 사용해 충돌 없이 구동되어야 함.

## 4. Risks and test strategy

**Risks**
- **타이머 제어 복잡도 증가:** React 생명주기 내에서 `setTimeout`과 Hover/Focus 이벤트를 복합적으로 다루면서 타이머가 제때 해제되지 않아 발생하는 메모리 누수나 무한 잔류 버그 위험.
- **제스처 오작동:** 복귀 애니메이션(`transition`)이 스와이프를 진행 중인 상태에도 잘못 적용되어, 터치 이동을 버벅거리게 만들고 사용자 제스처 추적을 방해할 위험.

**Test Strategy**
- **타이머 테스트(Unit):** `jest.useFakeTimers()`를 활용하여 Hover 상태 시뮬레이션(MouseEnter/MouseLeave) 시 타이머가 정상적으로 멈추고 재개되는 전체 생명주기 흐름을 단위 테스트로 정밀 검증.
- **애니메이션 테스트(E2E):** Playwright 터치/마우스 이벤트를 시뮬레이션하여 임계값 도달 전 드래그를 해제했을 때 `transform` 상태의 복원과 `transition` 관련 클래스가 올바르게 부여되는지 DOM 상태 검증.
- **접근성 테스트(E2E):** 키보드 `Tab` 입력을 시뮬레이션하여 닫기 버튼으로 포커스가 진입하는지 확인하고, `Enter` 키 입력 시 알림이 닫히는지 자동화 흐름으로 증명.

## 5. Design intent and style direction

- **기획 의도:** 사용자가 어떤 기기(모바일/데스크톱)나 입력 방식(마우스/키보드/터치/스크린 리더)을 사용하든, 워크플로우 실행 상태와 결과를 놓치지 않고 명확하게 인지하며 편안하게 읽을 수 있는 안전한 정보 전달 경험을 제공.
- **디자인 풍:** DevFlow 대시보드 환경에 조화롭게 녹아드는 **모던한 카드형** 알림 스타일. 불필요한 장식을 덜어내고 정보의 위계와 닫기/펼치기 액션 버튼에만 시선이 집중되는 깔끔한 대시보드 UI를 지향.
- **시각 원칙:** 
  - 컬러/타이포: 프로젝트 내 기존 `tokens.css`의 시스템 디자인 변수 체계를 따르며, 배경과 텍스트 간 명도 대비를 유지해 가독성을 최대한 확보.
  - 패딩/마진: 주변 UI와 간섭하지 않도록 여백 규칙을 일관성 있게 준수.
  - 모션: 제스처 피드백과 상태 변화 시에만 부드러운 최소한의 트랜지션(0.2s~0.3s)을 제공해, 조작감이 끊기지 않고 자연스럽게 이어지도록 처리.
- **반응형 원칙:** **모바일 우선(Mobile-first)** 규칙 적용. 좁은 모바일 화면에서의 터치/스와이프 인터랙션을 우선 설계하되, 리사이즈로 데스크톱 뷰포트로 전환될 시 마우스 기반 호버(Hover) 경험까지 매끄럽고 완벽하게 호환되도록 구성.

## 6. Technology ruleset

- **플랫폼 분류:** web
- **프레임워크/라이브러리:** React 기반 (TypeScript, Vite 빌드 환경)
- **테스트 도구:** Jest (Unit Test), Playwright (E2E Test)
- **스타일링:** 바닐라 CSS (`app.css`, `tokens.css` 활용)
- **실행 및 포트 규칙:** 실행 가이드 및 웹 서버 포트 구성이 필요한 경우 `3000`번대 포트(예: 3000, 3100)만 사용토록 설정.
