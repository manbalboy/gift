# REVIEW

## Functional bugs
- **Pause/Resume API 및 UI 누락**: 백엔드 엔진(`api/app/services/workflow_engine.py`)에는 노드 실행 타임아웃이나 반복 횟수 초과 시 워크플로우를 강제로 `paused` 상태로 전환하는 시스템 보호 로직이 반영되어 있습니다. 그러나 PLAN.md에 명시된 핵심 제어 기능인 재개(Resume) 기능이 누락되어 있습니다. `api/app/api/workflows.py`에 워크플로우 재개를 위한 `POST /runs/{run_id}/resume` 엔드포인트가 없으며, 프론트엔드 대시보드(`web/src/App.tsx`)에도 일시정지 상태를 확인하고 재개하는 제어 버튼이 없습니다. 따라서 한 번 중단된 워크플로우는 사용자가 개입하여 재개할 수 없는 치명적인 기능적 결함이 존재합니다.

## Security concerns
- **뷰어 토큰(Viewer Token) 미들웨어 Fail-open 취약점**: `api/app/api/dependencies.py`의 `require_viewer_token` 함수는 환경 변수 `viewer_token`이 비어있거나 설정되지 않은 경우(`if not configured: return`) 인증 절차를 그대로 통과시키도록 구현되어 있습니다. 시스템 관리자의 설정 누락이나 환경 변수 주입 실패 시 플랫폼 전체가 외부에 인증 없이 노출되는 심각한 보안 위험이 있으므로, 기본적으로 접근을 차단(Fail-closed)하도록 정책을 수정해야 합니다.
- **로컬 접근 포트 우회 차단 미흡**: PLAN.md에서 요구한 "로컬 테스트 시 `3100`번 포트 등을 통한 우회 접근 방어" 관련 로직이 명시적으로 존재하지 않습니다. Host 헤더나 IP 스푸핑을 악용하여 로컬 내부(`3100`번 대역 포트 등) 망에서 접근하는 것처럼 위장하는 요청을 필터링하고 차단하는 추가 검증 로직이 미들웨어에 도입되어야 합니다.

## Missing tests / weak test coverage
- **Resume API 테스트 부재**: 워크플로우의 재개(Resume) 기능이 구현되지 않았으므로, 당연히 이에 대한 백엔드 단위 테스트 및 프론트엔드 E2E 테스트가 완전히 누락되어 있습니다. 강제 Pause 상태로 전이된 런타임이 Resume 요청을 통해 정상적으로 복구되고 남은 노드를 실행하는지 검증하는 테스트 작성이 시급합니다.
- **인증 보안 예외 케이스 검증 부족**: 잘못된 뷰어 토큰 주입 시 차단되는 테스트는 일부 존재할 수 있으나, 설정이 누락되었을 때 전체 시스템이 노출되지 않고 `500` 또는 `401` 에러로 확실히 차단되는지 점검하는 테스트 케이스 및 포트 `3100` 우회 공격 상황을 묘사한 보안 테스트가 부족합니다.

## Edge cases
- **정상적인 장기 실행 노드의 강제 중지**: 현재 워크플로우 엔진은 전역 설정 타임아웃에 의존하여 장기 실행 노드를 Pause 처리합니다. 복잡한 빌드 스크립트나 다단계 E2E 테스트 노드의 경우 정상적으로 오랜 시간이 소요될 수 있으나, 이 전역 임계치를 초과할 시 시스템이 예고 없이 작업을 중단시키는 억울한 엣지 케이스가 존재합니다.
- **Pause 상태 진입 시 스트림 통신**: 실행 중인 워크플로우가 Pause 상태로 급격히 전환되었을 때, 이를 수신하는 SSE(Server-Sent Events) 스트림이 불필요한 재연결을 시도하거나 상태 업데이트를 적절히 반영하지 못하는 네트워크 엣지 케이스에 대한 고려가 부족할 수 있습니다.
