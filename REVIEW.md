# REVIEW

이 문서는 `SPEC.md`와 `PLAN.md`를 바탕으로 현재 저장소의 상태를 리뷰하고 기능적 문제, 보안, 테스트 커버리지 및 엣지 케이스에 대한 개선 사항을 정리한 내용입니다.

## Functional bugs

- **UI 알림 컴포넌트(Toast) 오류 방어 부재**: `web/src/components/Toast.tsx`에서 `durationMs` 파라미터에 음수 값이 전달될 경우를 대비한 기본값(Fallback) 처리 로직이 누락되어 타이머가 비정상적으로 동작할 수 있습니다.
- **긴 텍스트 레이아웃 붕괴**: 대량의 로그나 긴 에러 스택이 Toast 알림 등에 출력될 때 UI 레이아웃이 화면 밖으로 밀려 깨지는 현상이 발생합니다. 텍스트 이탈을 막기 위한 `word-break: break-all` CSS 속성 처리가 필요합니다.
- **알림 폭주 제어 부재**: 노드 실행 실패 등 이벤트가 폭주할 경우 수많은 알림 팝업이 겹쳐서 노출되어 화면을 완전히 가리게 됩니다. 이를 해결하기 위해 Toast 알림 큐잉(Queueing) 스케줄러를 도입하고 동시 노출 개수를 제한(예: 3개)해야 합니다.
- **실행 엔진의 구조적 결함 (API 의존성)**: 워크플로우 실행이 독립적인 백그라운드 워커에서 작동하지 않고 조회 API(`GET /runs/...`)가 호출될 때만 한 스텝씩 진행되는 심각한 논리적 버그가 있습니다. UI 클라이언트가 SSE 이벤트를 받고 실행을 "밀어주는" 트리거 역할을 하고 있으므로 백그라운드 실행 엔진으로 전환해야 합니다.
- **그래프 Edges 논리 무시**: 로직 상 그래프의 엣지(연결선)가 무시되고 노드 배열의 순서(sequence)에 의존하여 다음 단계가 실행되고 있습니다. 이는 노드 기반 워크플로우의 본래 목적과 부합하지 않으므로 실제 연결망(edges) 기반으로 전이되도록 픽스해야 합니다.

## Security concerns

- **Webhook 무결성 및 인가 검증 미흡**: 외부 연동을 위한 Webhook 수신 엔드포인트(`api/app/api/webhooks.py`)에 IP Spoofing(위조) 차단을 위한 IP 대역 검증 로직과 요청 헤더 내 HMAC 암호화 서명 검증 로직이 부족하여 악의적인 외부 공격에 노출될 수 있습니다.
- **Human Gate 권한 우회 가능성**: QA, UX 검수 등을 위한 휴먼 게이트 노드에서 승인(Approve) 처리 API 호출 시, 요청 작업자에 대한 강력한 권한 검증 및 워커 실행 컨텍스트의 정합성 체크 로직이 필요합니다. 인가되지 않은 사용자가 강제 승인 API를 호출하는 것을 막아야 합니다.

## Missing tests / weak test coverage

- **Webhook 방어 단위 테스트 누락**: 잘못된 HMAC 서명을 포함하거나 인가되지 않은 IP에서 접근하는 위조 시나리오를 모킹(Mocking)하여 올바르게 401 또는 403 HTTP 에러를 반환하는 방어망 단위 테스트가 존재하지 않습니다.
- **Human Gate 상태 지속성 통합 테스트 부재**: 특정 노드에서 `approval_pending` 상태로 오랜 시간 대기한 이후에 사용자가 승인을 내렸을 때, 워크플로우 엔진이 메모리 상의 상태를 잃지 않고 DB에서 안정적으로 컨텍스트를 불러와 다음 노드로 이행(Resume)하는지 점검하는 통합 테스트가 부족합니다.
- **SSE 스트림 메모리 누수 검증 스크립트 부재**: 여러 클라이언트가 지속적으로 SSE 연결을 맺고 강제 종료하는 상황을 모사하여 `active_stream_connections`가 올바르게 해제되고 메모리 누수가 발생하지 않음을 증명하는 로컬 부하 테스트 스크립트가 추가되어야 합니다.
- **Visual Workflow Builder E2E 테스트 보강 필요**: 로컬 프리뷰 환경에서 캔버스 노드 조작, 순환 연결 에러 팝업 시나리오, 저장 후 시뮬레이션(드라이런) 성공 여부를 포괄적으로 검증하는 E2E 테스트 스크립트가 없습니다. 테스트 실행은 포트 충돌을 피하기 위해 `http://localhost:3100` 환경을 바라보고 진행되는 Playwright 기반 E2E가 구성되어야 합니다.

## Edge cases

- **대용량 아티팩트 렌더링에 의한 브라우저 크래시**: 수십 MB 크기에 달하는 시스템 로그나 다수의 E2E 캡처 스크린샷 아티팩트를 프론트엔드 UI 뷰어에서 한 번에 렌더링하려고 시도할 경우, 브라우저 탭의 메모리가 초과되어 다운될 수 있습니다. 렌더링 시 Chunk loading 또는 Virtualization(가상화) 기법의 엣지 케이스 처리가 도입되어야 합니다.
- **실행 취소(Cancel) 시 리소스 반환 실패**: 실행 큐에 대기 중이거나 Human Gate 대기 중인 상태에서 사용자가 강제로 워크플로우 실행을 취소(Cancel)할 경우, Graceful Shutdown 처리가 이뤄지지 않으면 로컬 스레드 및 활성화된 SSE 채널 등 서버 리소스가 원자적으로 반환되지 않고 낭비될 위험이 있습니다.
- **에디터와 백엔드 간 상태 일시 불일치 방어**: 프론트엔드의 ReactFlow 캔버스 상태와 백엔드의 DAG 유효성 규격 매핑이 일시적으로 어긋난 페이로드가 전송되었을 때, 이를 백엔드 API 단에서 안정적으로 캐치하여 422 상태 코드 반환을 통해 DB 오염을 방지하는 엣지 케이스 로직 점검이 필요합니다.
