# REVIEW

## Functional bugs
- 현재 P0(버그 수정)부터 P2(UX 개선)까지의 주요 기능인 Cursor 기반 페이징, ReDoS 방어 문자열 제한, UI 레이아웃 붕괴 방지, Clear All 및 Export Logs 버튼 기능 등은 모두 올바르게 구현되었으며 기능적 버그는 발견되지 않았습니다.
- 하지만 P3 고도화 플랜에 명시된 **알림 필터링(Client-side)** 및 **자동 스크롤 중지 및 재개(Auto-scroll Pause)** 기능이 `SystemAlertWidget.tsx`에 아직 구현되어 있지 않습니다. 이로 인해 실시간으로 대량의 로그가 수신될 때 사용자가 과거 로그를 확인하려 하면 스크롤이 강제로 최하단으로 튕기는 사용성 버그(UX 결함)가 여전히 발생합니다.

## Security concerns
- `system_alerts.py` 내 `_sanitize_string` 함수에 10,000자 길이 제한(Truncate)이 적용되어 정규표현식 연산 전 악의적으로 긴 텍스트 페이로드가 유발할 수 있는 ReDoS(서비스 거부) 공격이 안전하게 차단되었습니다.
- 민감 정보(Bearer 토큰 및 경로) 마스킹 처리도 문제없이 동작하고 있으며, 현재 저장소에서 추가적인 조치가 필요한 크리티컬한 보안 취약점은 관측되지 않습니다.

## Missing tests / weak test coverage
- 단위 테스트(pytest)와 E2E 테스트(Playwright) 모두 성공적으로 통과했으며, P1 플랜에 요구된 모바일(320px) 레이아웃 검증, 무한 루프 한계 초과 방어(Budget 초과), 그리고 동시 락 경합 해제 검증 스크립트(`test-port-timeout.sh`) 커버리지가 우수합니다.
- 단, P3 기능(알림 필터링, 스크롤 제어)이 아직 미구현 상태이므로, 향후 해당 기능이 구현될 때 관련 프론트엔드 컴포넌트 단위 테스트 및 스크롤 인터랙션 E2E 테스트가 추가로 요구됩니다.

## Edge cases
- 데이터베이스에 동일한 밀리초(ms) 단위의 시스템 알림 로그가 대량으로 동시 적재될 때, Cursor 기반 페이징 로직이 데이터 누락 없이 리스트를 반환하는지 완벽하게 검증되었습니다.
- P3 필터링 컴포넌트(Chip)가 추후 UI에 추가될 때, 모바일 320px 뷰포트에서는 가로 공간이 협소하여 기존의 버튼 영역("CLR", "EXP")과 간섭이 생겨 레이아웃 붕괴가 다시 발생할 수 있는 잠재적인 엣지 케이스가 존재합니다.
- 포트 3100번대(프론트엔드 3100, API 3101)를 다중 점유하는 테스트 상황에서도 락 시스템이 정상적으로 포트를 순차 반환하고 데드락에 빠지지 않도록 엣지 케이스 처리가 완료되었습니다.
