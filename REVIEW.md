# REVIEW

## Functional bugs
- 현재 `api` 모듈의 테스트 슈트는 122개 모두 정상 통과하고 있어 백엔드 API의 치명적인 기능적 결함은 보이지 않습니다.
- 하지만 프론트엔드의 커스텀 마크다운 렌더러(`web/src/utils/sanitize.ts`)는 기초적인 정규식(`Regex`)에 의존하고 있어 중첩된 리스트, 인용구, 복잡한 코드 블록 혼합 시 렌더링이 깨지거나 일부 마크다운 문법이 무시되는 기능적 한계가 존재합니다.
- 대시보드나 클라이언트 로컬 실행 포트(예: `http://localhost:3100` 환경)에서 SSE 연결 시, Nginx 설정 외에 프론트엔드 재연결 로직에 대한 견고함이 아직 코드상 명확히 드러나지 않아 일시적 네트워크 장애 시 로그 스트리밍이 누락될 수 있습니다.

## Security concerns
- **XSS 방어 및 ReDoS 위험:** `web/src/utils/sanitize.ts`에 `DOMPurify`를 적용하여 기본적인 악성 스크립트 실행은 막고 있으나, 커스텀 정규식을 통한 마크다운 변환 로직은 복잡한 문자열 입력 시 정규식 서비스 거부 공격(ReDoS)에 노출될 수 있습니다.
- 안전성을 위해 `marked` 등 검증된 마크다운 라이브러리를 거친 뒤 최종 출력물만 `DOMPurify`로 처리하는 것이 안전합니다.
- **Race Condition 방어 검증 부족:** `workflow_engine.py` 등 백엔드에서 `with_for_update()`를 통해 트랜잭션 락을 시도하고 있지만, 이에 대한 동시성 제어 모의 테스트가 없으므로 다수의 요청이 동시다발적으로 유입될 때 락 대기 시간 초과나 우회로 인한 데이터 정합성 파괴 위험이 남아 있습니다.

## Missing tests / weak test coverage
- **휴먼 게이트 다중 동시 요청 테스트 누락:** `PLAN.md`의 완료 기준에 명시된 "휴먼 게이트 승인 엔드포인트에 10개 이상의 다중 동시 요청을 보냈을 때 1건만 정상 처리되는가"를 확인하는 비동기 병렬 호출(`asyncio.gather` 등) 기반의 테스트가 `api/tests/test_human_gate.py`에 구현되어 있지 않습니다.
- **XSS 살균 및 렌더링 단위 테스트 누락:** `web/src/utils/sanitize.ts`를 검증하는 단위 테스트 또는 악성 페이로드 주입 시나리오가 없습니다. `DOMPurify`가 의도대로 작동하는지 검증하는 자동화 뷰 테스트가 누락되어 있습니다.
- **SSE Heartbeat 및 스트리밍 지연 테스트 부족:** 15초 이상의 지연 상황에서도 Nginx `proxy_read_timeout`과 백엔드 연결이 끊기지 않는지를 확인하는 네트워크 단절 모의 통합 테스트가 존재하지 않습니다.

## Edge cases
- 휴먼 게이트 승인 시, 서로 연관된 노드들이 동시에 상태 전환을 시도할 때 백엔드 DB의 `with_for_update()` 락 획득 순서가 꼬이면 데드락(Deadlock)에 빠질 수 있습니다.
- 아티팩트의 크기가 비정상적으로 큰 파일(수십 MB의 텍스트 로그)일 경우, 프론트엔드 마크다운 렌더러가 렌더링을 시도하다 브라우저 메모리 초과로 탭이 멈출(Crash) 가능성이 있습니다. 페이징이나 라인별 가상화(Virtualization) 처리를 고려해야 합니다.
- 스케줄러가 타임아웃된 휴먼 게이트 노드를 스캔할 때, 분산 환경(다중 워커)에서 중복 스캔을 막지 않으면 동일 노드에 대해 타임아웃 처리가 여러 번 트리거될 수 있는 엣지 케이스가 존재합니다.
