# REVIEW

## Functional bugs
- 스크롤 최하단 판별 시 브라우저의 줌(Zoom) 비율이나 OS 디스플레이 배율 설정에 의해 소수점 픽셀 단위의 오차가 발생할 수 있습니다. 현재 계획된 5px 임계값만으로는 기기 환경에 따라 자동 스크롤 상태로 복귀하지 않거나 의도치 않게 풀려버리는 현상이 발생할 수 있습니다.
- 필터 칩(`Error`, `Warning`)을 클릭하여 클라이언트 필터링이 발생할 때, 기존 로그 목록의 높이가 줄어들면서 현재의 스크롤 위치(`scrollTop`)가 뷰포트를 벗어나 허공을 가리키거나 비정상적으로 스크롤이 튀는 현상이 예상됩니다.

## Security concerns
- 시스템 알림 내에 포함될 수 있는 스택 트레이스나 내부 경로, 민감한 API 토큰 등이 마스킹 처리 없이 클라이언트 측 뷰(`SystemAlertWidget`)로 전달되어 렌더링될 위험이 있습니다. 클라이언트 사이드 필터링을 수행하기 전에, 데이터를 제공하는 출처나 렌더링 파이프라인에서 적절한 살균(Sanitization) 처리가 이루어지는지 검증이 필요합니다.
- 알림 출력부를 통해 XSS 공격을 유발할 수 있는 악성 스크립트가 실행되지 않도록, React의 기본 텍스트 이스케이프 외에 추가적인 `dangerouslySetInnerHTML` 사용처가 없는지 확인해야 합니다.

## Missing tests / weak test coverage
- 운영체제별(Windows/Mac) 스크롤바의 유무 및 마우스 휠/터치패드 가속도 차이에 따른 스크롤 이벤트 발생 빈도(Throttle 작동)를 검증하는 크로스 브라우징 및 OS 테스트가 누락되어 있습니다.
- 한 번에 수천 개의 로그가 적재된 상태에서 필터 칩 상태를 전환할 때 브라우저의 메인 스레드가 블로킹되지 않는지(프레임 드랍 여부)를 확인하는 성능(Performance) 단위 테스트가 부족합니다.
- 로컬 개발 환경 및 Playwright E2E 테스트 실행 시 지정할 포트 설정이 누락되어 포트 충돌 가능성이 있습니다. 실행 및 재현 예시는 반드시 `3100`번대 포트를 사용하여 환경을 격리해야 합니다. (예: `http://localhost:3100`)

## Edge cases
- 브라우저 탭이 백그라운드로 전환되어 비활성화된 상태에서 대량의 로그 이벤트가 Push될 경우, 브라우저가 DOM 렌더링을 지연시킴에 따라 사용자가 탭으로 복귀했을 때 스크롤 일시 정지 상태(`isAutoScrollPaused`)와 실제 DOM 스크롤 위치 간의 동기화가 어긋날 수 있습니다.
- 특정 필터(예: `Error`)를 선택해 결과가 0개인 상태(Empty State)에서 스크롤 이벤트를 발생시킨 뒤, 새로운 에러 로그가 들어왔을 때 스크롤 제어 로직이 정상적으로 초기화 및 작동하는지에 대한 시나리오 처리가 모호합니다.
