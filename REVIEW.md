# REVIEW

## Functional bugs
- **상태 변이(Mutation) 부작용**: `App.tsx`의 `setToasts` 상태 업데이트 함수 내부에서 `Ref`를 직접 조작(`dedupedToastKeysRef.current.delete()`)하는 방식은 React의 렌더링 생명주기를 위반합니다. 동시성 모드 등에서 예기치 않은 렌더링 버그를 유발할 수 있으므로, Plan에서 명시한 바와 같이 해당 로직을 `useEffect`나 이벤트 핸들러 내부로 분리해야 합니다.
- **리사이즈 이벤트 과부하**: `Toast.tsx`에서 모바일 뷰포트 판단을 위해 `resize` 이벤트 리스너를 사용할 때, 디바운스(Debounce)나 쓰로틀(Throttle) 처리가 누락될 경우 창 크기를 연속적으로 조절할 때 과도한 리렌더링이 발생하여 성능 저하 및 UI 멈춤 현상이 일어날 수 있습니다.
- **레이아웃 덜컹거림(Jumping)**: 텍스트 확장(Expand) UX 적용 시 Toast 컴포넌트의 높이가 갑작스럽게 변형되면, 화면 내 스택된 다른 알림의 위치가 부자연스럽게 밀려나는 레이아웃 깨짐 현상이 발생할 수 있습니다.

## Security concerns
- **XSS(교차 사이트 스크립팅) 취약점**: 웹훅 데이터 페이로드인 `error.detail`에 대한 완벽한 사니타이징을 MVP에서 제외(Out-of-scope)하기로 했으나, 프론트엔드에서 외부 입력값을 직접 렌더링할 때는 항상 악의적인 스크립트 주입 위험이 존재합니다. React의 기본 이스케이프 처리를 우회하는 방식(`dangerouslySetInnerHTML` 등)이 코드 내에 절대 사용되지 않도록 구조적 제한과 리뷰가 필요합니다.

## Missing tests / weak test coverage
- **동시성 및 상태 경합 테스트 부족**: 알림 큐가 최대치(3개)에 도달한 상황에서 기존 알림의 타이머 만료 시점과 새로운 웹훅 알림 수신 시점이 정확히 겹치는 Race Condition 통합 테스트 시나리오가 더욱 구체적으로 보강되어야 합니다.
- **리사이즈 이벤트 모킹(Mocking) 한계**: 단위 테스트(`Toast.test.tsx`) 수준에서 `window.resize` 이벤트를 흉내 내는 것만으로는 실제 브라우저 렌더링 엔진의 텍스트 말줄임 및 CSS 레이아웃 변경을 완벽히 검증하기 어려우므로, E2E 수준의 뷰포트 테스트 보완이 요구됩니다.

## Edge cases
- **다중 상호작용 충돌**: 사용자가 Toast의 닫기 버튼을 매우 빠르게 연타하거나 모바일에서 텍스트 확장 영역을 연속으로 터치할 경우, 비동기 상태 업데이트가 꼬여 앱에 에러가 발생하거나 불필요한 알림 식별자가 남아있을 가능성이 있습니다.
- **극단적인 화면 해상도**: 화면 높이가 매우 비좁은 가로 모드 모바일 기기 등에서 장문의 텍스트를 확장했을 때, 알림 카드가 화면 밖으로 벗어나거나 가려져 사용자가 닫기 동작 자체를 수행할 수 없게 되는 상황을 방지해야 합니다.
- **정확히 동일한 타이밍의 알림 수신**: 동일한 식별자(`dedupeKey`)를 가진 중복 웹훅이 밀리초 단위로 동시에 수신될 때, Ref 갱신과 React 상태 렌더링 사이의 미세한 간극으로 인해 중복 필터링이 실패할 가능성이 존재합니다.
