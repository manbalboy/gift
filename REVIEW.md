# REVIEW

## Functional bugs
- 현재 구현된 워크플로우 재개(Resume) API 및 프론트엔드 UI 연동 로직은 백엔드 엔진의 상태 전이(`paused` -> `running`)와 일치하게 원활히 동작합니다.
- 프론트엔드의 E2E 테스트 통과 내역을 확인한 결과, Resume 버튼 클릭 시 API 호출 및 실시간 상태 동기화에 문제가 없는 것으로 파악됩니다.
- 장기 실행 노드 맞춤형 타임아웃 예외 처리(`timeout_override`) 로직도 스키마 파싱 및 엔진 레벨에 잘 적용되어 의도한 대로 작동 중입니다.
- **현재 시점에서 발견된 치명적이거나 블로킹되는 기능적 버그는 없습니다.**

## Security concerns
- `require_viewer_token` 미들웨어에 Fail-closed 정책이 강제 적용되어, 환경 변수에 `viewer_token`이 누락된 상태에서는 모든 요청이 500 에러로 즉시 차단되므로 보안성이 크게 개선되었습니다.
- `_enforce_localhost_spoof_guard`를 통해 `Host: localhost:3100` 등의 헤더 위조 요청이 들어왔을 때, 실제 접속 클라이언트 IP와 Proxy 헤더(`X-Forwarded-Host`, `X-Forwarded-For`)를 대조하여 403 에러로 차단하는 로직이 성공적으로 구현되었습니다.
- **추가 고려사항**: 현재 방어선이 `3100~3199` 포트 대역의 로컬 서비스에 집중되어 있습니다. 차후 시스템이 확장되어 3100번대 외의 다른 중요 내부 포트(예: Redis, DB 등)가 노출될 위험이 발생할 경우, 소스코드의 하드코딩된 포트 검사 범위를 설정(Configuration)으로 분리하거나 모든 로컬포트로 확장하는 방어적 설계가 필요할 수 있습니다.

## Missing tests / weak test coverage
- 요구사항에 명시된 백엔드 보안/기능 통합 단위 테스트와 프론트엔드 E2E 테스트(Pause/Resume 흐름 등)가 성공적으로 통과(Pass)하고 있어 기본적인 커버리지는 양호합니다.
- 다만, 노드 단위의 `timeout_override`가 적용되었을 때의 실제 엔진 스케줄링 변화를 구체적으로 검증하는 명시적인 엣지 케이스 테스트가 백엔드 단위 테스트(Unit test) 층위에 조금 더 보강되면 완벽할 것입니다.

## Edge cases
- **동시성(Concurrency) 문제**: 대시보드를 보고 있는 복수의 운영자가 거의 동일한 시간에 동일한 `paused` 런타임에 대해 Resume 버튼을 눌렀을 경우, `POST /runs/{run_id}/resume` 요청이 중복 처리되어 엔진 내부에서 동일한 노드를 2회 스케줄링하지 않도록 멱등성(Idempotency)이나 상태 락(Lock)이 안전하게 걸려있는지 주의가 필요합니다.
- **장기 방치된 런타임 재개**: 워크플로우가 수일 동안 `paused` 상태로 방치된 후 재개될 때, 이전 노드가 생성한 임시 아티팩트가 스토리지(S3 등) 객체 생명주기 규칙에 의해 지워져 있을 경우 연쇄적인 런타임 에러(Crash)를 방지하고 "안전한 실패(Graceful fail)" 상태로 떨어지게 하는 처리가 필요할 수 있습니다.
- **스트림(SSE) 재연결 타이밍**: 사용자가 페이지를 새로고침하거나 네트워크 단절 후 재연결되는 찰나의 순간에 워크플로우 상태가 `running`으로 전환될 경우, UI가 이전 상태인 `paused`를 보여줄 수 있는 레이스 컨디션을 프론트엔드에서 완전히 해소했는지 엣지 테스트에서 한 번 더 확인하면 좋습니다.
