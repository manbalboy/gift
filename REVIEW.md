# REVIEW

## Functional bugs
- 시스템 알림 위젯(`SystemAlertWidget.tsx`)에서 자동 스크롤 기능이 활성화되어 있을 때, 과거 로그를 탐색하기 위해 스크롤을 올린 상태라도 새로운 알림이 수신되면 강제로 최하단으로 스크롤이 이동하여 사용자 경험(UX)을 심각하게 저해하는 결함이 있습니다.
- 모바일 해상도(320px 뷰포트 기준)에서 새롭게 추가될 필터 칩(Chip) UI와 기존의 [CLR], [EXP] 액션 버튼이 겹치거나 가로 오버플로우(가로 스크롤바)가 발생하여 레이아웃이 붕괴될 위험이 큽니다.
- 클라이언트 사이드 필터링 적용 시, 결과 목록이 0개일 때 실제로는 커서 기반 페이징의 `hasMore` 상태가 남아있음에도 불구하고 사용자에게는 "더 이상 로그가 없다"는 잘못된 시각적 피드백을 줄 수 있는 상태 불일치 버그 가능성이 존재합니다.

## Security concerns
- 클라이언트 사이드에서 로그를 필터링하는 방식 자체는 보안상 큰 위협이 아니나, `Error`나 `Warning` 레벨의 알림 텍스트 내에 시스템 내부 스택 트레이스나 민감한 환경 정보가 마스킹 없이 포함될 경우 클라이언트 브라우저로 정보가 유출될 우려가 있습니다.
- CORS 정책과 관련하여, 허용된 Origin(`https://manbalboy.com` 등) 외의 접근을 철저히 차단해야 하며, 로컬 재현 환경(예: `http://localhost:3100`)에서의 테스트 시에도 불필요한 데이터가 외부로 노출되지 않도록 네트워크 접근 제어가 필요합니다.

## Missing tests / weak test coverage
- 필터 칩 클릭에 따른 상태 변경(`All`, `Error`, `Warning`) 및 화면에 렌더링되는 DOM 요소의 개수를 검증하는 단위/컴포넌트 테스트(React Testing Library)가 누락되어 있습니다.
- 모바일 사이즈(320px) 에뮬레이터 환경에서 컴포넌트의 가로 오버플로우 발생 여부 및 뷰포트 바운더리를 검증하는 Playwright E2E 테스트 스펙이 부족합니다.
- 임의로 스크롤을 위로 올린 상태에서 대량의 로그를 주입(Mocking)했을 때, 리스트 컨테이너의 `scrollTop` 위치가 변동 없이 고정되어 자동 스크롤 일시 정지 상태가 유지되는지 확인하는 E2E 시나리오 검증이 필요합니다.

## Edge cases
- 런타임 환경에서 알림 이벤트가 매우 빠른 속도(예: 밀리초 단위)로 폭주할 경우, 빈번한 `onScroll` 감지 및 배열 필터링 재연산으로 인해 브라우저 렌더링 병목 및 성능 하락이 발생할 수 있습니다.
- 스크롤을 뷰포트의 완전한 최하단이 아닌 미세한 오차 범위(예: 1~3px) 위에 걸쳐 두었을 때, 이를 스크롤 업(일시 정지)으로 판단할지 최하단(자동 스크롤 재개)으로 판단할지 모호한 경계 조건이 발생합니다.
- 브라우저에 적재된 알림 데이터가 수만 건 이상 누적된 상태에서 `Error` 칩을 클릭할 경우, 클라이언트 사이드 필터링 연산이 메인 스레드를 블로킹하여 화면 프리징을 유발할 수 있습니다.
