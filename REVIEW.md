# REVIEW

## Functional bugs
- **보안 유틸리티 XSS 필터 과적합**: 프론트엔드의 `web/src/utils/security.ts` 내부 태그 제거 정규식(`/ <[^>]+>/g`)이 지나치게 강하게 설정되어 있습니다. 이로 인해 `<T>`와 같은 제네릭 문법이나 정상적인 꺾쇠괄호 텍스트까지 모두 삭제되어 화면에 출력되어야 할 코드 조각이나 타입 명세 데이터가 유실되는 버그가 있습니다.
- **지시사항 Drop 예외 처리 피드백 누락**: 루프 엔진 큐가 가득 찼을 때 오래된 지시사항을 `dropped` 상태로 전환하고 있으나, 프론트엔드(`web/src/components` 영역)에는 이를 감지하고 사용자에게 명시적으로 알려주는 경고 알림(Toast 등) UI가 연결되어 있지 않습니다.
- **상태 모니터링 컴포넌트 미구현**: PLAN에서 요구한 단일화된 엔진 프로세스 진행 상태 및 Quality Score 시각화를 위한 `LoopMonitorWidget.tsx` 컴포넌트가 아직 생성되지 않았습니다. 현재 사용자는 직관적으로 루프 상태를 확인할 수단이 부족합니다.

## Security concerns
- **악성 텍스트의 투명한 렌더링 실패**: 현재의 DOMPurify 및 커스텀 정규식 조합은 악성 스크립트(예: `<img src=x onerror=alert(1)>`)를 완전히 삭제하는 방식으로 동작하고 있습니다. 보안 측면에서 스크립트 실행은 방어하고 있지만, 시스템 경고나 로그의 무결성을 위해서는 해당 문자열을 HTML 인코딩(`&lt;`, `&gt;`)하여 화면에 텍스트 형태로 안전하게 노출해야 합니다. 현재 방식은 원본 로그를 훼손하여 관리자의 추적 및 감사를 방해할 소지가 있습니다.

## Missing tests / weak test coverage
- **프론트엔드 XSS 유닛 테스트 실패**: 악성 스크립트 문자열이 안전한 텍스트로 화면에 표시되는지 검증하는 `SystemAlertWidget.test.tsx`의 기대 결과가, 현재의 강제 태그 삭제 로직과 충돌하면서 테스트가 실패(`Unable to find an element with the text...`)하고 있습니다. 관련된 역검증 테스트 케이스 보완이 누락되었습니다.
- **장애 전파 통합 E2E 테스트 부재**: 백엔드 `api/tests/test_workflow_engine.py`에 Redis Lock 객체 자체의 fail-fast 단위 테스트는 존재하지만, 실제로 Lock 획득 실패 시 전체 시스템이 에러 상태로 전환되고 포트 3100 기반 로컬 API 응답에 경고가 올바르게 전파되는지를 확인하는 종단 간(E2E) 테스트 시나리오가 아직 작성되지 않았습니다.

## Edge cases
- **다중 제어 명령으로 인한 Race Condition 방어 미검증**: 백그라운드 엔진 실행 중 Pause, Resume, Stop 등 상태 제어 신호가 0.1초 단위의 짧은 간격으로 여러 번 인가될 때 발생할 수 있는 스레드 데드락이나 상태 꼬임 현상을 검증하는 동시성 스트레스 테스트가 없습니다.
- **큐 오버플로우 시 Silent Drop 인지 지연**: 최대 대기열 큐 한도 도달 시 가장 오래된 명령을 `dropped` 처리하지만, 클라이언트가 API 폴링 시점을 놓칠 경우 지시사항이 무시된 것을 영원히 인지하지 못하는 엣지 케이스가 존재합니다.
